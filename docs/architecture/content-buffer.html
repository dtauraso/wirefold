wirefold — architecture

A concurrent dataflow network where the topology is the logic. Go owns the model; TypeScript only draws it.

The whole scene, as one binary buffer

Go packs everything into Buffer/ and streams it as length-prefixed frames on fd 3 on every change. TS decodes the latest snapshot and draws it generically — no positions or geometry computed on the TS side.

TS → Go: exactly three edit ops

create / delete — add or remove an edge.
update — set an attribute on a typed entity (node / edge / camera / overlays / scene).

New editor→Go capability is a new entity kind or attribute — never a new op. Fading an edge, moving a port, orbiting the camera, toggling an overlay are all updates. Parity guarded across messages.ts, stdin_reader.go, handle-message.ts, and fenced in Go by EDIT_OPS_START/EDIT_OPS_END.

Three edit ops — but seven top-level message types. applyEdit does hold exactly create/update/delete, fenced by EDIT_OPS_START/END. The enclosing stdin dispatch is wider: edit, play, pause, resend, raw-input, save, and fade-toggle. fade-toggle is a deliberate bare command — it names no entity because Go owns the selection, so there is nothing for TS to address an update at. It is fully enforced (guarded INPUT_LAYOUT_FINGERPRINT, message-kind parity, a test). CLAUDE.md's bridge paragraph omits it and says fade is an update — that doc is stale, not the code. See Drift Log.
