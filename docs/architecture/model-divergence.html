wirefold — architecture

A concurrent dataflow network where the topology is the logic. Go owns the model; TypeScript only draws it.

How far is this code from what an AI would have written?

Read this first — these numbers are not measurements. No model can measure its distance from its own training data: there is no access to the corpus, no lookup, no counter. Any figure claiming to be a measured "% difference from training" would be fabricated. What follows is something weaker but honest — a self-reported prior: for each decision slot, the model's own estimate of how often it would emit the industry default if asked to fill that slot cold, with no CLAUDE.md and no correction. Treat it as a calibrated guess about a pull, not a fact about a corpus. The evidence column is the real content; the percentages are a way of ranking it.

Why the pull exists at all

Training data is overwhelmingly "logic in procedures, topology as plumbing." This project is the deliberate inversion — the topology is the logic. So a model completing wirefold code isn't neutral: it is being asked to write the rarer form of nearly every construct it knows. That is what CLAUDE.md's medium vs. substance rule encodes, and why the substance half says ignore industry defaults.

Slot-by-slot: what the default would have been

Each row is a real decision in this codebase where the industry-default answer is known — because in most of these cases it was actually built first, then rejected. That history is what makes this table evidence rather than speculation.

Execution model — goroutines + channels vs. async/await pull: very high

Default: translate goroutines to async/await + Promises. This is the industry-correct JS translation, and it was built. It hid pacing inside the event loop and coupled nodes that must stay independent. Here: Go owns one clock; the TS side awaits nothing (guarded: check-no-await-on-bridge.sh). Recorded in CLAUDE.md as the canonical prior failure — right answer for the medium, wrong answer for the substance.

Coordination — decentralized messages vs. a central worklist pull: very high

Default: a coordinator — collect the affected nodes, walk a queue, apply in order. Built and reverted: the central cascade "exploded"; rootMove was deleted outright. Here: per-node inboxes, sendMove, no worklist. An unstated Go-layer slot gets filled with a coordinator by default — which is exactly why the project rule is to state the Go-layer answer first.

Camera — polar-only nav vs. OrbitControls pull: very high

Default: drop in OrbitControls; it's what every three.js example does. Rejected: it sacrifices control, and 3D-editor interaction here is substance, not medium — the pattern-match is the bug. A related detour built an arcball because a plain-language spec got matched onto the textbook algorithm instead of read literally. Here: great-circle polar nav chosen so the wrong-axis bug class cannot be expressed (guarded: check-polar-only-nav.sh).

Wire — straight PacedWire vs. a bead chain pull: high

Default: model the wire as a chain of linked bead items — the tempting OO decomposition. Built, then rejected: O(N²) follow latency. Here: a straight line with an inflight FIFO. The memory entry exists specifically to stop it being re-proposed — which is itself evidence of how strong the pull is.

Delivery — no handshake vs. ack/nack pull: high

Default: networking instincts — acks, back-pressure, send-gating, delivery guarantees. Here: nodes do local work and drive their outputs; a mismatched bead is consumed and dropped silently. No delivery guarantee exists anywhere, by design.

Webview state — reflect the buffer vs. a store pull: high

Default: Redux/Zustand/context — a React app "obviously" owns its state. Here: TS authors nothing; it reflects Go's buffer read-only through useSyncExternalStore. Now mechanically guarded (check-no-webview-state.sh) because prose alone did not hold the line.

Wire props — uniform pulse speed vs. per-wire speed pull: medium

Default: make it configurable; add a per-wire knob. Here: explicitly rejected — pulse speed is uniform across all wires. Note the code still allows the divergence (pulseSpeed is an instance field); only caller convention keeps it uniform. This is the row where the default could still creep back in.

Bridge encoding, build tooling, test runner pull: low — and that's fine

Length-prefixed binary framing, esbuild, vitest, eslint, Go modules. Conventional on purpose. The medium is where industry has already solved the problem; being weird here is pure overhead.

What this predicts about working with an AI on this repo

Regression is toward the mean, not toward chaos. A model's wrong answer here will be tidy, idiomatic, and reviewable — a clean coordinator, a sensible store. That is what makes it dangerous: it doesn't look like a mistake.
The guards are the real defense. Each check-*.sh is a fossil of a default that once got in. Prose lost; code held. This is why fix-order is code-first.
Unstated slots get filled. If the Go-layer answer isn't named up front, the pull fills it with the coordinator-shaped default — the model won't leave it blank.
Stale prose amplifies the pull. A comment naming a deleted symbol (WaitTick, useCameraStore) points the model back at the pattern the code moved away from. See Drift Log.
The honest summary. On the medium, this codebase is ordinary and an AI is a good collaborator. On the substance, nearly every load-bearing decision is the less common of the available forms, and several were reached by building the default first and throwing it away. The percentages above are a self-report and should be read as ranking, not measurement — but the rejected-and-reverted history behind them is on the record in git log and memory/, and that part is checkable.
