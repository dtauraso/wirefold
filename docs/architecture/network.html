wirefold — architecture

A concurrent dataflow network where the topology is the logic. Go owns the model; TypeScript only draws it.

Nodes, wires, beads, channels

The network is the running program. Nodes coordinate only through channels and topology. There is no central runner, no walker, no lockstep rounds.

A node is several goroutines; a wire is none. Each node runs Update (the pausable bead loop), UpdateLayout (always-on, not gated by play/pause), and a per-node mover nodeMover.run — all three launched per node (main.go, node_move.go). A wire has no loop goroutine of its own: beads are placed by the source and stepped by the owning node's goroutine driving StepOnce — there is no per-wire or per-bead goroutine. The only wire-adjacent goroutine is the per-edge geometry mover.
Bead — a value in transit from a source node to a destination node.
Wire — transport and visual depiction. A multi-bead FIFO: times traversal in ticks, advances beads, emits positions, delivers on completion. Applies no send policy. It does own two queues — inflight and delivered — so a delivered bead parks on the wire until a reader consumes it.
Node — receives beads, holds them in node-local state, fires when its rule is met. No slots — held state replaces them. Drives its own wires' StepOnce.
Channel — connects source→wire or wire→node. Distinct inputs are distinct channels.

Sending has no handshake

A node places a bead on its wire whenever its rule says to. No clear/busy, no ack, no back-pressure. A wire may carry several beads at once. Coordination between nodes is the topology + each node's local rule — not a delivery guarantee.
