wirefold — architecture

A concurrent dataflow network where the topology is the logic. Go owns the model; TypeScript only draws it.

Can the code answer a question without being grepped?

An agent reads comments and doc-claims first. So the question isn't "is the code clean" — it's how much of what the code says about itself is enforced. Prose drifts silently; a guard cannot. This repo's own doctrine is code self-defends: enforce in code before adding more prose.

What makes this repo unusually readable to an agent

Generated registry — node-defs.ts is generated from each SPEC.md; the kind list cannot disagree with itself.
Parity guards — the same fact living in two languages is checked, not trusted (buffer-layout, edit-op, message-kind, send-rule, input-layout, bridge-literal).
Fence markers — EDIT_OPS_START/END give a guard an exact span to read instead of a heuristic.
One version number — buffer layout v20 pins both sides of the wire to one artifact.
Negative tests — a test asserts the removed node-label sidecar is rejected. Deletion is enforced, not just done.
Breadcrumb channel — one structured debug path instead of scattered Fprintf, so diagnosis is greppable.

Where an agent gets misled

Guards cover what is mechanically checkable; the prose around them was never checked. The result is a repo where the enforced layer is excellent and the read-first layer lies. That asymmetry is the finding — not "there are some stale comments."

Fix order is code-first. The lesson of this audit isn't "update the comments." It is that a claim worth making is worth enforcing: a stale symbol name in a docstring survived because nothing could fail when it went wrong. Prefer deleting an unenforceable claim over refreshing it.
