wirefold — architecture

A concurrent dataflow network where the topology is the logic. Go owns the model; TypeScript only draws it.

Plan — what the audit says to do

Everything below follows from one finding: the enforced layer is excellent and the read-first layer lies. Not a single false claim on this site came from the code being wrong. They came from prose that nothing could fail on. So the plan is not "update the comments" — it is move claims down a tier or delete them.

Ordering rule (the repo's own): fix order is code-first — enforce in code (a guard/hook) before adding more prose. Items are ordered by that rule, not by how annoying the symptom is. A prose fix that could have been a guard is a deferred recurrence, not a fix.

P0 · Make the highest-authority doc true

Fix CLAUDE.md's bridge-surface paragraph ~30 min · do first

It is wrong twice, and it is the one file every agent reads before touching anything: (1) it claims a node-label sidecar that was deleted and is negatively tested; (2) it omits fade-toggle from the TS→Go list and says fading is an update on an entity, when it is a deliberate bare command because Go owns the selection. Why first: highest authority × highest read-frequency × zero risk. Nothing else on this page changes behavior if an agent starts from a false map.

P1 · Kill the stale-symbol class outright

New guard: tools/check-doc-symbols.sh ~half day · highest leverage

Extract backticked/`code`-shaped identifiers from Go doc comments, CLAUDE.md, MODEL.md and the SPEC.md files; fail if an identifier resolves to no definition anywhere. WaitTick and useCameraStore both die permanently, and — more to the point — so does the next one, before anyone reads it. Why this over spot-fixes: the general mechanism beats the point patch; three of six Drift Log rows are one bug class (a comment naming a symbol that no longer exists). Expect a noisy first run — allowlist prose words, don't weaken the check.

P2 · Close the gap the guards don't cover

Fence the stdin dispatch: MSG_TYPES_START/END ~2 hrs

The EDIT_OPS_START/END fence already lets a guard read an exact span rather than guess — that pattern works, so extend it. Fence the seven-case dispatch switch and have check-message-kind-parity.sh compare the fenced cases against the file header's own list. The header then cannot undercount its switch. While there: reword the "fade edits no longer cross this seam" line, which is true but reads as a contradiction of the fade-toggle case below it.

P3 · Delete a free axis before the default creeps back

Make uniform pulse speed structural ~2 hrs · needs sign-off

Per-wire speed props are explicitly rejected doctrine, but pulseSpeed is still an instance field on every wire — uniform only because every caller happens to pass the same constant. Nothing fails if one caller doesn't. Collapse it to the package-level constant and drop the field, so "wires with different speeds" becomes unwritable rather than merely discouraged. Why it matters: Model Divergence rates this the row most likely to regress — "make it configurable" is the strongest default in the table that the code still permits. Check first whether any test or tool passes a non-default speed; this is a real (small) behavior-surface change, not a cleanup.

P4 · Narrow claims that overstate a true invariant

Rewrite the "ONE cartesian value" comment ~30 min

scene_sphere_persist.go asserts the center is the only cartesian value in the system. False — camera pose, port anchors, bead segments and derived centers are all cartesian. The true invariant is stronger for being narrower: the only cartesian that is persisted and authoritative; everything else is derived or quarantined at the renderer edge. An overstated invariant teaches a reader to distrust the accurate ones next to it.

P5 · Delete, don't refresh

Two removals ~20 min

Drop the vestigial json:"…" tags on stdinMsg (nothing unmarshals them; they imply a JSON seam that no longer exists), and the useCameraStore mention in overlay-flags.ts — a comment pointing at a store the guards exist to forbid. Both are pure subtraction. If P1 lands first, the second one is already caught.

P6 · This site, honestly

Decide what these pages are decision, not work

This site is tier-3 prose and will drift exactly like the comments it documents — it is the newest instance of the problem it describes. Three options, pick one deliberately: (a) date-stamp and accept (what it does now — the date is the only warranty); (b) generate the checkable parts (block table, kind list, guard list) from source, so they cannot go stale; (c) delete it once its findings land as guards, and keep the durable doctrine in MODEL.md where it belongs. Left open on purpose — (b) is real build surface for a doc, and this repo's rule is that planning docs are branch-local and don't ride to main.

Sequencing

P0 → P1 are the whole plan if attention is short: fix the map every agent starts from, then make the class of error that produced this audit unrepresentable. P2 and P5 are cheap follow-ons. P3 is the only item with a behavior surface and wants sign-off. P4 is a rewrite. P6 is a question for the repo owner, not a task.

What success looks like: re-run this audit in six months and the false-claim count is zero because a guard failed first — not because someone re-read the comments. Every check-*.sh in tools/ today is a fossil of a default that once got in. This plan adds one more fossil and removes a bug class.

Effort figures are rough estimates, not measurements. Per the repo's cost convention, only a ≥$5 bundle carries a cost marker — P1 is the only item likely to reach it alone.
