wirefold — architecture

A concurrent dataflow network where the topology is the logic. Go owns the model; TypeScript only draws it.

Two processes, one binary bridge

Everything real lives in the Go runtime — nodes, wires, the clock, all geometry. It packs the whole scene into a binary content buffer and streams it to a VS Code webview that only decodes and draws, and forwards raw input back. The bridge is binary both ways and fire-and-forget (no awaits, no request/response).

"Binary both ways" is true of the bridge, not of every pipe. stdout is a third channel and it is still JSON: debug breadcrumbs ({"kind":"breadcrumb",…} → .probe/go-debug.jsonl) and one EmitSpecLine topology-spec line the ext host forwards as load. What was removed is the JSON trace-event emitter — those now decode from the buffer's Event block.
Governing principle — medium vs. substance. For the medium (VS Code, React, three.js) adopt the dominant industry choice. For the substance (what a node is, how ticks work, what a wire is) ignore industry defaults — "logic in procedures, topology as plumbing" is exactly the inversion this project challenges.

Nodes, wires, beads, channels

The network is the running program. Nodes coordinate only through channels and topology. There is no central runner, no walker, no lockstep rounds.

A node is several goroutines; a wire is none. Each node runs Update (the pausable bead loop), UpdateLayout (always-on, not gated by play/pause), and a per-node mover nodeMover.run — all three launched per node (main.go, node_move.go). A wire has no loop goroutine of its own: beads are placed by the source and stepped by the owning node's goroutine driving StepOnce — there is no per-wire or per-bead goroutine. The only wire-adjacent goroutine is the per-edge geometry mover.
Bead — a value in transit from a source node to a destination node.
Wire — transport and visual depiction. A multi-bead FIFO: times traversal in ticks, advances beads, emits positions, delivers on completion. Applies no send policy. It does own two queues — inflight and delivered — so a delivered bead parks on the wire until a reader consumes it.
Node — receives beads, holds them in node-local state, fires when its rule is met. No slots — held state replaces them. Drives its own wires' StepOnce.
Channel — connects source→wire or wire→node. Distinct inputs are distinct channels.

Sending has no handshake

A node places a bead on its wire whenever its rule says to. No clear/busy, no ack, no back-pressure. A wire may carry several beads at once. Coordination between nodes is the topology + each node's local rule — not a delivery guarantee.

One clock, in ticks

There is exactly one clock: the system monotonic clock read through a scale so it advances in integer ticks at human-watchable speed. Halted time is subtracted, so pausing genuinely freezes the tick:

tick = ⌊(now − start − haltedTotal) / tickPeriod⌋  (MsPerTick = 16 → ≈62.5 ticks/sec)

The model is sleep-only. The clock interface is Tick() · SleepCycle(ctx) · Halt() · Resume(). Pacing loops call SleepCycle to wait exactly one cycle rather than blocking on a target tick — there is no WaitTick(k) (the symbol does not exist; only stale comments name it). Halt touches no goroutine — it sets a flag and a timestamp. SleepCycle is a raw timer and is not pause-aware: paused loops keep spinning, they just observe a frozen Tick(). Pause governs beads only — UpdateLayout never waits on the clock and keeps running.
No lockstep rounds, no simultaneity layer. Goroutines schedule independently against the shared tick. Any reasoning that treats activity as globally-aligned rounds is drift.

The whole scene, as one binary buffer

Go packs everything into Buffer/ and streams it as length-prefixed frames on fd 3 on every change. TS decodes the latest snapshot and draws it generically — no positions or geometry computed on the TS side.

TS → Go: exactly three edit ops

create / delete — add or remove an edge.
update — set an attribute on a typed entity (node / edge / camera / overlays / scene).

New editor→Go capability is a new entity kind or attribute — never a new op. Fading an edge, moving a port, orbiting the camera, toggling an overlay are all updates. Parity guarded across messages.ts, stdin_reader.go, handle-message.ts.

Positions & locks are pure polar

The scene sphere's center is the only authoritative, persisted cartesian value — the one world anchor in scene.json. Every node position and every movement lock is stored in polar coordinates about it.

Careful: "the only cartesian value in the system" is a claim the code makes about itself (scene_sphere_persist.go) and it is not literally true. Cartesian vec3 is everywhere it has to be: the camera pose (gesture_camera.go is a whole file of Cartesian math, self-described as "quarantined"), port anchors, bead segments, and node centers derived per-frame as sceneCenter + polar2cart(…). The true invariant is narrower and still strong: cartesian is derived or quarantined at the renderer edge, never persisted and never a source of truth. Nav is polar-only, and that is guarded (check-polar-only-nav.sh).
The "positions fly to infinity" bug came from reconstructing an offset as cart2polar(node − center) against a mid-moving center. That reconstruction is now structurally impossible — the offset is stored and carried node→node, never re-derived from a live moving world. Lock propagation is a decentralized node-to-node cascade over per-node inboxes (sendMove) with no central worklist; the central rootMove was deleted. Local polar lives on LayoutHolder (SetLocalPolar / localPolarSteps).

Render + forward only

The webview authors no domain state — no render store, no spec store, no camera store. It reflects the buffer read-only via useSyncExternalStore (overlay-flags.ts) and forwards raw input. Guards enforce this (check-no-webview-state.sh, check-no-await-on-bridge.sh, check-ts-computes-no-geometry.sh) — 20 check-*.sh guards in all.

"No state" means no authority over a streamed bit, not zero useState. Render-local state is fine and present: instance-buffer capacities (buffer-scene.tsx), tube segment counts and faded rows (EdgeTube.tsx), label positions and occlusion counts (ThreeView.tsx), run-status context. None of it is the source of truth for anything Go owns.

Adding a node kind (same commit)

  • Go package under nodes/<Kind>/ with a SPEC.md
  • NODE_DEFS entry in schema/node-defs.ts (generated from SPEC.md)
  • If it needs a new buffer column: add it to Buffer/ + buffer-layout.ts and its reader

Rendering is fully generic — no per-kind component file exists (grep the webview for any kind name: zero hits). The 8 active kinds, PascalCase exactly as the Go kind name and the NODE_DEFS key: Hold, HoldFlip, HoldNewSendOld, Input, Pacer, Pulse, WindowAndInhibitLeftGate, WindowAndInhibitRightGate.

Can the code answer a question without being grepped?

An agent reads comments and doc-claims first. So the question isn't "is the code clean" — it's how much of what the code says about itself is enforced. Prose drifts silently; a guard cannot. This repo's own doctrine is code self-defends: enforce in code before adding more prose.

What makes this repo unusually readable to an agent

Generated registry — node-defs.ts is generated from each SPEC.md; the kind list cannot disagree with itself.
Parity guards — the same fact living in two languages is checked, not trusted (buffer-layout, edit-op, message-kind, send-rule, input-layout, bridge-literal).
Fence markers — EDIT_OPS_START/END give a guard an exact span to read instead of a heuristic.
One version number — buffer layout v20 pins both sides of the wire to one artifact.
Negative tests — a test asserts the removed node-label sidecar is rejected. Deletion is enforced, not just done.
Breadcrumb channel — one structured debug path instead of scattered Fprintf, so diagnosis is greppable.

Where an agent gets misled

Guards cover what is mechanically checkable; the prose around them was never checked. The result is a repo where the enforced layer is excellent and the read-first layer lies. That asymmetry is the finding — not "there are some stale comments."

Fix order is code-first. The lesson of this audit isn't "update the comments." It is that a claim worth making is worth enforcing: a stale symbol name in a docstring survived because nothing could fail when it went wrong. Prefer deleting an unenforceable claim over refreshing it.

Stale self-descriptions found in this audit

Each row is a place the code, or its instructions, assert something untrue. Verified against source on 2026-07-14. Ordered by how likely an agent is to act on it.

CLAUDE.md — the node-label sidecar highest risk

The project's own instruction file describes the bridge as "the binary content buffer plus the node-label id/label/kind sidecar." That sidecar is gone — labels ride the Node block's LABEL_OFF/LABEL_LEN columns into the Label section, and parseHostToWebview.test.ts actively asserts the message type is rejected. MODEL.md agrees with the code. An agent following CLAUDE.md would build against a deleted seam.

stdin_reader.go header — undercounts its own switch

The file header enumerates five top-level message types; the dispatch switch has seven — including an undocumented fade-toggle. The header also states fade edits "no longer cross this seam," which fade-toggle directly contradicts.

WaitTick — a symbol that no longer exists

Named in comments in input/node.go, layout_holder.go, and paced_wire.go. Zero definitions repo-wide. The clock has been sleep-only (SleepCycle) for some time. This one propagated straight into this page's Clock tab.

"The Center is the ONE cartesian value in the system"

A comment in scene_sphere_persist.go. False as written — camera pose, port anchors, bead segments and derived node centers are all cartesian. The intended invariant (only persisted cartesian anchor; everything else derived or quarantined) is true and worth stating precisely.

Vestigial json:"…" tags on stdinMsg cosmetic, still misleading

Nothing unmarshals them — input_codec.go decodes the binary record. They imply a JSON seam that no longer exists.

overlay-flags.ts references useCameraStore

In a comment only. No such store exists anywhere in src/ — it was erased with the old system. Points an agent at a state-authority pattern the guards exist to forbid.
