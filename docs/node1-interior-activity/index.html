Wirefold — Design Spec (draft, branch-local)
Node 1 — Interior Depleting / Refilling 4-Bead Buffer

What changes, and why.

Node 1 is the Input node (Go kind Input, nodes/input/node.go). Today it holds an Init array (e.g. [0,1]) and, in the feedback ring, advances a read index i by the downstream feedback step (0 = hold, 1 = advance, wrapping) and emits Init[i] on its output wire 1To2. The interior of the node body is currently inert.

The new design turns the interior into a visible double-buffer: two arrays, each [1,0] — a working array (the bottom 2 beads) and a backup array (the top 2 beads), 4 beads total, stacked in a vertical column inside the node body. Sending pops the END of the working array (the popped value rides out as a bead on 1To2). "Empty" means the working array (length 2) is empty — not all 4 beads gone. When working empties (after its 2 pops) the 2 backup beads slide down into the working slots and 2 new beads appear at the top — an animated refill, back to 4. The 4 interior beads are a faithful, frame-accurate view of the buffer's contents.

Double-buffer: working=[1,0] (bottom 2) + backup=[1,0] (top 2) = 4 beads. Pop: on feedback i==1, pop the END of working[] → one bead removed, its value launches on 1To2. i==0 = hold. End-popping [1,0] sends 0 first, then 1 (intended; array stored [1,0] for this). No reversed backup. Empty: "empty" = working (length 2) is empty, after its 2 pops. Refill: backup slides DOWN into the working slots + 2 new beads appear at the TOP — animated at human speed on Go's clock. Geometry: Input.Update() does the array mutation AND the interior bead geometry in ONE coupled step, then streams the INCREMENTAL changes; TS renders them and computes no geometry.

Double-buffer: working [1,0] + backup [1,0], pop-from-end, slide-down refill.

Two arrays, each [1,0], back the buffer — 4 beads total:

  • working = [1,0] — the live, depleting array (the bottom 2 beads). Sending pops its last element (the end), and that popped value is emitted as a bead on 1To2. End-popping [1,0] sends 0 first, then 1 — intended; the array is stored [1,0] precisely so end-pop yields that order. No reversed-backup trick is needed.
  • backup = [1,0] — the top 2 beads. On refill it slides down into the working slots; 2 new beads ([1,0] again) appear at the top.
The driving signal — feedback value i

The feedback value i read from downstream (on FeedbackIn, produced by the downstream ChainInhibitor) drives the buffer. Its meaning is redefined for this design:

i == 1 → pop the end of working[] and send that value as a bead // advance i == 0 → do nothing (hold) // hold
Pop
v = working[len(working)-1] // last element = "the end" working = working[:len(working)-1] // shrink by one emit(v) on 1To2 // popped value rides out as a bead
Refill — when working (length 2) is empty

"Empty" means the working array is empty (after its 2 pops) — not all 4 beads gone. When that happens, the backup slides down into the working slots and a fresh backup appears at the top:

if len(working) == 0: // working's 2 beads both popped working = backup // backup beads slide DOWN into working slots backup = [1,0] // 2 NEW beads appear at the TOP // back to 4 beads; the slide is animated at human speed (see Animation tab)
Settled — pop order. Each array is stored [1,0] and end-popped, so the emitted sequence is 0 then 1. This is the intended order; the storage [1,0] is chosen specifically so end-pop yields it. No reversed-backup is required.
Settled — refill. Refill triggers the moment working (length 2) empties. The 2 backup beads slide down; 2 new beads ([1,0]) appear at the top. Both arrays are length 2, so the buffer is always 4 beads.

4 beads: consume bottom 2 (working) → backup slides down → new backup at top.

The node body shows 4 beads in a vertical column. Slots are indexed top (slot 0) to bottom (slot 3). The bottom 2 slots are the working array ([1,0]); the top 2 slots are the backup array ([1,0]). The buffer is consumed from the bottom; the bottom slot (slot 3) corresponds to the end of working — the element that pops next.

  • As the bottom 2 working beads are consumed (two pops, i==1 twice — sending 0 then 1), the bottom 2 slots empty and working is empty.
  • The 2 backup beads slide down into the vacated working slots (this slide IS the refill).
  • 2 new backup beads appear at the top — backup back to [1,0]. Buffer is 4 again.

Beads disappear on pop and the slide-down refill happens the moment working empties. This must coincide exactly with the buffer's contents and length at each step. The column is a faithful render of the array state — never a free-running animation. The slide is animated at human speed, timed on Go's clock with the same human-speed slowdown used for in-flight beads.

Settled — animated slide. The slide-down is animated at human speed, timed on Go's clock with the same human-speed slowdown used for in-flight beads. The refill IS this slide, begun the moment working empties. Go streams the moving beads' positions per frame over the slide duration.

No index i — array state, pop-from-end, slot positions, drop-down — computed in Go's Update().

This project is Go-authoritative for geometry: TS computes no positions. Node 1's Update() does the array mutations and the geometric calculation of the 4 interior bead positions, then streams them. The formulas below are proposed and use real code-tied names.

Removed The persistent index variable i is gone. There is no read-head counter and no wrapping formula (i % len) anywhere in Update(). The working array IS the state: the end of the array is the next value to emit. Pop-from-end replaces indexed access; the feedback signal gates pops directly against array length; nothing about an index persists between ticks.
1 · Pop decision from feedback signal (no index)

The feedback signal gates a pop directly against the array — no counter is read or advanced:

s := FeedbackIn.TryRecv() // downstream step, 0 or 1 if s == 1 { // pop & send one — no index, the end is the value v := working[len(working)-1] working = working[:len(working)-1] ToReadGate.TryEmit(v) // bead rides out on 1To2 } // s == 0: hold, do nothing
2 · Refill condition — slide down + new backup
if len(working) == 0 { // working (length 2) drained by its 2 pops working = backup // backup slides DOWN into the working slots backup = []int{1, 0} // 2 NEW beads at the TOP } // each array is length 2; buffer is always 4 beads
3 · Slot-position formula (4 beads = backup[0..1] + working[0..1])

Let the node body center be (cx, cy, cz), the column vertical span be H, and N = 4 slots indexed k = 0..3 top to bottom. Top 2 slots (k=0,1) are backup; bottom 2 slots (k=2,3) are working. Each present bead sits at its slot center:

slotY(k) = cy + H/2 - (k + 0.5) * (H / N) // k=0 top … k=3 bottom slotPos(k) = (cx, slotY(k), cz) // slots 0,1 = backup[0],backup[1] ; slots 2,3 = working[0],working[1] // bottom slot k=3 == end of working (pops next); working stored [1,0] → pops 0 then 1

Present-and-value per slot — working occupies the bottom 2 slots, bottom-aligned as it depletes; backup occupies the top 2:

present(k=0,1) = (k < len(backup)) // backup slots present(k=2,3) = (k - 2 ≥ 2 - len(working)) // working, bottom-aligned valueAt(backup slot k) = backup[k] valueAt(working slot k) = working[len(working) - (4 - k)]
4 · Slide-down transition (refill)

On refill, the 2 backup beads slide from the top slots into the working slots, and 2 new backup beads appear at the top. The slide is animated at human speed with a transition parameter t ∈ [0,1] advanced by Go's clock — the same human-speed slowdown used for in-flight wire beads:

beadY(t) = lerp(slotY(kFrom), slotY(kTo), t) // backup bead slot 0 → slot 2, slot 1 → slot 3 (kFrom → kTo, slide down) // new backup beads fade/appear at slotY(0), slotY(1) // Go advances t each Update tick from the same clock that paces wire beads (human-speed) // the refill IS this slide, begun the moment working empties
5 · Bead value → style (existing convention, unchanged)

Reuse VALUE_BEAD_STYLE in scene-content.tsx: value 0 = {fill:#ffffff, ring:#000000} (white fill, black ring); value 1 = {fill:#000000, ring:#000000} (black fill, black ring). The interior beads carry the same value field, so TS styles them identically with no new style table.

The Go-authoritative streaming contract.

Node 1's Update() does the geometric calculation of the interior bead positions in the SAME step as the array mutation (pop / refill) — the array change and the geometry change are one coupled operation. Go then sends those geometric changes to TS. This is the same Go-authoritative-geometry pattern as node-geometry, edge-geometry, and the wire-bead position stream — just extended to node 1's interior:

Go → TS (extend the position trace stream — INCREMENTAL, not a re-broadcast of all 4) On a pop: array shrinks by one → Update recomputes interior geometry (popped bead gone, remainder repositioned) → sends the CHANGE. On a refill: Update computes the SLIDE geometry and streams the moving beads' positions OVER the slide duration on Go's clock — per-frame, human-speed, exactly like the existing in-flight wire-bead position stream. Go sends ONLY what changed — NOT a per-node "send all 4 beads every frame" batch. TS RENDERS the interior beads from this stream. TS computes NO geometry, no interpolation, no interior layout math (consistent with check-ts-computes-no-geometry). Trace event: node-bead — interior-bead geometry/position event keyed by node id + slot, payload = bead value (0|1) + position. Slots into the trace stream + pump.ts like the other geometry/position events. node-bead must be added to the trace-kind allowlist on BOTH Go and TS sides in lockstep (the trace-kind-parity guard enforces it), same as the other trace kinds.
Resolved Event kind name: node-bead. The interior-bead geometry/position trace event is kind node-bead (keyed by node id + slot, payload = value 0/1 + position). node-bead is added to the trace-kind allowlist on BOTH Go and TS sides in lockstep (the trace-kind-parity guard enforces it), like the other trace kinds. The per-slot keying, the incremental semantics, and where it slots into the trace stream / pump.ts are all settled. No open items remain.
