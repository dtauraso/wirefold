Wirefold — Design Spec (draft, branch-local)
Node 1 — Interior Depleting / Refilling 4-Bead Buffer

What changes, and why.

Node 1 is the Input node (Go kind Input, nodes/input/node.go). Today it holds an Init array (e.g. [0,1]) and, in the feedback ring, advances a read index i by the downstream feedback step (0 = hold, 1 = advance, wrapping) and emits Init[i] on its output wire 1To2. The interior of the node body is currently inert.

The new design turns the interior into a visible double-buffer: two arrays, each [1,0] — a working array (the bottom row) and a backup array (the top row), 4 beads total, arranged as 2 rows of 2 (a 2×2 grid) inside the node body — top row = backup [1,0], bottom row = working [1,0]. Sending pops the END of the working (bottom) row (the popped value rides out as a bead on 1To2). "Empty" means the working (bottom) row (length 2) is empty — not all 4 beads gone. When working empties (after its 2 pops) the whole top row slides down into the bottom-row slots (becoming the new working row) and a new top row appears (backup refilled to [1,0]) — an animated refill, back to 4 (2 rows). The 4 interior beads are a faithful, frame-accurate view of the buffer's contents.

Double-buffer: 2 rows of 2 (2×2 grid). Top row = backup=[1,0], bottom row = working=[1,0] = 4 beads. Pop: on feedback i==1, pop the END of the working (bottom) row → one bead removed, its value launches on 1To2. i==0 = hold. End-popping [1,0] sends 0 first, then 1 (intended; array stored [1,0] for this). No reversed backup. Empty: "empty" = working (bottom) row (length 2) is empty, after its 2 pops. Refill: the whole TOP row slides DOWN into the bottom-row slots (becomes new working) + a NEW top row appears (backup → [1,0]) — animated at human speed on Go's clock. Geometry: Input.Update() does the array mutation AND the interior bead geometry in ONE coupled step, then streams the INCREMENTAL changes; TS renders them and computes no geometry.

Double-buffer: working [1,0] + backup [1,0], pop-from-end, slide-down refill.

Two Go []int slices, each [1,0], back the buffer — 4 beads total:

  • working []int = {1,0} — the live, depleting slice (the bottom row of the 2×2 grid). Sending removes its end element by reslice — read the end value with working[len(working)-1], then shrink with working = working[:len(working)-1] (guarded by len(working) > 0) — and that value is emitted as a bead on 1To2. Removing the end of {1,0} sends 0 first, then 1 — intended; the slice is stored {1,0} precisely so end-removal yields that order.
  • backup []int = {1,0} — the top row. On refill the whole top row slides down into the bottom-row (working) slots via working = backup; a new top row appears via backup = []int{1,0}.
The driving signal — feedback value i

The feedback value i read from downstream (on FeedbackIn, produced by the downstream ChainInhibitor) drives the buffer. Its meaning is redefined for this design:

i == 1 → pop the end of working[] and send that value as a bead // advance i == 0 → do nothing (hold) // hold

if len(working) > 0 { // guard before indexing the end v := working[len(working)-1] // read the end value working = working[:len(working)-1] // reslice: drop the end emit(v) on 1To2 // value rides out as a bead }
Refill — when working (bottom row, length 2) is empty

"Empty" means the working (bottom) row is empty (after its 2 pops) — not all 4 beads gone. When that happens, the whole top row slides down into the bottom-row slots (becoming the new working row) and a fresh top row appears:

if len(working) == 0 { // working's 2 beads both removed working = backup // the whole TOP row slides DOWN into the bottom-row slots backup = []int{1, 0} // a NEW top row appears } // back to 4 beads (2 rows); slide animated at human speed (see Animation tab)
Settled — pop order. Each array is stored [1,0] and end-popped, so the emitted sequence is 0 then 1. This is the intended order; the storage [1,0] is chosen specifically so end-pop yields it. No reversed-backup is required.
Settled — refill. Refill triggers the moment working (bottom row, length 2) empties. The whole top row slides down into the bottom-row slots; a new top row ([1,0]) appears. Both rows are length 2, so the buffer is always 4 beads (2 rows).

4 beads in 2 rows: consume bottom row (working) → top row slides down → new top row appears.

The node body shows 4 beads as 2 rows of 2 (a 2×2 grid). The top row is the backup array ([1,0]); the bottom row is the working array ([1,0]). The buffer is consumed from the bottom row; the bottom row's last (rightmost) slot corresponds to the end of working — the element that pops next.

  • As the 2 working beads in the bottom row are consumed (two pops, i==1 twice — sending 0 then 1), the bottom row empties and working is empty.
  • The whole top row slides down into the vacated bottom-row slots, becoming the new working row (this slide IS the refill).
  • A new top row appears — backup back to [1,0]. Buffer is 4 again (2 rows).

Beads disappear on pop and the slide-down refill happens the moment working empties. This must coincide exactly with the buffer's contents and length at each step. The grid is a faithful render of the array state — never a free-running animation. The slide moves a whole row down (not individual stacked beads), and is animated at human speed, timed on Go's clock with the same human-speed slowdown used for in-flight beads.

No index i — array state, pop-from-end, row/col slot positions, row-drop — computed in Go's Update().

This project is Go-authoritative for geometry: TS computes no positions. Node 1's Update() does the array mutations and the geometric calculation of the 4 interior bead positions, then streams them. The formulas below are proposed and use real code-tied names.

Removed The persistent index variable i is gone. There is no read-head counter and no wrapping formula (i % len) anywhere in Update(). The working array IS the state: the end of the array is the next value to emit. Pop-from-end replaces indexed access; the feedback signal gates pops directly against array length; nothing about an index persists between ticks.
1 · Pop decision from feedback signal (no index)

The feedback signal gates a pop directly against the array — no counter is read or advanced:

s := FeedbackIn.TryRecv() // downstream step, 0 or 1 if s == 1 && len(working) > 0 { // remove the end — the end IS the value v := working[len(working)-1] // read the end value (guarded) working = working[:len(working)-1] // reslice: drop the end ToReadGate.TrySend(v) // bead rides out on 1To2 } // s == 0: hold, do nothing
2 · Refill condition — slide the top row down + new top row
if len(working) == 0 { // working (bottom row, length 2) drained by its 2 pops working = backup // the whole TOP row slides DOWN into the bottom-row slots backup = []int{1, 0} // a NEW top row appears } // each row is length 2; buffer is always 4 beads (2 rows)
3 · Slot-position formula — ROW × COLUMN (2 rows of 2)

Slots are a 2×2 grid: a row ∈ {0 = top/backup, 1 = bottom/working} and a col ∈ {0, 1}. Let the node body center be (cx, cy, cz), the row pitch (vertical offset between the two rows) be rowGap, and the column pitch be colGap. Each present bead sits at its grid slot center:

slotPos(row, col) = ( // row 0 = top/backup, row 1 = bottom/working cx + (col - 0.5) * colGap, // col 0 left, col 1 right cy + (0.5 - row) * rowGap, // row 0 above center, row 1 below cz) // row 0 = backup[col] ; row 1 = working[col] // bottom-row end (row 1, col len(working)-1) pops next; working stored [1,0] → pops 0 then 1

Present-and-value per slot — working occupies the bottom row, filling from col 0 and depleting from its end; backup occupies the top row:

present(row=0, col) = (col < len(backup)) // top row (backup) present(row=1, col) = (col < len(working)) // bottom row (working), end-popped valueAt(row=0, col) = backup[col] valueAt(row=1, col) = working[col]
4 · Slide-down transition (refill) — the top row drops one row offset

On refill, the whole top row slides from row 0 into row 1 (down by one row's vertical offset, rowGap), and a new top row appears at row 0. The slide is animated at human speed with a transition parameter t ∈ [0,1] advanced by Go's clock — the same human-speed slowdown used for in-flight wire beads:

beadY(col, t) = lerp(slotY(row=0), slotY(row=1), t) // whole top row drops one rowGap (both cols together) // top-row bead (row 0, col) → (row 1, col), moving down by rowGap; cols keep their x // new top-row beads fade/appear at slotPos(row=0, col) for col=0,1 // Go advances t each Update tick from the same clock that paces wire beads (human-speed) // the refill IS this row slide, begun the moment working empties
5 · Bead value → style (existing convention, unchanged)

Reuse VALUE_BEAD_STYLE in scene-content.tsx: value 0 = {fill:#ffffff, ring:#000000} (white fill, black ring); value 1 = {fill:#000000, ring:#000000} (black fill, black ring). The interior beads carry the same value field, so TS styles them identically with no new style table.

The Go-authoritative streaming contract.

Node 1's Update() does the geometric calculation of the interior bead positions in the SAME step as the array mutation (pop / refill) — the array change and the geometry change are one coupled operation. Go then sends those geometric changes to TS. This is the same Go-authoritative-geometry pattern as node-geometry, edge-geometry, and the wire-bead position stream — just extended to node 1's interior:

Go → TS (extend the position trace stream — INCREMENTAL, not a re-broadcast of all 4) On a pop: array shrinks by one → Update recomputes interior geometry (popped bead gone, remainder repositioned) → sends the CHANGE. On a refill: Update computes the SLIDE geometry and streams the moving beads' positions OVER the slide duration on Go's clock — per-frame, human-speed, exactly like the existing in-flight wire-bead position stream. Go sends ONLY what changed — NOT a per-node "send all 4 beads every frame" batch. TS RENDERS the interior beads from this stream. TS computes NO geometry, no interpolation, no interior layout math (consistent with check-ts-computes-no-geometry). Trace event: node-bead — interior-bead geometry/position event keyed by node id + slot, payload = bead value (0|1) + position. Slots into the trace stream + pump.ts like the other geometry/position events. node-bead must be added to the trace-kind allowlist on BOTH Go and TS sides in lockstep (the trace-kind-parity guard enforces it), same as the other trace kinds.
Two bead kinds — matched pair. node-bead (interior) has a counterpart edge-bead (a bead traveling ON an edge/wire — the in-flight bead). edge-bead is today's position trace event being RENAMED. So the two bead trace-event kinds are node-bead (inside a node) and edge-bead (on a wire) — counterparts.
Resolved Event kind name: node-bead. The interior-bead geometry/position trace event is kind node-bead (keyed by node id + slot, payload = value 0/1 + position). node-bead is added to the trace-kind allowlist on BOTH Go and TS sides in lockstep (the trace-kind-parity guard enforces it), like the other trace kinds. The per-slot keying, the incremental semantics, and where it slots into the trace stream / pump.ts are all settled. No open items remain.
