Node 1 — Interior Depleting / Refilling 4-Bead Buffer

What changes, and why.

Node 1 is the Input node (Go kind Input, nodes/input/node.go). Today it holds an Init array (e.g. [0,1]) and, in the feedback ring, advances a read index i by the downstream feedback step (0 = hold, 1 = advance, wrapping) and emits Init[i] on its output wire 1To2. The interior of the node body is currently inert.

The new design turns the interior into a visible depleting / refilling buffer backed by two arrays (a working array and a backup array) and renders 4 beads stacked in a vertical column inside the node body. Sending pops the end of the working array (the popped value rides out as a bead on 1To2); when the working array empties it is refilled from the backup. The 4 interior beads are a faithful, frame-accurate view of the working array's contents.

Before: Init[i] with a wrapping read index; interior inert. After: working[] (pop from end) backed by backup[]; refill working ← backup when empty. Interior column of 4 beads = live view of working[]; beads disappear on pop, reappear on refill. Go computes the array ops AND the 4 interior bead positions in Update(); TS renders the streamed positions.

Working array, backup array, pop-from-end, refill-when-empty.

Two arrays back the buffer:

  • working []int — the live, depleting array. Sending pops its last element (the end), and that popped value is emitted as a bead on 1To2.
  • backup []int — holds the initial data (a copy of Init). It is never popped; it is the source for refills.
The driving signal — feedback value i

The feedback value i read from downstream (on FeedbackIn, produced by the downstream ChainInhibitor) drives the buffer. Its meaning is redefined for this design:

i == 1 → pop the end of working[] and send that value as a bead // advance i == 0 → do nothing (hold) // hold
Pop
v = working[len(working)-1] // last element = "the end" working = working[:len(working)-1] // shrink by one emit(v) on 1To2 // popped value rides out as a bead
Refill — when working is empty

When the working array becomes empty, it is refilled from the backup's initial data:

if len(working) == 0: working = copy(backup) // working ← backup (the initial data) // backup is unchanged; it always holds the initial data
Assumption — confirm Refill timing. This spec refills after a pop that empties working (so the next i==1 pops from a freshly full array). The alternative is to refill before a pop when empty (lazy/just-in-time). These differ by one frame in when the top-2 beads reappear. Pick one.
Assumption — confirm Pop order vs. emitted sequence. Popping from the END emits working in reverse of its stored order. If the desired emitted sequence is the initial data in forward order, backup must be stored reversed (or working pushed in reverse at refill). Confirm the intended emitted value order.
Assumption — confirm Backup length vs. 4 beads. The interior shows exactly 4 beads. If len(backup) ≠ 4, the column needs a rule (show last 4 of working? pad empty slots?). This spec's diagrams assume len(backup) == 4.

4 beads: consume bottom 2 → top 2 drop down → top 2 refill.

The node body shows 4 beads in a vertical column. Slots are indexed top (slot 0) to bottom (slot 3). The buffer is consumed from the bottom; the bottom slot (slot 3) corresponds to the end of working — the element that pops next.

  • As the bottom 2 beads are consumed (two pops, i==1 twice), the bottom 2 slots empty.
  • The top 2 beads drop down into the vacated bottom slots.
  • The now-empty top 2 slots are refilled — beads reappear from the backup / initial data.

Beads disappear on pop and reappear on refill, and this must coincide exactly with working's contents and length at each step. The column is a faithful render of the array state — never a free-running animation.

No index i — array state, pop-from-end, slot positions, drop-down — computed in Go's Update().

This project is Go-authoritative for geometry: TS computes no positions. Node 1's Update() does the array mutations and the geometric calculation of the 4 interior bead positions, then streams them. The formulas below are proposed and use real code-tied names.

Removed The persistent index variable i is gone. There is no read-head counter and no wrapping formula (i % len) anywhere in Update(). The working array IS the state: the end of the array is the next value to emit. Pop-from-end replaces indexed access; the feedback signal gates pops directly against array length; nothing about an index persists between ticks.
1 · Pop decision from feedback signal (no index)

The feedback signal gates a pop directly against the array — no counter is read or advanced:

s := FeedbackIn.TryRecv() // downstream step, 0 or 1 if s == 1 { // pop & send one — no index, the end is the value v := working[len(working)-1] working = working[:len(working)-1] ToReadGate.TryEmit(v) // bead rides out on 1To2 } // s == 0: hold, do nothing
2 · Refill condition
if len(working) == 0 { working = append(working[:0], backup...) // working ← backup (initial data) } // backup ← initial data, set once at load
3 · Slot-position formula (4 beads)

Let the node body center be (cx, cy, cz), the column vertical span be H, and N = 4 slots indexed k = 0..3 top to bottom. Each present bead sits at its slot center:

slotY(k) = cy + H/2 - (k + 0.5) * (H / N) // k=0 top … k=3 bottom slotPos(k) = (cx, slotY(k), cz) // bottom slot k=3 == end of working (pops next) // a bead is present at slot k iff k maps to a live working[] element

Mapping array index to slot: with the bottom slot as the working end, the present beads occupy the bottom len(working) slots:

present(k) = (k ≥ N - len(working)) // bottom-aligned fill valueAt(k) = working[len(working) - (N - k)] // when present(k)
4 · Drop-down transition

On refill, surviving beads move from their old slot to a lower slot, and refilled beads appear in the top slots. With a transition parameter t ∈ [0,1] (driven by Go's clock, like wire beads):

beadY(t) = lerp(slotY(kFrom), slotY(kTo), t) // kFrom = slot before drop, kTo = slot after drop // refilled top beads: fade/appear at slotY(kTo) (kFrom = kTo, t snaps) // Go advances t each Update tick from the same clock that paces wire beads
5 · Bead value → style (existing convention, unchanged)

Reuse VALUE_BEAD_STYLE in scene-content.tsx: value 0 = {fill:#ffffff, ring:#000000} (white fill, black ring); value 1 = {fill:#000000, ring:#000000} (black fill, black ring). The interior beads carry the same value field, so TS styles them identically with no new style table.

The Go-authoritative streaming contract.

Go streams the interior bead positions/states the same way it streams wire-bead positions and node-geometry today; TS renders them and computes no geometry. This implies a new (or extended) trace event:

Go → TS (extend the position trace stream) interior-bead event { nodeId, beadIndex k (0..3), pos (x,y,z), value (0|1), present (bool) } emitted from Input.Update() per tick, alongside the existing node-geometry / pulse position events. TS renders each present interior bead at the streamed pos with the value-styled material. TS computes no slot positions, no drop interpolation, no present/empty decision — all of that is in Go.
Assumption — confirm New trace event shape. The exact event kind name and payload (one event per bead vs. one event per node carrying 4 bead slots) is a design point. Per-node batching (one interior event with a 4-slot array) is likely cheaper and keeps the column atomic per frame. Confirm preferred shape and where it slots into the existing trace stream / pump.ts switch.

What nodes/input/node.go needs.

All changes are local to the Input node and its trace emission. Ports and the feedback wiring are unchanged — only the meaning of the feedback step and the interior state/geometry are new.

Removed Delete the persistent index i from nodes/input/node.go. Remove the i/index field from the Node struct and every i % len(Init) / index-advance line from the Update() loop. The working and backup arrays are the only emission state — there is no counter to persist or wrap.
