Topology Tree · Go-Owned Read/Write

wirefold · design spec · branch task/persist-geometry-from-go-stream

Replace the monolithic topology.json (+ topology.scene.json) with a directory tree of small files that Go reads and writes, and delete the editor's reload/restart machinery. Go owns the model and its persistence; the editor becomes a pure webview client of Go.

The problem today

The spec is a monolithic topology.json (plus a topology.scene.json sidecar). The editor is a VS Code custom text editor bound to that document. It listens to onDidChangeTextDocument; on an external or structural change it reloads the webview and restarts the Go process (kill + respawn go run .).

  • Why it restarts: Go builds its whole network at startup (LoadTopology) and has no in-process "reload topology" command. An external file change carries no diff — so the editor can only rebuild Go from scratch.
  • The damage: any writer that is not the editor (notably Go persisting geometry) trips the file-watch → reload → restart loop, causing loops and jolts.
  • Persistence flows backwards through TS: flowToSpec serializes React Flow node data and the editor saves the document. Go only os.ReadFiles — it never writes. So TS must keep a parallel copy of every Go-owned value (positions, the new port anchor) just so the save can read it.

The inversion

Go already owns the model. It should own the model's persistence too. Once Go takes CRUD edits directly, Go always has the diff, updates its model, and writes exactly the file(s) that changed. The editor never asks VS Code "did the document change" and never serializes geometry.

This is a design artifact — intent, structure, flow, and diagrams. No code yet.

Read / Write / CRUD flow

Go is the sole writer

Go reads the tree at startup, streams the model to the editor, and on each CRUD op writes exactly the file(s) that op affects. Nothing else writes the tree.

Read at startup

A tree-walker replaces os.ReadFile of the single monolith: walk topology/, parse each small file, build the in-memory model, then stream it to the editor over the existing trace/geometry stream (bead positions, node events, edge curves, shading params). Go walks a tiny dataset fast.

Write on CRUD

Each edit op writes its own file(s). Go knows what changed (it received the op), so it never rewrites the monolith and never asks TS for the value.

Per-op → file(s) written

Edit opFile(s) Go writesEffect
node-moveview/nodes/<id>.jsonupdate position {x, y}

Sole-writer invariant. Because every mutation arrives as a CRUD op, Go always has the diff and is the only process that writes the tree. There is no external-change detection because there is no external writer to detect.
Bridge stays fire-and-forget. TS → Go is one geometry-CRUD edit message (op = create/update/delete/fade) plus play/pause — no await, no Promise, no delivery signal. Go → TS remains the trace stream. This work adds no new bridge channel; it removes a parallel one (the TS save path).

Editor model

webview client of Go

The editor stops being a VS Code custom text editor bound to a document. It becomes a pure webview client: it renders Go's stream, sends CRUD fire-and-forget, and never serializes geometry or watches a document.

What the editor does now

  • Renders Go's stream. Bead positions, node events, edge curves, shading params come in; the webview plots them. (Unchanged from today's Go-authoritative model.)
  • Sends CRUD fire-and-forget. User edits become one edit message; the editor does not await Go and gets no delivery signal.
  • Reads the tree only at first open — or relies entirely on Go's startup stream (OPEN: which of the two).
  • Owns the viewpoint. Camera, orbit/pan/zoom stay in TS, as today.

Launch: open the webview on the folder

A command opens the webview panel on the workspace folder (the topology/ tree) instead of opening topology.json as a custom text editor. Because the editor is a webview, VS Code never opens the many small files as documents — the file count is a non-issue. The Explorer tree is lazy, Go walks a tiny dataset fast, and git diffs get more surgical.

No document watching. The editor never asks VS Code "did the document change." There is nothing to watch — Go is the sole writer and the editor learns of every change from the stream it caused.

Removed

deleted entirely

Once Go takes CRUD (always has the diff), updates its model AND writes the tree, and the editor renders the stream, the entire external-change-detection apparatus is unnecessary.

What goes away

RemovedWhy it existedWhy it's now unnecessary
onDidChangeTextDocument handlerdetect external/structural file changesno external writer; Go is the sole writer
structuralKey gatingdecide if a change needs a rebuildGo has the diff per op; never rebuilds wholesale
send()-reload of the webviewrefresh after a structural changeeditor renders the stream continuously
stopAndAwait() + run() RESTARTno in-process reload; rebuild from scratchGo updates its live model in place on each op
lastAppliedVersion suppressionignore the editor's own save echono save path to echo
flowToSpec / save (geometry)serialize Go-owned values back to diskGo writes its own values directly
manifest file—there is none; the tree is self-describing

Why no restart

Current sequence: write → onDidChangeTextDocument → webview reload → ~300 ms → Go stop + respawn. It existed because of two missing things: an external change carried no diff, and Go had no in-process reload — so the only recovery was rebuild-from-scratch.

Both disappear once Go takes CRUD directly: Go always has the diff, updates its model in place, and writes the affected tree file(s). The editor renders the resulting stream. No external change ever occurs, so there is nothing to detect, reload, or restart.

OPEN follow-up. The one case the restart used to cover is a human hand-editing files outside the editor. That is not part of this work. Options: accept a manual relaunch, or add an optional in-process reload op later. Stated as an open follow-up, not in scope.

Phases

A build plan in four phases. Each phase is independently shippable; the editor keeps working throughout.

#PhaseScope
1Tree format + Go tree-readerDefine the topology/ tree shape; replace os.ReadFile of the monolith with a tree-walker that builds the model and streams it. Editor unchanged.
2Go tree-writer on CRUDOn each edit op, Go writes the affected file(s) (node-move, port-anchor, wire create/delete, fade). Go becomes the sole writer.
3Editor webview-on-folder; delete reload/restartOpen the webview on the topology/ folder; delete the custom text editor, onDidChangeTextDocument, structuralKey, send()-reload, stopAndAwait()+run() restart, lastAppliedVersion, and the flowToSpec/save geometry path.
4Fold scene into view/Fold topology.scene.json into view/scene.json (gitignored) alongside view/fades.json and view/nodes/.
Ordering rationale. Reader before writer (Phase 1 before 2) so the model can round-trip on disk before Go mutates it. Editor cleanup (Phase 3) only after Go is the sole writer, so deleting the save path never strands a value.
