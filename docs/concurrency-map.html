Concurrency map

Goroutines here are the architecture, not a way to serve load. One per node, one per edge, one per wire — so the count scales with the graph you draw, and every shared boundary between them is enumerable.

Why this is unusual

In a typical Go service, goroutines are short-lived — one per request, sharing almost nothing, handing off through a channel and dying. The race surface is small and uniform.

Here they are long-lived and graph-shaped. They run the whole session, they both message-pass and touch shared state, and their number is set by the topology on screen.

The ownership rule

A node writes only itself. No coordinator, no worklist. Almost all state is therefore protected by nothing at all — which is what makes the handful of sync sites worth enumerating: each one marks a place that rule is deliberately crossed.

How each was settled

Not by reading the comment. Each guard was removed and the suite re-run under -race. A guard that could be deleted with nothing noticing was treated as unproven, not as safe.

geomMu is gone

Its struct mixed write-once identity with per-move mutable state, so a reader that only wanted Kind had to guard fields it never touched. Splitting the struct removed the hazard instead of guarding it.

Three comments were wrong

Each named the wrong pair of goroutines. That is why every surviving guard now cites a test that fails when it is removed — prose was not holding.

gesture.go · quantized_move.go · node_mover.go

9,345 of 9,600 — neighbour messages that were being silently dropped under load, under an earlier direct-to-inbox lossy sender. That was fixed first by routing the fan through a shared outbox drained by a dedicated sender goroutine, and later by today's per-direction channels with sender-side retain-and-retry — no queue, no lock, no dedicated sender. The drop is still zero.

One writer moves

Dragging X changes X’s position and nothing else’s. The neighbour is told its stored description is stale; it stays exactly where it was.

Why the edge changes length

Because an edge is drawn between two centres and one of them moved. The edge has its own goroutine holding both endpoints — neither node draws it.

Per drag vs per move

RootMove runs on every pointer-move, not once per drag. Two separate bugs came from assuming otherwise; anything drag-scoped is anchored at the pending→dragging edge instead.

paced_wire.go · ports.go · node_mover.go

Go times its own delivery

The wire's own goroutine reads its own clock copy, records the tick a bead was placed, and sleeps cycle by cycle until it has crossed. Every wire reads its own clock independently. Nothing on the render side tells Go a bead arrived.

Why this lock is gone

The wire is no longer a passive struct three goroutines reach into — it IS the edge mover's own goroutine (the same one that already existed to revise in-flight geometry on a node move, so goroutine count is unchanged). The source and destination now only ever touch its channel ends (Send/Recv), never inflight directly, so there is nothing left to lock.

Resolved lead

The old open question — splitting inflight from delivered by owner — is moot: there is only one owner now, so there was nothing to split.

That was exactness re-imported after it had already been dropped: the error at that subtraction is the same 1–2% accepted everywhere else. RealClock.mu is deleted, not reduced.

Sized, not feared

1–2%of a wire / change

Each goroutine hears a speed change when it next wakes, so the spread is about one cycle. On a control operated by hand, that is imperceptible.

Offsets are signed by direction, so speeding up and slowing down partly cancel rather than accumulating. Reaching the clamp that would snap a bead to the end of its wire needs roughly sixty speed changes.

What got dropped

An earlier draft carried an “effective at” instant on every transition so all copies banked identically. It was machinery for a problem that measurement showed was not there — and it sent a clock backward over ticks it had already acted on.

The real cost

Not timing, and no longer a correctness cliff. A copy that hears late is off by the band already accepted, so no timestamps and no special handling. What is left is plumbing: every paced loop grows a second poll.

Trace is different, but its lock still went

Trace cannot be copied the way the clock was — every event has to land in one place to become one buffer, so copies would produce partial streams and no whole picture. That claim was true, but it was never what Trace.mu guarded: in production nothing ever raced events (the drain was already sole writer). The lock's real job was serializing Breadcrumb's direct sink writes and surviving a shutdown that didn't wait for every goroutine; fixing both removed it. See the mutexes tab.
