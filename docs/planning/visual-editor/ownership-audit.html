Ownership audit — Go runtime vs the editor

The organizing principle

Route each calculation to the part that owns the live truth. Go owns the diagram — the one clock, traversal timing, bead transport + delivery, the bead's world position (emitted every ~16 ms), and the node position (held in Go, re-emitted as node-geometry on every move). The editor (TS) owns the viewpoint — camera, projection, raycast-picking, GPU render. TS only renders what Go streams; it holds no position data of its own. The interaction sends a fire-and-forget edit·update to Go, which updates its held position and re-emits the geometry.

Go runtime holds the node position, runs the clock, paces traversal, transports + delivers beads, and emits: bead world positions, node-geometry (center + port world pos/dir) on startup and on every move, and edge-geometry (wireSegment) on load and on every node-move. TS editor renders Go's stream — plots bead position from Go's emitted world pos, draws wire from Go's edge-geometry store, renders the node body from Go's node-geometry. TS owns camera/projection/picking and the drag interaction (sending the fire-and-forget update). The bridge is two channels: Go→TS trace stream (bead positions, node events, geometry, node-geometry, arrive); TS→Go spec save/load + one `edit` op + play/pause — fire-and-forget, no await.

Three columns: who computes what

Go owns the diagram. Node positions, geometry at rest, the clock, every bead's world position. nodes/Wiring/clock.go, nodes/Wiring/paced_wire.go, nodes/Wiring/stdin_reader.go, Trace/Trace.go.
The editor owns the viewpoint + interaction. Render, camera, picking, and the drag gesture (fire-and-forget edit·update to Go). pump.ts is the only TS↔Go boundary; everything downstream of it is plotting.

The drag case — Go holds the position, TS renders from Go's stream

The correct model: Go holds the node position. A drag sends a fire-and-forget edit·update to Go; Go updates its held position, re-emits node-geometry (node body + port positions), and re-emits geometry (edge wireSegment) for every incident edge. TS renders node body, wire, and bead all from Go's stream — no local position state in TS.

What the bug was

The bug was a missing RETURN emit in applyNodeMove: it updated Go's held position and re-emitted edge geometry, but never re-emitted the node's node-geometry. So TS had no fresh node center — the node body looked stuck on screen while the wire and bead updated. Fix: applyNodeMove now calls emitNodeGeometry after updating the held position, so the body, wire, and bead all update from Go's stream in the same round-trip.

The rules that fence this surface

Fire-and-forget. The TS→Go send is never awaited — no Promise chain, no request/response, no delivery signal. webview→host is vscode.postMessage; host→Go is runner.writeStdin; neither blocks.
Drift guards (CI). tools/check-ts-computes-no-geometry.sh — TS plots/draws, computes no bead positions or edge curves. tools/check-no-await-on-bridge.sh — the TS→Go path stays fire-and-forget. tools/check-trace-kind-parity.sh — the kind set in trace-kinds.ts equals pump.ts's switch labels (both derived from TraceEventKinds). Also: check-message-kind-parity.sh (top-level msg kinds), check-ts-shading-from-go.sh (shading values from Go).
