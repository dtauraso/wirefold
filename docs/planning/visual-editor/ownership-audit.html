Ownership audit — Go runtime vs the editor

The organizing principle

Route each calculation to the part that owns the live truth. Go owns the diagram — the one clock, traversal timing, bead transport + delivery, and the bead's progress (the fraction t along a wire). The editor (TS) owns the viewpoint — camera, projection, raycast-picking, GPU render — and the live placement during interaction: while a node is being dragged, the editor's local React Flow node.position is the live truth for the node body, the wire endpoints, and where the bead sits.

Go runtime holds geometry at rest, runs the clock, paces traversal, transports + delivers beads, and emits each bead's fractional progress t. TS editor renders Go's stream, owns camera/projection/picking, and during a drag owns the live node position, wire endpoints, and bead placement (lerp of Go's t over local endpoints). The bridge is two channels: Go→TS trace stream (positions+fraction, node events, geometry, node-geometry); TS→Go spec save/load + one `edit` op + play/pause — fire-and-forget, no await.

Three columns: who computes what

Go owns the diagram. Geometry at rest, the clock, every bead's progress. nodes/Wiring/clock.go, nodes/Wiring/paced_wire.go, Trace/Trace.go.
The editor owns the viewpoint + live placement. Render, camera, picking, and the dragged node's live position. pump.ts is the only TS↔Go boundary; everything downstream of it is plotting and viewpoint.

The drag case — routing each piece to its live owner

The recent refinement: while a node is being dragged, the editor owns the live node position, the wire endpoints, and the bead's placement; Go still owns the bead's progress (fraction t) and delivery. The bead rides the SAME live segment the tube draws, at Go's fraction.

During a dragOwnerSourceWhy here
Node body position Editor / local nodeWorldPos → node.position (geometry-helpers.ts) Sourcing the body from Go's emitted center made dragging invisible — Go re-emits only edge curves on node-move, not node-geometry, so the body read a stale center.
Wire endpoints Editor / local portWorldPos (scene-content.tsx SingleEdgeTube) Reading them from Go's edge-geometry store lagged a round-trip behind the body (already local), so the wire trailed the node.
Bead progress (fraction t) Go / t position event field f (paced_wire.go → Trace.go) Progress is timing, off the one clock; the editor must not invent it.
Bead placement Editor (lerp) mesh.position.lerpVectors(start, end, frac) Go's t lerped over the live local endpoints — the bead rides the exact segment the tube draws.
Bug-class note (both fixed). Computing from a stale source (sourcing the body / endpoints from Go's round-tripped geometry) caused wire lag. Conserving the wrong invariant (placing the bead by absolute distance instead of fraction t) caused a pulse race when the wire length changed mid-flight. The fix: editor owns live placement, Go owns fraction t — distance is re-derived from t over whatever the live endpoints currently are.
At rest vs. live. Go's emitted node-geometry is authoritative at rest and is always the source for port offsets + directions (which don't change when a node translates). The editor takes over only the live position during a drag. Legacy geometry (wireSegment) trace events + the edge-geometry store still exist but are no longer read for rendering — kept for the delete path.

The rules that fence this surface

Fire-and-forget. The TS→Go send is never awaited — no Promise chain, no request/response, no delivery signal. webview→host is vscode.postMessage; host→Go is runner.writeStdin; neither blocks.
Drift guards (CI). tools/check-ts-computes-no-geometry.sh — TS plots/draws, computes no bead positions or edge curves. tools/check-no-await-on-bridge.sh — the TS→Go path stays fire-and-forget. tools/check-trace-kind-parity.sh — the kind set in trace-kinds.ts equals pump.ts's switch labels (both derived from TraceEventKinds). Also: check-message-kind-parity.sh (top-level msg kinds), check-ts-shading-from-go.sh (shading values from Go).
