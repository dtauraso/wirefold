The edit round-trip — why 12 hops are needed

A geometry edit makes 12 hops: webview → host → Go → host → webview. After reading the actual code: all 12 are load-bearing. Nothing is removable. The floor is 12.

Correction. Earlier drafts of this audit claimed 2–4 removable hops (a re-validate at the host, hop 4, and at the webview, hop 11). Reading the code refuted it twice over:
• Hop 4 isn't a verbatim forward. The host's switch(msg.op) destructures per op (msg.entries, msg.target, msg.edges…) and re-serializes a different shape for Go. The parse is what gives typed field access; remove it and routing + field extraction both die.
• Hop 11 doesn't duplicate hop 9. Hop 9 parses Go's stdout line; hop 11 is an O(1) envelope type-guard on the postMessage. Different payloads, different boundaries. Both parse* functions are tag-checks, not deep re-validation — there was no fat to cut.

Why three runtimes — the root of the count

12 follows from one fact: the path spans 3 separate runtimes, and they exist for two unrelated reasons — neither negotiable.

Runtime 1

Webview (sandboxed iframe)

React + Three.js. Origin of the gesture and the final render target.

Forced apart by VS Code: a webview is a sandboxed iframe — no Node, no child_process, no stdio. It physically cannot reach Go.
Runtime 2

Extension host (Node)

The only runtime that can both talk to the webview (postMessage) and cp.spawn the Go binary with stdio pipes (runCommand.ts:177).

Exists purely as the bridge between the two boundaries below — the webview can't span them itself.
Runtime 3

Go network

Owns the one clock, runs the node/wire goroutines, computes every position.

Forced apart by the model: Go owns time. It must be its own process — that's substance, not plumbing.
↑↓ boundary A · postMessage (VS Code sandbox) ↑↓ boundary B · stdio pipe (Go owns clock)

A round trip transits each boundary twice (out + back) = 4 crossings. And every runtime it passes through must decode what arrived, act, and encode what leaves — that's the 8 work hops. 4 + 8 = 12. None of the 8 repeats another: each is a distinct decode / mutate / compute / route / render step.

The count, settled

12 hops — all fixed
8 work
4 crossings

Floor = 12. Removable = 0. Every hop either does irreplaceable work or transits a process boundary that can't be wished away.

The only structural lever is eliminating a runtime — and both are nailed down:

  • Merge webview + host? Can't. VS Code sandboxes the webview; only the Node host can cp.spawn Go. Boundary A and the host relay (hops 4, 9, 10) are the sandbox tax — imposed by the editor medium, not chosen. (Outside VS Code — say an Electron renderer with node integration — this boundary could vanish. Inside VS Code it's fixed.)
  • Merge host + Go? Can't, and shouldn't. Go is a separate runtime because it owns the one clock and computes all geometry. That's the model's substance; folding it into Node is exactly the drift MODEL.md forbids.

So 12 isn't bloat — it's 2 boundaries × 2 directions plus decode/act/encode at each runtime the data passes through. The honest minimization budget is zero hops. (Micro-tidying is still possible — e.g. hops 4→5 and 9→10 are each one logical act split across two modules — but that shortens call chains, not the hop count, and changes nothing structural.)

Bridge contract (CLAUDE.md / MODEL.md): the 2 crossings stay fire-and-forget; never add an await on the TS→Go send, never a new bridge message, and never fold Go's geometry/timing up into the host to "save a crossing." The crossings are load-bearing precisely because Go owns the one clock.
