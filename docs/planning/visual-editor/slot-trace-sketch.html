Slot Trace — Design Sketch

Extending the Go-to-webview trace pipeline so held slot values are visible during animation, not just fire flashes.

What the pipeline emits today

The Go runtime streams JSONL over stdout. The VS Code extension host reads it line-by-line and forwards each parsed object to the webview via postMessage. The pump (pump.ts) dispatches on kind and mutates the RF store.

Three event kinds exist (Trace/Trace.go:30-33):

  • recv — a node received a value on an input port (pump currently ignores this)
  • fire — a node's firing rule satisfied; triggers the flash animation
  • send — a node sent a value on an output port; triggers a pulse on the matching RF edge
Slots are implicit. Each node holds partial state between events (e.g. HasValue, HasChainInhibitor in ReadGateNode), but nothing in the trace says "slot X is now filled with value V". The webview can only see the flash when the node fires — not the accumulation before it.

Proposed slot event — JSONL schema

Schema

{
  "step":   <int>,          // monotonic ordinal from Trace.drain
  "kind":   "slot",
  "nodeId": <string>,      // node Name (same as node field on fire/recv events)
  "port":   <string>,      // input port name (e.g. "FromInput", "FromChainInhibitor")
  "phase":  "filled" | "empty",
  "value":  <int>          // present when phase="filled"; omit when phase="empty"
}

Example lines

{"step":4,"kind":"slot","nodeId":"readLatch0","port":"FromInput","phase":"filled","value":3}
{"step":5,"kind":"slot","nodeId":"readLatch0","port":"FromChainInhibitor","phase":"filled","value":1}
{"step":7,"kind":"slot","nodeId":"readLatch0","port":"FromInput","phase":"empty"}
The value field follows the same convention as recv/send events: value 0 is a valid signal, so "no value" is represented by omitting the field entirely (not by setting it to 0). This mirrors the hasValue sentinel already in Trace/Trace.go:45.

Lifecycle

A filled event fires immediately after a successful channel receive (same point as the existing recv event). An empty event fires after the node clears the slot — i.e., after its firing rule runs and it resets its local boolean flags (e.g. HasValue = false in ReadGateNode).

This lets the webview mirror the node's local state continuously, not just at fire time.

Emit site — central vs. per-kind

Option A: per-kind, inline in Update()

Each node's Update loop calls s.Trace.Slot(...) next to the existing Recv and Fire calls. ReadGateNode already shows the pattern at nodes/ReadGateNode/ReadGateNode.go:34,43: after s.Trace.Recv add s.Trace.Slot(name, port, "filled", v); after clearing the flag add s.Trace.Slot(name, port, "empty", 0).

  • Precise: the emit is collocated with the state change.
  • Requires touching every node kind.

Option B: central emit from the Wiring slot-fill primitive

The Go Wiring layer already has one place where all initial slot priming happens: nodes/Wiring/loader.go:135, function LoadTopology. That line is: 

edgeChan[label] <- slotVal  // loader.go:135 — pre-sends initial slot value into channel

This covers only startup priming, not runtime transitions. Runtime slot changes (filling during execution, clearing after fire) happen inside each node's Update loop — there is no shared Wiring primitive for that today. The nodes/Wiring/builders.go wires struct fields from spec data (tag-driven, lines 143-165) but does not touch runtime channel receives.

Recommendation: per-kind, with a convenience wrapper on Trace

Add Trace.Slot(node, port, phase string, value int) to Trace/Trace.go (alongside Recv/Fire/Send). Then add the two call sites to each node's Update. There are currently three node kinds with visible slot state:

  • nodes/ReadGateNode/ReadGateNode.go — HasValue, HasChainInhibitor
  • nodes/ChainInhibitorNode/ChainInhibitorNode.go — HeldValue (always filled; transitions on recv)
  • nodes/InputNode/InputNode.go — fires once per queue item; no partial-fill state

Central Wiring extraction would require a new abstraction that doesn't exist yet and would not cover the empty-after-fire half of the lifecycle. Per-kind is narrower and can be done without touching Wiring contracts.

The event vocabulary in Trace/Trace.go:30-33 is described as "closed". Adding KindSlot = "slot" there keeps the pattern consistent — one constant, one helper method, one marshal case.

Pump handler (pump.ts)

tools/topology-vscode/src/webview/rf/pump.ts currently dispatches on recv, fire, send. Add a slot case that writes into a per-node slots map on the RF store.

The RF store node data object is already mutable via rfSetNodes. Add a slots key that holds Record<port, {phase, value?}>: 

// pump.ts — add slot case (delta only; surrounding switch unchanged)

case "slot": {
  const { nodeId, port, phase, value } = event as SlotEvent;
  rfSetNodes((nodes) =>
    nodes.map((n) => {
      if (n.id !== nodeId) return n;
      const prev: SlotMap = (n.data as NodeData & { slots?: SlotMap }).slots ?? {};
      const next: SlotMap = {
        ...prev,
        [port]: phase === "filled"
          ? { phase: "filled", value: value ?? 0 }
          : { phase: "empty" },
      };
      return { ...n, data: { ...n.data, slots: next } };
    }),
  );
  return;
}

// Types (can live in messages.ts alongside TraceEvent):
export type SlotPhase = "filled" | "empty";
export type SlotEntry = { phase: "filled"; value: number } | { phase: "empty" };
export type SlotMap  = Record<string, SlotEntry>;

export interface SlotEvent {
  step: number; kind: "slot";
  nodeId: string; port: string;
  phase: SlotPhase; value?: number;
}
The slots map persists across events for the node — it is overwritten port-by-port, not replaced whole-object. This mirrors how lastFire is written: one field per event, rest unchanged.

TraceEvent in tools/topology-vscode/src/messages.ts is a discriminated union; add SlotEvent to the union there, and narrow the event as SlotEvent cast in the pump to a proper discriminant check.

Render — value badges on input ports

tools/topology-vscode/src/webview/rf/nodes/GenericNode.tsx renders input handles in renderPortHandles (line 100). Each <Handle> is currently just a dot. When a slot is filled, show a small monospace badge next to the handle showing the held value.

JSX delta — in renderPortHandles

// Before (existing line ~115):
return <Handle key={...} .../>;

// After — wrap in a fragment, conditionally render badge:
const slotEntry = slots?.[p.name];
const showBadge = isInput && slotEntry?.phase === "filled";
return (
  <React.Fragment key={`${side}-${p.name}`}>
    <Handle
      id={p.name} type={isInput ? "target" : "source"}
      position={SIDE_POS[side]}
      style={{ ...portStyle(side, pct, color), opacity: drag?.portName === p.name ? 0.4 : 1 }}
      onPointerDown={(e) => onPointerDown(e, p.name, curSide, curSlot)}
    />
    {showBadge && (
      <div style={badgeStyle(side, pct)}>
        {slotEntry.value}
      </div>
    )}
  </React.Fragment>
);

Badge style helper

function badgeStyle(side: Side, pct: number): CSSProperties {
  const iv = side === "left" || side === "right";
  const offset = side === "left" ? { left: 12 } : side === "right" ? { right: 12 } : {};
  return {
    position: "absolute",
    ...(iv ? { top: `${pct}%`, transform: "translateY(-50%)" } : { left: `${pct}%`, transform: "translateX(-50%)" }),
    ...offset,
    background: "#1a237e",
    color: "#fff",
    fontFamily: "monospace",
    fontSize: 9,
    padding: "1px 3px",
    borderRadius: 3,
    pointerEvents: "none",
    zIndex: 10,
    whiteSpace: "nowrap",
  } as CSSProperties;
}

Prop threading

renderPortHandles currently receives (inputs, outputs, def, drag, onPointerDown). Add slots: SlotMap | undefined as the last argument and pass (data as NodeData & {slots?: SlotMap}).slots from GenericNode's render body. No store subscription change needed — the slots map already lives on n.data, which flows in via the existing NodeProps<NodeData>.

The dashed outline for empty slots is optional — it makes the waiting state visible without adding a second event kind. Alternatively just show nothing for empty and let the absence of a badge communicate emptiness. Prefer the dashed outline only if David finds it useful during animation scrubbing.
