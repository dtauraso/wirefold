Port Ring-Anchor Layout

branch: task/port-ring-anchors  ·  docs/planning/visual-editor/port-ring-anchors-spec.html

The redesign replaces today's port placement — a side + slot(0|1|2) grid (4 sides × 3 slots) PLUS ad-hoc free anchor 3D vectors that override side/slot — with ONE uniform ring of evenly-spaced anchor positions on the node's torus.

C = 2·π·R N = floor( C / (d + p) )

Where R = torus major radius (circle traced by the center of the tube around the node), d = port anchor diameter, p = small padding.

Ring capacity N scales with node size: because R is derived from each kind's node_dims, a larger node gets a larger ring and therefore more anchor ids. N varies by kind.

N IS the capacity of the ring — a fixed list of visual anchor positions, each assigned an id (0..N-1). A per-node map { portName → anchorId } places each named port (FromLeft, ToNext0, FeedbackOut, In, Out, FromPrev, FromRight, ToPassed, …) onto one anchor id; unmapped ids are empty slots. Node logic addresses ports by name; the name resolves to an anchor id for rendering and edge attachment. This retires BOTH the free anchor vector and the side+slot grid.

Geometry is Go-owned — Go computes the N positions and streams them; TS renders, computes none.

d and p are tunable constants (defaults: d = 8 wu, p = 2 wu), adjusted by eye in the editor. The model N = floor(2·π·R / (d + p)) is parametric in d, p, and R.

Current model: ports placed by side (top/bottom/left/right) + slot (0|1|2, at 25/50/75% along the side), with an optional free anchor 3D vector that OVERRIDES side+slot entirely (port_geometry.go:157).

Both are ChainInhibitor. Per-instance inconsistency is the motivation for the redesign.

Most ports also carry authored anchor vectors that bypass the slot grid — ad-hoc per instance.

  • Every node has a fixed list of N evenly-spaced visual anchor positions around its torus ring, each with an id (0..N-1). N is the capacity; it is derived from geometry: floor(C / (d + p)).
  • Ports are NAMED roles (FromLeft, ToNext0, FeedbackOut, In, Out, FromPrev, FromRight, ToPassed, …). A per-node map { portName → anchorId } assigns each named port to one anchor id. Unmapped ids are empty positions.
  • Node logic addresses ports by name; the name resolves to an anchor id for rendering and edge attachment.
  • Positions are uniform across all instances of a node kind — no per-instance drift.
  • An edge attaches to the anchor id its port name maps to.

Was: fixed 3-per-side grid + free anchors — New: one geometry-derived evenly-spaced ring of anchor ids with named ports mapped onto them.

Old model = free-drag a port to any continuous angle around the torus. New model = SWITCH which discrete anchor id a port name occupies — snap between the N anchor ids, not slide.

The drag gesture becomes a discrete reassign: switching an edge's port = changing that port name's anchorId in the { portName → anchorId } map (no continuous angle). The bridge edit op carries the port name and its new anchorId. Message-shape design is still open (Q 5).
