Node Runs Its Own Outgoing Edges

branch: task/node-runs-own-edges  ·  docs/planning/visual-editor/node-edges-goroutine-spec.html

A node emits one bead per fire; the bead rides a wire the node owns, advanced by the node's own goroutine.

  • A node emits one bead per fire — not a 2s/400ms train. No trainDurationMs, no beadSpacingMs, no persistent re-send.
  • The bead travels on a wire the source node owns. The node's own goroutine advances the bead — there is no separate per-wire pacer goroutine and no per-bead walker goroutine.
  • No rule enforces how many beads are on a wire. No occupancy / back-pressure gate. "One bead" is the per-emit granularity, not a per-wire capacity — several fires over time can leave multiple beads in flight on one wire simultaneously.
before: fire → 2s/400ms train of beads; per-wire pacer goroutine + N walker goroutines; receiver dedups train by seq after: fire → 1 bead; node goroutine advances it on a node-owned wire; no pacer, no walkers, no bead-count rule

This reverses the sustained-bead-train direction: the train model, seq-collapse, and persistent re-send are being removed.

What is unchanged

  • One clock, one pause button
  • pulseSpeed as the sole distance÷time conversion
  • Fire-and-forget sends on the Go→TS bridge
  • A bead's delivery time is still derived: placement + arc / pulseSpeed

The model being removed. Captured here so the reversal is legible.

Aspect Train model (removed)
Per-fire output Multi-bead train: trainDurationMs=2000, beadSpacingMs=400 — one fire emitted ~5 beads over 2 s
Pacer goroutine Per-wire pacer goroutine launched lazily on first fire; re-placed the held value every beadSpacingMs for trainDurationMs
Walker goroutines N per-bead walker goroutines; each parked on clk.WaitUntil and moved its bead inflight→delivered
Receiver dedup trainSeq / lastAcceptedSeq: receiver collapsed a whole train to one fire by accepting only the first bead with a new seq
Persistent flag Re-sent the held value with a fresh trainSeq on rollover, causing edges 2→5, 3→4, 4→5 to re-fire continuously
Back-pressure Occupied() gate: senders blocked until the wire was not occupied; persistent wires returned false to avoid stalling (fix d3a79e2b)

All of this is being removed.

The node loop after the change.

Node loop invariants

  • A fire produces exactly one bead appended to the owned wire's inflight FIFO (inflight []inflightBead reused unchanged).
  • The node goroutine advances beads: inflight[0] is always the earliest deadline. All beads on a wire share arc/pulseSpeed travel time, so placement order = deadline order — a FIFO, no scheduler or heap needed per wire.
  • At the head's deadline (placement + arc / pulseSpeed) the node moves it inflight→delivered; the receiver's Recv consumes from delivered.
  • A fan-out node owns several wires; it compares the few FIFO heads (min over out-degree, typically 1–3) — not a heap data structure.
  • Two-source wake: the one node goroutine must wake on EITHER an incoming input (which may fire and append a bead) OR the earliest head deadline; park until the earliest head, wake early when an input lands.
  • No bead-count cap; no Occupied gate.
Invariant Why it holds
One clock / one pause Advance still parks on clk.WaitUntil; only the goroutine location changes
Fire-and-forget sends Unchanged; no acks added
Delivery time derived = placement + arc / pulseSpeed Unchanged; the formula is the same whether the walker is a goroutine or a data record the node advances
Locality — a node's outbound advancement is wholly inside its own goroutine Strengthened: the node-owned wire model makes this structurally true, not just a convention
Inflight FIFO ordering = deadline ordering All beads on a wire share the same arc/pulseSpeed; append order is deadline order, so the FIFO head is always earliest

The destination node reads delivered while the source node writes inflight — same wire, so the wire lock (pw.mu) stays. The change removes goroutines, not the lock.

Indicative shape, pending Open Questions. Not a commitment.

Node-by-node rollout — start with node 1

The conversion is incremental. Begin with node 1 (the Input bootstrap node) and verify it in isolation before touching any other node.

  • Node 1 first. Convert node 1's emit path: one bead per fire, on a wire node 1 owns, advanced by node 1's own goroutine. No pacer goroutine. No walker goroutine.
  • Verify node 1 alone. Node 1 fires once and delivers exactly one bead on its owned wire. Confirm with go test -race and a headless trace showing one bead in, one bead delivered, no stall.
  • Proceed outward. Only after node 1 passes, repeat the same mechanical steps (below) for nodes 2, 3, 4, 5 in order — one at a time, verifying each before advancing.

Mechanical steps — applied first to node 1, then repeated per node

  1. Delete the train pacer (runTrain) and the per-bead walker goroutine launch; keep inflight/delivered slices.
  2. StartTrain becomes "append one bead" — no goroutine launch, no train state.
  3. Node run loop: select over {inputs that fire-and-append, earliest inflight head deadline}; on deadline move head inflight→delivered.
  4. Remove trainDurationMs, beadSpacingMs, persistent re-send, and Occupied/back-pressure gating.
  5. Decide seq's fate (Open Q1) before deleting trainSeq/lastAcceptedSeq.
  6. Verify: go build + go test -race; headless trace: the converted node emits one bead per fire and delivers.
# Question Notes
Q1 With one bead per fire, is per-train seq dedup (trainSeq / lastAcceptedSeq) still needed at all? Seq existed to collapse a train to one fire. One bead per fire may make it redundant — but it also dedups re-delivery/restart. Open.
Q2 What happens to the just-merged persistent per-edge flag? Persistent re-sent a held value as a fresh-seq train; with no train and one bead, "persistent" must be redefined or removed. Open.
Q3 Two-source wake: timer-as-channel (select input-chan vs deadline-chan) vs wake-on-arm (WaitUntil park cancelled when an input appends an earlier-due bead)? Pick the loop shape. Open.
Q4 Loop location: generic wrapper around every node's Update() vs inside each node kind's Update()? Generic needs the framework to know a node's owned wires. Open.
Q5 Confirm removing Occupied/back-pressure has no stall/flooding consequence now that nothing caps beads on a wire. The recent d3a79e2b stall fix becomes moot if Occupied is gone. Open.
