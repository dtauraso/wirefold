Sphere-Chain Node Layout

branch: task/sphere-chain-layout  ·  docs/planning/visual-editor/sphere-chain-layout-spec.html

Replace the discrete cubic lattice (nodes parked at integer cells) with a recursive sphere-chain. Each node is the center of a sphere; the nodes it sends to — its output targets — sit on that sphere's boundary (surface). Each of those targets is itself the center of its own sphere, with its targets on its surface. The structure unfolds recursively from node 1 (the Input / root).

child_pos = parent_center + R · dir
  • Node = sphere center. Every node is the center of a sphere of radius R.
  • Output-targets on the boundary. The nodes a node sends to are placed on its sphere's surface, at distance R in some unit direction dir.
  • Recursive from node 1. Node 1 centers a sphere; nodes 2 and 6 (its targets) sit on node 1's boundary and are themselves centers of spheres; repeat down every output edge.
  • Go computes + streams positions. Layout is geometry, so it is Go-owned: Go derives node world positions from the graph structure and streams them (as it streams centers today); TS only renders.
  • Replaces the lattice. No more lattice cell storage and no latticeToWorld; positions come from the spheres, not from stored (i,j,k) cells.

The current topology is not a clean tree: node 5 has two parents (6 and 4), and there is a feedback cycle (2 → 1). How those cases place is open — see Open Questions.

A child's world position is its parent's center pushed out by the sphere radius along a chosen unit direction:

child_pos = parent_center + R · dir |dir| = 1
  • R — sphere radius. A layout parameter. Whether it is a uniform tunable constant or derived per-node (e.g. scaled by the number of children, or by node size) is open (Q1). Leaning uniform constant to start.
  • dir — per-child direction. When a node has multiple output-targets, each target needs its own dir on the boundary. How those directions are chosen (even spread, general downstream direction, golden spiral, full 3D sphere vs a 2D circle) is the crux and is open (Q2).
  • Traversal. Layout is computed by walking the graph from the root (1) along output edges. Each node, once placed, places its own output-targets on its surface — the recursion in the Model tab.
  • Forward edges only (leaning). Back/feedback edges (e.g. 2 → 1) would make placement circular; excluding them from the traversal is leaning-open (Q4).

What is being removed: the discrete cubic lattice.

  • Each node is parked at an integer cell (i, j, k), stored per node.
  • World position is the cell scaled into space: worldPos = origin + cell · spacing (latticeToWorld).
  • Dragging a node snaps to the nearest cell — positions are quantized to the grid.
  • Positions are stored (the cell is authoritative); they do not depend on the wiring.

The lattice cell storage and latticeToWorld go away entirely; the sphere-chain derives positions from the graph instead.

What replaces it: a recursive spherical hierarchy rooted at node 1.

  • Each node is a sphere center; its output-targets ride its surface at radius R.
  • Positions are derived from the graph (parent → children on spheres), not stored as cells.
  • The "chain" follows output edges: node 1 → {2,6}, node 2 → {3,7}, node 6 → {5}, node 7 → {4}, node 4 → {5}, terminating at sinks (5) and dangling outputs (3).
  • No grid, no snapping, no quantized cells — the wiring is the layout.

The big open question: with positions derived from the topology, what happens to node dragging? Two framings:

A — Auto-computed (no dragging)

The layout is fully a function of the graph. Go computes every position from the wiring; the user never moves nodes by hand. Changing the layout means changing the wiring (or the layout parameters), not dragging.

B — Draggable on the sphere

Nodes remain movable, but constrained to the model: e.g. a child slides around on its parent's sphere surface (changing its dir at fixed R), or the tree can be re-rooted. Dragging adjusts dir/angle rather than free-placing.

Which framing wins is open (Q5). Either way, the lattice's drag-snap-to-cell behavior is gone — there are no cells to snap to.

# Question Notes / leaning
Q1 Sphere radius R: uniform constant (tunable), or per-node / derived (e.g. scaled by number of children or node size)? Open. Lean uniform constant to start.
Q2 Child placement on the boundary: when a node has multiple output-targets (e.g. 1 → {2,6}, 2 → {3,7}), how are they positioned on the sphere — evenly spaced (like the port ring), spread in a general downstream direction, golden-spiral, or…? And in 3D (full sphere) vs a 2D circle? Open. The crux.
Q3 Multi-parent nodes: node 5 is a target of both 6 and 4 — it can't sit on two spheres at once. Which parent places it (first / primary), a blend / midpoint, or does it get its own rule? Open.
Q4 Cycles / feedback: the 2 → 1 feedback edge would make placement circular. Are feedback / back edges excluded from the layout traversal (only forward output edges define the tree)? Open. Lean exclude back-edges.
Q5 Interaction: auto-computed layout (no dragging) vs draggable (slide a node on its parent's sphere / re-root). Does this remove node dragging entirely? Open. The other crux.
Q6 Root + disconnected / dangling: node 1 is the root. What about nodes with no incoming forward edge, or node 3's dangling output? How is the traversal seeded, and what places the root itself? Open.
Q7 Storage: the lattice stored (i,j,k) per node. Sphere-chain positions are derived from the graph — is anything stored per node (e.g. the chosen dir / angle on the parent sphere, so it's stable across loads), or fully recomputed each load? Open.
