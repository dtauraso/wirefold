Sphere-Chain Node Layout

branch: task/sphere-chain-layout  ·  docs/planning/visual-editor/sphere-chain-layout-spec.html

The sphere-chain is the placement frame — it replaces the lattice frame, the same way the lattice was the placement frame for discrete cells. Where the lattice provided a grid of cells to park nodes in, the sphere-chain provides a set of radius-R spheres whose surfaces nodes sit on. It is not a separate decoration; it is where positions come from.

Each node with outputs is the center of a sphere of radius R, and its directly connected neighbors ride that sphere's surface. Crucially this is symmetric and applies to every edge — in or out, back edges included:

A sphere is local: it concerns only that one node and the nodes it directly connects to (its neighbors) — never the whole graph. A sphere = a node + its direct neighbors on its surface. Only connected nodes share a surface: if two nodes have no edge between them, neither is on the other's sphere (e.g. nodes 6 and 7 are not connected, so neither sits on the other's surface).

neighbor_pos = node_center + R · dir A on B's sphere ⇔ B on A's sphere
  • Only output-bearing nodes center a sphere. A node with outputs centers a sphere of radius R. Output-less nodes are surface-only: they own no sphere; they sit on the surfaces of the node(s) that connect to them, and may belong to more than one surface (e.g. sink node 5 sits on both node 6's and node 4's surface; node 3's dangling output gives it no sphere). Multi-surface membership is fine.
  • Local sphere. Each sphere is about one node and its direct neighbors only, never the whole graph. Only connected nodes share a surface; unconnected pairs (e.g. 6 and 7) are on neither's sphere.
  • Symmetric neighbors-on-sphere. Every node connected to a given node by any edge — incoming, outgoing, or a feedback back-edge — sits on that node's surface, and it is reciprocal: if A is on B's sphere then B is on A's sphere.
  • Equivalent statement. Every edge has length R; equivalently, each node's neighbors all lie on its radius-R sphere.
  • Example. Node 2 sits on node 1's surface (the 1 → 2 edge), and nodes 1, 3, and 7 all sit on node 2's surface — node 1 via the 2 → 1 feedback back-edge, nodes 3 and 7 via 2 → 3 and 2 → 7. The 1↔2 relationship is reciprocal.
  • Go computes + streams positions. Layout is geometry, so it is Go-owned: Go derives node world positions from the graph structure and streams them (as it streams centers today); TS only renders.
  • Replaces the lattice frame. No more lattice cell storage and no latticeToWorld; positions come from the spheres, not from stored (i,j,k) cells.

The topology has a multi-surface node (5, on both 6's and 4's surfaces) and a feedback edge (2 → 1). Both are handled by the symmetric, local frame: back edges are in, multi-surface membership is fine, and the graph is sparse enough that a per-node placement exists — see Open Questions Q3.

A neighbor's world position is a node's center pushed out by the sphere radius along a unit direction. Symmetrically: every edge has length R, so both endpoints lie on each other's radius-R sphere.

neighbor_pos = node_center + R · dir |dir| = 1 · ‖edge‖ = R for every edge
  • R — per-node starting radius. R is a per-node starting value, independent and adjustable per node — not the length of any designated/reference edge (there is no designated edge). The user supplies the starting values. Resizing a node's sphere (dragging its surface toward the center) changes that node's R. Where the starting values come from is open (Q1).
  • Local frame. A sphere constrains only the node and its direct neighbors; only connected nodes lie on each other's surfaces. Output-less nodes have no sphere of their own and appear only as surface members of the node(s) they connect to (possibly several).
  • dir — per-neighbor direction. When a node has multiple neighbors, each one needs its own dir on the surface. How those directions are chosen (even spread, deterministic recursive BFS placement from node 1, golden spiral, full 3D sphere vs a 2D circle) is the crux and is open (Q2).
  • First-cut render (committed 837087f7). A see-through torus ring at radius R (styled like the node's border torus), toggled by a "sphere" button for the selected node, with discrete diameter-sized draggable surface slots. R there currently stands in as the live primary-output-edge distance until per-node starting values exist (Q1).
  • All edges define the frame. Back/feedback edges (e.g. 2 → 1) are not excluded — they constrain placement like any other edge (node 1 on node 2's sphere and vice versa). The frame is the whole edge set, not a spanning tree.
  • Exact placement, no relaxation needed. Achieving every-edge-= R exactly is only impossible for dense / over-constrained graphs — e.g. a 5-clique K5 with uniform edge length can't exist in 3D (at most 4 equidistant points = a tetrahedron), or a shared neighbor whose two neighbors are pinned more than 2R apart. This topology is sparse / under-constrained: 7 nodes give 3·7 − 6 = 15 degrees of freedom against only ~7 edges, so an exact uniform-R placement exists with slack to spare. A deterministic exact placement works — no force-directed relaxation required.

What is being removed: the discrete cubic lattice.

  • Each node is parked at an integer cell (i, j, k), stored per node.
  • World position is the cell scaled into space: worldPos = origin + cell · spacing (latticeToWorld).
  • Dragging a node snaps to the nearest cell — positions are quantized to the grid.
  • Positions are stored (the cell is authoritative); they do not depend on the wiring.

The lattice cell storage and latticeToWorld go away entirely; the sphere-chain derives positions from the graph instead.

What replaces it: the sphere-chain as the new placement frame — a set of radius-R spheres, one per node, whose surfaces hold that node's neighbors. It takes over the lattice's job (giving nodes somewhere to sit) rather than adding a layer.

  • Each output-bearing node is a sphere center; every node it shares an edge with — in, out, or back — rides its surface at radius R, reciprocally. Output-less nodes (3, 5) own no sphere and are surface-only, possibly on several surfaces.
  • Positions are derived from the graph (neighbors-on-sphere), not stored as cells.
  • The edge set defines the frame: node 1 ↔ {2,6}, node 2 ↔ {1,3,7}, node 6 ↔ {1,5}, node 7 ↔ {2,4}, node 4 ↔ {7,5}, node 5 ↔ {6,4}, including the 2 → 1 back-edge.
  • No grid, no snapping, no quantized cells — the wiring is the layout.

The big open question: with positions derived from the topology, what happens to node dragging? Two framings:

A — Auto-computed (no dragging)

The layout is fully a function of the graph. Because the topology is sparse, Go can compute an exact uniform-R placement deterministically (no relaxation); the user never moves nodes by hand. Changing the layout means changing the wiring (or the layout parameters), not dragging.

B — Draggable on the sphere

Nodes remain movable, but constrained to the model: e.g. a child slides around on its parent's sphere surface (changing its dir at fixed R), or the tree can be re-rooted. Dragging adjusts dir/angle rather than free-placing.

Which framing wins is open (Q5). Either way, the lattice's drag-snap-to-cell behavior is gone — there are no cells to snap to.

First-cut visualization (committed 837087f7)

A "sphere" button toggles a see-through torus ring at radius R for the selected node (styled like that node's border torus). It shows the node's own sphere. R there currently uses the live primary-output-edge distance as a stand-in for the node's starting R value (the real per-node starting value is still Q1). Neighbors land on the surface as discrete, diameter-sized draggable slots; dragging a slot inward shrinks that node's R.

Resizing a node's sphere changes that node's R. RESOLVED (framing) — R is a per-node starting value. Q2 Child placement on the boundary: when a node has multiple output-targets (e.g. 1 → {2,6}, 2 → {3,7}), how are they positioned on the sphere — evenly spaced (like the port ring), spread in a general downstream direction, golden-spiral, or…? And in 3D (full sphere) vs a 2D circle? Open. The crux.
Q3 Multi-parent / multi-surface nodes: node 5 is a neighbor of both 6 and 4, so it sits on both their surfaces at once. RESOLVED — multi-surface membership is fine. Node 5 is output-less, so it owns no sphere; it is placed purely as a surface member of its connecting nodes' spheres — here both 6 and 4. Sitting on two surfaces just means it is at distance R from both, satisfiable whenever 6 and 4 are within 2R. Because the graph is under-constrained, the layout keeps them within 2R, so exact placement works. No primary-parent / midpoint rule needed.
Q4 Cycles / feedback: how does the 2 → 1 feedback edge affect the frame? RESOLVED — back edges are not excluded; all edges define the frame. The 2 → 1 edge constrains placement exactly like any other (length R, node 1 on node 2's sphere and reciprocally). There is no forward-only traversal / spanning tree; the whole edge set is the frame.
Q5 Interaction: auto-computed layout (no dragging) vs draggable (slide a node on its parent's sphere / re-root). Does this remove node dragging entirely? Open. The other crux.
Q6 Output-less / dangling nodes: node 3's dangling output and the sink node 5. RESOLVED — output-less nodes have no sphere of their own; they are surface-only, placed solely as surface members of the node(s) that connect to them (possibly several — node 5 on both 6 and 4). This is a local, per-node frame, not a rooted traversal of the whole graph.
Q7 Storage: the lattice stored (i,j,k) per node. Sphere-chain positions are derived from the graph — is anything stored per node (e.g. the chosen dir / angle on the parent sphere, so it's stable across loads), or fully recomputed each load? Open.
