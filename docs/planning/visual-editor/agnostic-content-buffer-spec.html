Agnostic Content Buffer — one render-content path through Go & TS

Design spec · branch task/agnostic-content-buffer · full version: a language-neutral binary buffer, written by Go, viewed by TS, never re-materialized

The principle

There is one representation of the render content, and it moves through Go and TS seamlessly. The content is language-agnostic — plain numbers in a fixed layout — so neither side owns a translated copy. Go writes the buffer; TS views the same bytes and renders them. No struct, no intermediate object, no store in between.

Why "full" over a shared decoded shape: a shared shape still forces each side to decode text into its own objects every frame. A typed binary buffer is read directly as a TypedArray and handed to the GPU — zero parse, zero per-entity object, zero re-copy. Agnostic at the bytes, not just at the type.

One representation — content and logs

Everything Go emitsEncoding
Bead positions, node/edge geometry, interior slots, statusbinary columns
Semantic events (recv / fire / send / arrive / done) — the causal logbinary columns (transient fields in the snapshot)

Numeric content and the semantic/log events are the one agnostic binary representation. The human-readable .probe log and the tests read a decode of the same binary.

Semantic names live in the representation. Event kinds (fire, send…) and column names are declared once in the shared schema (an enum / a field table). The binary carries a small id into that schema; the name is resolved from the schema, identically on both sides. Names aren’t lost — they’re part of the layout, resolved from the shared schema, not re-spelled as text on every message.

The whole system is one loop: input → Go (owns all state) → one binary buffer → TS renders. TS renders and forwards input; it holds no domain state.

The loop (the “round trip”)

  1. User performs a gesture — orbit / zoom / drag / add / delete / click.
  2. TS forwards the raw pointer event (pointerdown/move/up, wheel, modifiers) plus the stateless raycast hit to Go immediately, fire-and-forget (no await, no response expected). TS does not interpret the gesture.
  3. Go’s gesture state machine decides what the raw input means (orbit / zoom / drag / create / delete) and updates its owned state (camera / topology / …).
  4. Go writes the updated state into the one binary buffer.
  5. The buffer streams to TS (framed stdout → ext host → transferable ArrayBuffer → webview).
  6. TS views the bytes and renders.
  7. … repeat.
Not a request/response. Step 2 (input out) and steps 4–6 (buffer in) are decoupled: TS never blocks waiting for Go to answer its input — it just keeps rendering the latest buffer. “Round trip” means the cycle input→Go→buffer→render→(user)→input, not a blocking call. The only thing that crosses TS→Go is input; the only thing that crosses Go→TS is the buffer.

Reliability — why it can’t scatter

All state lives in Go as explicit state machines (domain and gesture); TS is stateless by construction, so there is no TS state to spread across files. A check-no-webview-state guard fails the build if the webview grows a store or a stateful hook holding domain/gesture state — the rule is enforced, not merely intended.

Today: the content is materialized 3×

Every red arrow copies the same numbers into a new shape. The stores are the third copy — a TS-held mirror of Go state, lagging one hop, needing lifecycle resets. The whole chain is the impedance boundary; the buffer collapses it.

The path: write once, view once

Go writes the layout → framed binary bytes on stdout → the extension host transfers the ArrayBuffer (zero-copy) into the webview → viewed as TypedArrays → fed to instanced GPU attributes. The same bytes exist once and are handed along the path.

The transferable postMessage(buf,[buf]) hands buffer ownership to the webview with no copy. With no parse and no store, the render content is touched exactly at write (Go) and at GPU-upload (TS). The log events ride the same binary stream; a decoder renders them to text only for humans.

The layout (defined once)

Content is a set of fixed-width numeric columns. A row = one entity in that snapshot. The layout — column names, types, and offsets — is declared in one schema and generated for both sides. The row count is per-snapshot (in the snapshot header), not a fixed maximum.

// one schema (illustrative) — offsets/strides generated for Go writer + TS reader
beads   : { count, cols:[ x:f32, y:f32, z:f32, value:i32, frac:f32, live:u8 ] }
nodes   : { count, cols:[ cx:f32, cy:f32, cz:f32, r:f32, ...ports ] }
edges   : { count, cols:[ sx:f32, sy:f32, sz:f32, ex:f32, ey:f32, ez:f32 ] }
status  : { count, cols:[ torusRed:u8, missVal:i32, mx:f32, my:f32, mz:f32 ] }
nodes   : { ...persistent..., fired:u8, sent:u8, arrived:u8, recv:u8 ] // transient event flags

Fields are persistent or transient. Persistent fields carry state between ticks (position, geometry, held value). The semantic events are transient fields — a fired/sent/arrived/recv flag set only on the tick it occurs. They’re numeric like everything else; the name (“fired”) lives in the schema, and the human .probe log is decoded from these fields. The log is not a separate representation or a string — it’s columns in the snapshot, read through the schema. Since nothing is dropped, a flag set for a single tick is seen by every consumer.

What a snapshot contains

  • Every snapshot is the full current state — all live beads (each with position + value), all nodes (geometry + status + transient event flags), all edges.
  • Positions change tick to tick (beads moving); geometry is stable between edits but still present in every snapshot — full state, cheap at this scale (~10 nodes). Each snapshot stands alone: no accumulator, nothing “held between updates.”
  • A bead that arrived is simply absent from the next snapshot; a node that fired has its transient flag set that tick and clear the next. That is how mount/unmount and the “arrive clear” are expressed.

There are no per-entity objects on either side — a bead / node / edge is just a row of numbers in the snapshot’s columns.

Transport & framing

Go’s stdout carries framed binary output — one representation. Every frame is a snapshot; the extension host reads frames, transfers each snapshot to the webview (render), and decodes its event columns to the log — one frame serves both.

Clock & the buffer

Go enqueues one full snapshot per tick in which anything changed — a bead step, a node-window transition, a geometry/camera edit. The tick is the emit driver; there is no separate render beat. Ticks are human-paced (the clock’s scale), so buffer updates arrive at human-watchable speed; play/pause freezes the tick, which freezes buffer updates and thus the render. The clock model is pinned in MODEL.md (human-speed clock, WaitTick, tick-count windows) — this plan is where it meets the buffer.

Two buffers (SPSC) — decoupled timescales

The bridge is two double-ended queues (deques), one per direction, each single-producer / single-consumer. Go→TS: Go adds one end, TS reads the other — content snapshots (the log's event columns ride inside them). TS→Go: TS adds one end, Go reads the other — input events. Producer-one-end / consumer-other-end never contend, and the producer never blocks on the consumer — that is fire-and-forget.

Why two buffers: they decouple Go’s timescale from TS’s. Go produces at the human-speed tick rate; TS consumes at the browser frame rate (rAF). Neither has to stay in sync with the other — the buffer absorbs the mismatch. (Realized over the transport — pipe → ext host → postMessage, each hop already FIFO — not one literal shared ring across the process boundary.)

Rate mismatch is absorbed by queue length, not by dropping. Each buffer is a deque: the producer adds one end, the consumer reads the other. Producer faster → the queue lengthens (a backlog); consumer faster → it empties. Nothing is dropped or coalesced — the consumer reads every element, in order. Both directions use this same discipline. TS drains what’s queued each frame and renders, so it stays current; a transient TS hiccup just grows the queue briefly, then it drains as TS recovers. Because both sides run at ~human speed (Go’s tick rate ≈ TS’s frame rate), the backlog stays short in practice.

Empty is a normal state — both sides read non-blocking. Either deque can be empty at any read. TS finding nothing new re-draws the last frame (its GPU buffers still hold it); Go finding no input just continues its tick loop. Neither ever blocks waiting for the queue to fill — blocking would re-couple the two timescales and defeat the decoupling. An empty read simply means “nothing new this cycle,” and each side proceeds with what it already has.

Each deque element is a column-block snapshot (the full current state, in the fixed-column layout of the Layout tab) — not a delta. Go enqueues a snapshot; TS dequeues one and views its bytes as TypedArrays straight into the GPU. The deque is the transport; the column-block is what each element is made of.

Render consumption (no stores)

The webview holds the received ArrayBuffer and exposes typed views. The renderer reads columns straight into GPU attributes; nothing is copied into a store or a per-entity object.

const beadsXYZ = new Float32Array(buf, beads.off, beads.count*3);
// feed instanced attribute directly — one upload, no per-bead object
instanced.geometry.attributes.offset.array = beadsXYZ;
instanced.geometry.attributes.offset.needsUpdate = true;
instanced.count = liveBeadCount;

Deletions — every domain store

DELETE all TS-held domain state, not just the render buffers:

  • the 5 render stores — pulse-state, interior-bead-state, node-status-state, node-geometry, edge-geometry — and their useFrame-poll consumers;
  • the overlay-visibility flags (currently Go-reflected in camera-store) — they move into the buffer as bool/enum columns;
  • the spec/diagram store (store.ts) — Go owns the topology, persists it, and streams it back; the editor renders it and sends CRUD;
  • the camera pose — Go owns it too (a few columns); mouse gestures are input events to Go, the pose streams back like any other content.

TS keeps no state at all — not even gesture bookkeeping. Raw pointer events (pointerdown/move/up, wheel) are forwarded to Go; drag origin, which button is down, and the in-progress gesture all live in Go’s gesture state machine. (Browser pointer-capture is a DOM mechanism, not app state.)

Interaction

Raw pointer events (pointerdown/move/up, wheel) plus the stateless raycast hit are forwarded to Go immediately, fire-and-forget. Go’s gesture state machine decides what they mean (orbit / zoom / drag / create / delete). Everything the renderer needs (incl. port positions during a drag) is read from the buffer view. TS renders + forwards; it holds nothing.

Hit-testing (resolved — option a)

three.js builds its scene from the buffer; on a pointer event its raycaster computes which rendered entity is under the pointer (using the buffer’s camera). TS sends that hit — entity id + world point, in the shared format — directly to Go. Division of labor: three.js computes the hit geometry; Go’s gesture state machine decides what it means. TS stays stateless: build scene from buffer → raycast on demand → forward the hit. (Go’s data is a neutral numeric layout, not three.js’s format — TS maps columns into three.js structures.)

Codegen & guard (what makes it truly agnostic)

“Agnostic” is only real if both sides share one layout definition. Extend the existing generator (it already emits TS types + Go from single sources) to emit the buffer layout for both:

  • The semantic vocabulary is in the schema: event-kind enums (fire, send…) and column names are declared once here; both sides get identical ids + names. The binary carries the id; the name is part of the representation.
  • Go side: column offset constants + typed writer helpers (setBeadPos(row,x,y,z)) + event-kind enum.
  • TS side: matching offsets + TypedArray view factories over the received buffer + the same enum.
  • Guard: a check-buffer-layout-parity that fails if the generated Go and TS layouts (offsets and enum ids) diverge (same discipline as the trace-kind / edit-op parity guards).

Testability (storeless, still unit-tested)

  • Offset accessors are pure arithmetic — unit-test the row/column math directly.
  • Go writer: write known values, assert the bytes.
  • TS reader: build a buffer of known bytes, assert the views decode to the right numbers — and the snapshot frame reader tested like the existing stdout-reader / handleMessage tests.
  • Round-trip: Go writes → same layout → TS reads → equals the input (a cross-language golden test, like the trace-event fixture).

Scope, non-goals, open decisions

Changes

  • Go: pack all output — position/geometry/status and the semantic events (as transient columns) — into the snapshot layout; frame stdout output — one binary emitter.
  • Bridge/ext host: read framed stdout; transfer each snapshot to the webview; decode its event columns to the log.
  • Webview: hold the buffer, expose typed views, feed instanced GPU attributes; delete all domain stores (render + overlay flags + spec/diagram + camera pose); forward raw pointer events (+ raycast hit) to Go; read everything else from the buffer.
  • Codegen + guard: single layout schema → Go writer + TS reader + a layout-parity guard; docs (MODEL.md/CLAUDE.md bridge wording).

Non-goals

  • no change to the fire-and-forget TS→Go edit path, or to the meaning of the causal log — its semantics are unchanged; only its encoding moves to the shared binary.
  • no client-side domain state — Go owns topology, camera, overlays, everything; TS renders + forwards input only.
  • no new request/response or awaiting on the bridge — the buffer is one-way Go→TS, like the trace.

Rollout (design decisions all settled)

  • Risk & sequence: this is the largest change in the system — it re-shapes the Go output, the bridge wire format, the ext-host reader, the log/probe decode, and the whole webview render layer. Sequence it: (1) layout schema + codegen + guard, (2) Go binary writer + framing beside the current output, (3) ext-host read + transfer + event-decode, (4) webview views + GPU, (5) flip consumers to binary — delete all domain stores + forward raw input (Go’s FSM interprets), (6) remove the old emitter + update docs. Each step verifiable; the current path stays live as a fallback until step 5 and is deleted at step 6.
