Agnostic Content Buffer — one render-content path through Go & TS

Design spec · branch task/agnostic-content-buffer · full version: a language-neutral binary buffer, written by Go, viewed by TS, never re-materialized

The principle

There is one representation of the render content, and it moves through Go and TS seamlessly. The content is language-agnostic — plain numbers in a fixed layout — so neither side owns a translated copy. Go writes the buffer; TS views the same bytes and renders them. No struct, no JSON object, no store in between.

Why "full" over a shared JSON shape: a shared shape still forces each side to decode JSON into its own objects every frame. A typed binary buffer is read directly as a TypedArray and handed to the GPU — zero parse, zero per-entity object, zero re-copy. Agnostic at the bytes, not just at the type.

Two kinds of Go→TS content, split by their nature

ContentNatureRepresentation
Bead positions, node/edge geometry, interior slots, statusnumeric, high-frequency, "where things are"binary buffer (this spec)
recv / fire / send / arrive / donesemantic, low-frequency, "what happened" (the causal log)JSON events (unchanged — .probe, tests)

The numeric content becomes the agnostic buffer; the semantic events stay eventful for logging/tests. Numbers flow as numbers; meaning flows as events.

Today: the content is materialized 3×

Every red arrow copies the same numbers into a new shape. The stores are the third copy — a TS-held mirror of Go state, lagging one hop, needing lifecycle resets. The whole chain is the impedance boundary; the buffer collapses it.

The path: write once, view once

Go writes the layout → framed bytes on stdout → the extension host demuxes (binary frames vs JSON events) → the binary ArrayBuffer is transferred (zero-copy) into the webview → viewed as TypedArrays → fed to instanced GPU attributes. The same bytes exist once and are handed along the path.

The transferable postMessage(buf,[buf]) hands buffer ownership to the webview with no copy. Combined with no JSON.parse and no store, the render content is touched exactly at write (Go) and at GPU-upload (TS).

The layout (defined once)

Content is a set of fixed-width numeric columns. A row = one entity, indexed by a stable slot (a bead’s slot, a node’s index). The layout — column names, types, offsets, max rows — is declared in one schema and generated for both sides.

// one schema (illustrative) — offsets/strides generated for Go writer + TS reader
beads   : { maxRows, cols:[ x:f32, y:f32, z:f32, value:i32, frac:f32, live:u8 ] }
nodes   : { maxRows, cols:[ cx:f32, cy:f32, cz:f32, r:f32, ...ports ] }
edges   : { maxRows, cols:[ sx:f32, sy:f32, sz:f32, ex:f32, ey:f32, ez:f32 ] }
status  : { maxRows, cols:[ torusRed:u8, missVal:i32, mx:f32, my:f32, mz:f32 ] }

Update discipline (keeps it delta-cheap)

  • Positions (bead columns): written each frame while beads are in flight — the moving content.
  • Geometry (node/edge columns): written only on load or drag — a value retained in the buffer between updates (the buffer is the accumulator; no re-send).
  • A row’s live/present flag governs whether the renderer draws that slot — replaces mount/unmount and the “arrive” clear.

Slot indices are the “pool” idea reduced to its essence: a bead’s row number is its identity; there are no per-entity objects on either side.

Transport & framing

Go’s stdout carries framed output; the extension host demultiplexes. One stream, two lanes — still “Go reports what it does.”

// each frame: [kind:u8][len:u32-LE][payload]
kind 0 = JSON semantic event  // → existing trace path (log, tests) — unchanged
kind 1 = binary content frame // → transferable ArrayBuffer → webview render

Render consumption (no stores)

The webview holds the received ArrayBuffer and exposes typed views. The renderer reads columns straight into GPU attributes; nothing is copied into a store or a per-entity object.

const beadsXYZ = new Float32Array(buf, beads.off, beads.maxRows*3);
// feed instanced attribute directly — one upload, no per-bead object
instanced.geometry.attributes.offset.array = beadsXYZ;
instanced.geometry.attributes.offset.needsUpdate = true;
instanced.count = liveBeadCount;

Deletions

DELETE the 5 render stores — pulse-state, interior-bead-state, node-status-state, node-geometry, edge-geometry — and their useFrame-poll consumers, plus the JSON parse of the position/geometry/status trace kinds.

Interaction re-route

Interaction (port positions during a drag) reads the node columns of the buffer view instead of the node-geometry store — same numbers, one representation.

Kept

KEEP the spec/diagram store (editor state it mutates + sends as edits) and camera-store (user/persisted). These are TS-owned, not Go-pickup buffers.

Codegen & guard (what makes it truly agnostic)

“Agnostic” is only real if both sides share one layout definition. Extend the existing generator (it already emits TS types + Go from single sources) to emit the buffer layout for both:

  • Go side: column offset constants + typed writer helpers (setBeadPos(row,x,y,z)).
  • TS side: matching offsets + TypedArray view factories over the received buffer.
  • Guard: a check-buffer-layout-parity that fails if the generated Go and TS layouts diverge (same discipline as the trace-kind / edit-op parity guards).

Testability (storeless, still unit-tested)

  • Offset accessors are pure arithmetic — unit-test the row/column math directly.
  • Go writer: write known values, assert the bytes.
  • TS reader: build a buffer of known bytes, assert the views decode to the right numbers — and the frame-demux (JSON vs binary) tested like the existing splitJsonlLines/handleMessage tests.
  • Round-trip: Go writes → same layout → TS reads → equals the input (a cross-language golden test, like the trace-event fixture).

Scope, non-goals, open decisions

Changes

  • Go: pack position/geometry/status content into the buffer layout (in addition to keeping the semantic JSON events for the log); frame stdout output.
  • Bridge/ext host: demux framed stdout; transfer the binary ArrayBuffer to the webview.
  • Webview: hold the buffer, expose typed views, feed instanced GPU attributes; delete the 5 stores; re-route interaction to the buffer.
  • Codegen + guard: single layout schema → Go writer + TS reader + a layout-parity guard; docs (MODEL.md/CLAUDE.md bridge wording).

Non-goals

  • no change to the fire-and-forget TS→Go edit path, or the semantic causal trace/log.
  • not deleting the spec/camera (editor) stores.
  • no new request/response or awaiting on the bridge — the buffer is one-way Go→TS, like the trace.

Open decisions (resolve before build)

  • Frame cadence: emit a binary frame per render tick vs per Go clock event. Leaning per-tick for the moving columns; geometry columns only when they change.
  • maxRows policy: fixed ceiling vs grow-and-refit on overflow.
  • Whether the semantic events also move to the buffer eventually, or stay JSON permanently (this spec keeps them JSON).
  • Risk: this is the largest change in the system — it re-shapes the Go output, the bridge wire format, the ext-host reader, and the whole webview render layer. Sequence it: (1) layout schema + codegen + guard, (2) Go writer + framing behind the existing JSON path, (3) ext-host demux + transfer, (4) webview views + GPU, (5) delete stores + re-route interaction, (6) docs. Each step verifiable; the JSON render path can stay live until step 5 flips the consumer.
