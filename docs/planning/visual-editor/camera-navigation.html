3D Camera Navigation

branch: task/camera-nav-doc  ·  tools/topology-vscode/src/webview/three/ — editor camera controls

All navigation is handled by a custom controller in interaction-controls.ts — no OrbitControls. There are no left/right/middle-button distinctions; the controller is pointer-button-agnostic.

Action Input Notes
Orbit (arcball rotate) Drag on empty space Horizontal → world Y axis; vertical → world X axis. ROT_SPEED = 0.005 rad/px. Pivots around the persistent target (selected node, or seeded scene-depth point).
Pan Mouse wheel / two-finger scroll without Ctrl Slides the view: camera and target translate together along the camera right/up basis. Scale = distance to target (worldPerPixel = 2·dist·tan(fov/2)/viewportHeight). No z=0, no horizon-dependent scale.
Dolly (zoom) Wheel/pinch with Ctrl (or Cmd / trackpad pinch) Zooms toward the target: offset = camPos − target scaled by 1.01^deltaY, clamped to a min distance so it never crosses the target. deltaY > 0 zooms out, < 0 zooms in.
Fit ("Home") Home button in camera-ui.tsx Frames all nodes square-on and sets the target to the framed scene center. No keyboard shortcut.
Roll NOT IMPLEMENTED The arcball math can produce roll via composed quaternions, but no dedicated roll input exists. Documented-but-absent DOF.

The only modifier key is Ctrl / Cmd (on the wheel/pinch event) to switch pan → dolly. No other modifiers are used.

Orbit / pan / dolly are all defined from (camera pose, persistent target) — they never raycast the world z=0 plane. They are total operations: there is no grazing / edge-on singularity to reach. The separate unprojectToPlane (z=0 raycast) still exists, but only for placing things on the layout — node drag, port drag/anchor, click-pick — never for camera gestures.

Input map — gestures and their camera actions. All gestures are defined from the persistent target, not a z=0 raycast. Roll has no input binding today.

Gesture discrimination

pointerdown enters a pending state. If the pointer is released within CLICK_MAX_MS = 150 ms and moves less than MOVE_SLOP_PX = 6 px, it resolves as a click (select). Beyond those thresholds it resolves as a drag. What the drag does depends on what was hit:

  • Empty space — orbit.
  • Node body — move node.
  • Connected port — slide port to a new ring anchor (Go snaps).
  • Unconnected port — begin drawing a wire.

Camera3D state

Stored in viewer/types.ts:

Camera3D = {"{"} position: [x, y, z], quaternion: [x, y, z, w] {"}"}

Six degrees of freedom captured as position + unit quaternion. No Euler angles — quaternion composition avoids gimbal lock during arcball rotation.

In addition to the persisted pose, the camera carries a persistent target — a maintained look-at / pivot point that is finite at all times. Orbit, pan, and dolly are defined purely from (pose, target), which makes them total operations: no grazing / edge-on singularity is reachable. The target is set to the selected node when one is selected; otherwise it is seeded (when uninitialized) by projecting the scene center onto the camera's forward ray — a scene-depth point along the view direction.

The target is runtime-only for now — not yet persisted to scene.json; it is re-seeded on load.

Fit algorithm ("Home" button)

Implemented in camera-ui.tsx. Triggered by the Home button — no keyboard shortcut.

  1. Compute bounding box of all node centers ± node radius.
  2. Derive the required camera distance from the bbox dimensions and the camera's fov.
  3. Apply 1.2× padding factor so nodes aren't flush against the viewport edges.
  4. Reset to square-on: cam.up = [0, 1, 0], lookAt(center) — zero roll, zero tilt. Quaternion is the identity for the look direction toward center.
  5. Set the persistent target to the framed scene center, so subsequent orbit / pan / dolly pivot on the freshly-framed scene.

The target is runtime-only — it is not persisted to scene.json yet. On load it is re-seeded from the scene center along the camera's forward ray. Only position + quaternion round-trip through scene.json.

Save / restore flow

Camera save flow (top). Restore path on load described in caption.

Restore on load

  1. CameraRefBridge (in scene-content.tsx) checks viewerState.camera3d. If present, applies saved position + quaternion directly. Otherwise sets square-on default.
  2. CameraFitter fires auto-fit once per load epoch — only when there is no saved camera state. Prevents fit from overriding a restored camera.
  3. CameraSettleDetector fires ~250 ms after motion stops. Used to recompute occlusion only — does not gate interaction or trigger saves.

viewerState.camera3d flows through the store into the bridge. Debounce (400 ms) prevents rapid-fire saves during a pan or orbit gesture. The Go-side write is fire-and-forget (no await, no ack).

Total operations, not a partial z=0 raycast

The old design derived each gesture's anchor by raycasting the world z=0 plane under the cursor. That is a partial function: it is undefined when the view goes parallel to the ground — the ray never intersects, or the intersection diverges as the view grazes edge-on. The result was a reachable singularity (and the earlier "pan zooms / pan swings / direction reversed" symptoms).

The fix is representational: anchor every gesture on a camera-owned target that is finite at all times. Orbit, pan, and dolly become total by construction — the singularity is simply unrepresentable, because there is no plane intersection to diverge.

This was chosen over two alternatives:

  • (a) Clamp the tilt so the view can never go parallel to the ground — rejected: full 6-DOF is substance, not a convenience to trade away.
  • (b) A fallback band-aid when the raycast misses — rejected: it only masks the partial function instead of removing it.

Industry note. OrbitControls, Blender, Maya, and CAD packages all carry a target point — that is the robust part, and this project adopts the target. It does not adopt OrbitControls' rotation constraints (turntable azimuth/elevation, locked up-vector, roll-clamp), which it rejects: the arcball stays free 6-DOF.

Why no OrbitControls

Interaction control is classified as substance, not medium (per project design memory). Three.js OrbitControls is a medium default that encodes wrong constraints:

  • Fixed up-vector — OrbitControls prevents roll by locking the up-axis, trading full 6-DOF for convenience. The wirefold editor needs all 6 degrees available.
  • Constrained rotation — OrbitControls couples its target to a turntable / locked-up rotation model. The custom controller carries a persistent target too, but orbits it with a free arcball (world-fixed X/Y quaternion composition), keeping full 6-DOF.
  • Roll reachability — the arcball math (quaternion composition from world-axis rotations) can naturally produce roll as a side effect of combined rotations. Roll is reachable in the math today; it simply has no dedicated input binding yet.

The test: "can you recover full control from any pose on a device with only a single pointer?" OrbitControls fails this (roll is unrecoverable). The custom arcball passes.

Roll status

Roll is currently NOT IMPLEMENTED as a user input. A design comment exists acknowledging it as a future DOF. The arcball quaternion math does not prevent roll — composed horizontal+vertical rotations can produce incidental roll — but there is no deliberate roll gesture or keyboard shortcut today.
