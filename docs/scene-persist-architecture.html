wirefold · concurrency

scene_persist — the last locks nobody has looked at

Four locks guarding file I/O. Every other lock in the tree has been deleted or settled. These are different in kind: the observer is the filesystem, which has no perceptual threshold — so the "who could tell, and at what resolution" test that settled the others does not apply here.

goroutine lock / hazard timer-spawned file on disk

1 · the whole system — five persisters, four locks, two files

Every writer, the goroutine it runs on, the lock it takes, and the file it lands in. Five debounced persisters, each with its own independent mu.

2 · the race that is reachable today

Node X was dragged, so its position flush is pending on a timer. Within that ~250 ms window a different node Y is dragged, and X is Y's neighbour — so X's own mover goroutine synchronously rewrites X's meta.json while the timer goroutine is flushing the same file.

3 · atomic rename does not make these locks redundant

The obvious objection: writeJSONAtomic writes a temp file then renames, so readers never see a torn file. True — and irrelevant to what these locks prevent. Two distinct failures survive it, and both writers share one literal tmp filename.

4 · the question that killed the other four locks

Every lock removed so far fell to the same move: the state turned out to have, or could be given, exactly one owner. Asking it here gives three different answers.
