Mutex architecture — what is left, and who owns it

Every sync.Mutex and sync.Cond in non-test Go, the exact state each one guards, and the specific goroutines that touch it. Owners are named by the function that runs as the goroutine, because “which goroutine” is the whole question — a lock with one owner is not a lock, it is a leftover.

1 · PacedWire.mu — the one with three genuine contenders

One wire object is reached by three different goroutine roles. This is the clearest real contention in the system.

PacedWire.mu

nodes/Wiring/paced_wire.go:131 · locks at 188, 280, 294, 310, 318, 328, 365, 389

Guards
inflight []inflightBead, delivered []deliveredBead, nextGen, teardownGen (paced_wire.go:132–141)
Owners
  • writersource node's drive loop — placeBeadNoWalkerAt (:188) and StepOnceAt (:280–328), via Out.PlaceDrivenAt/Out.StepOnceAt. E.g. the DriveHeld goroutine, or a node's own Update loop.
  • writerdestination node's Update loop — PollRecvTick (:365), via In.PollRecv.
  • writeredgeMover.run — ReviseInFlightGeometry (:389), from recomputeGeometry. One such goroutine per edge, spawned at node_move.go:506.

CONTENTION VERIFIED. Three genuinely distinct, independently scheduled goroutines touch one PacedWire. Red-proven: removing the lock in ReviseInFlightGeometry yields WARNING: DATA RACE (paced_wire_concurrency_race_test.go).

RESTRUCTURING UNEXAMINED. inflight and delivered are guarded as one pair. Whether they separate by owner — placement and stepping on one side, delivery handoff on the other — has never been traced. The lock is needed as currently structured; that is a narrower claim than “needed”.

2 · outbox.mu + cond — the queue that prevents a deadlock

One outbox per node mover. The lock exists so a mover's handler never blocks on a neighbour, which is what breaks the mutual-adjacency cascade deadlock.

outbox.mu + outbox.cond

nodes/Wiring/node_mover.go:57–58 · methods on receiver ob at :70 (enqueue), :83 (run)

Guards
q []outboxItem, closed bool — the unbounded send queue
Owners
  • producernodeMover.run — the mover's own inbox-drain goroutine calls ob.enqueue from inside handle (node_move.go:574, goroutine at :494)
  • consumeroutbox.run — one dedicated sender goroutine per mover (spawned node_move.go:503)
  • shutdownctx-cancel watcher — anonymous closure at node_mover.go:85; sets closed and broadcasts only. Does not enqueue or dequeue.

CONTENTION VERIFIED. Bypassing the outbox reproduces the cascade deadlock as a timeout (outbox_mutual_adjacency_test.go — TestMutuallyAdjacentDragFloodNoDeadlock, TestOutboxFIFOPerTargetOrderNoDrop).

RESTRUCTURING UNEXAMINED. Per instance this is single-producer, single-consumer — verified above — which is precisely the shape with well-known lock-free forms. The unbounded requirement complicates that; it does not rule it out. Also unchecked: nothing bounds queue growth or drain time. Cascades are finite in practice, but only a bounded stress run completing within a timeout is asserted.

3 · Trace.mu — guards the ledger, not the handoff

The distinction here is the point: producers never touch this lock on the hot path. They send on a channel. The mutex only serializes the single drain goroutine's bookkeeping.

Trace.mu

Trace/Trace.go:294 · drain at Trace/drain.go:78 · spawned Trace/Trace.go:332

Guards
events, closed, sink, onEvent, debugSink (Trace.go:295–299)
Does NOT guard
event delivery. That is t.ch, a buffered channel — emit() does a bare select send with no lock (Trace.go:341–346)
Owners
  • writerTrace.drain — the single drain goroutine; appends events, writes sink in record()
  • writerClose — flips closed (drain.go:24–32)
  • readersEvents / WriteJSONL / SetDebugSink (drain.go:38–58)
  • no lockevery emitting goroutine — node Update loops, gate loops, nodeMover, edgeMover, stdin reader, main. They touch the channel, never the mutex.

STAYING. Trace is an accumulator: every goroutine's events must land in one place to become one buffer for the editor. Copies would give N partial streams and no whole picture — the move that removed RealClock.mu is not available here. Red-proven (trace_concurrency_test.go).

THE FRAMING ABOVE IT IS STILL OPEN. The merged log is already only arrival order at the drain — scheduler order, not causal order. Whether events need a global total order at all is the open row (task/framing-event-order).

4 · LayoutHolder.mu — uncontended; kept as insurance

LayoutHolder.mu

nodes/Wiring/layout_holder.go:118 · acquired at :141 (localPolarSteps), :161 (LoadLocalPolars)

Guards
localPolars, pole
Owners
  • singlethat node's own mover goroutine — and nothing else. A neighbour is reached by sending moveMsgKindNeighborSetC through the sender's outbox into the neighbour's inbox; the neighbour's own goroutine then calls SetLocalPolar/SetPole on its own holder.

CONTENTION REFUTED. The doc comment (layout_holder.go:77–115) records a retraction: an earlier version claimed cross-goroutine contention and was wrong three times. A test written to force it could not be made to fail with mu removed — 10/10 clean -race runs — so no vacuous test ships. This is the one lock here with no red-proof, deliberately.

Do not cite this lock as evidence that cross-goroutine holder access is expected. It is not. It is retained only against a future direct md.layoutHolders[m] call.

Gone — and why it is not coming back

RealClock.mu

Deleted. Replaced by ownership, not by a smaller lock: each goroutine holds one Clock.Copy(), so the only thing that reads or writes a clock is the goroutine that owns it. Speed changes travel on per-goroutine buffered-1 channels built at load time — the send ends are touched only by the stdin reader's goroutine, so nothing replaced the lock.

It was never removed for speed. Measured ~500–1700 acquisitions/sec against a mutex that does tens of millions, with no benchmark or profile in the repo ever motivating it. “Widest fan-in lock in the system” is a structural claim; it was being read as a cost claim. It went because a shared object forces every reader of the code to understand the whole network's clock wiring before understanding one goroutine — and because it defended an exactness (every reader agrees to the millisecond) that nothing here needs.

geomMu

Deleted by splitting nodeGeom into write-once identity plus mutable state, making the property structural rather than maintained. Zero non-comment references remain.
