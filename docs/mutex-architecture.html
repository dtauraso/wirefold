Mutex architecture — who touches what

Two lock groups left. Each diagram names the real goroutines, the real methods, and the real line numbers. A lock with one owner is not a lock.

gone · PacedWire.mu — deleted by making the wire a goroutine, not a smaller lock

Before, three independently scheduled goroutine roles (source drive loop, destination Update loop, per-edge mover) all locked one shared PacedWire struct to reach its inflight/delivered slices. After, PacedWire IS the per-edge goroutine (edgeMover.run, unchanged goroutine count — the edge mover already existed to revise in-flight geometry on a node move): a channel in from the source, a channel out to the destination, and the bead state touched by exactly one goroutine. Ownership replaces locking, the same move that deleted RealClock.mu below.

RESOLVED: the open question below this diagram used to be whether inflight/delivered split by owner. They no longer need to: there is only one owner, period — the wire's own goroutine. See docs/planning/visual-editor/wire-owns-itself.md and MODEL.md. task/mutex-pacedwire-restructure

gone · outbox.mu + cond — deleted by per-direction channels, not a smaller lock

Before, every mover's handler enqueued onto its own outbox (a mutex+cond queue) drained by a dedicated sender goroutine. After, each mover owns one plain channel PER ADJACENT NODE (neighborIn) plus one extIn for external entries; a send is a non-blocking attempt on the destination's own channel, and on a full channel the sender retains the message on its own pending slice and retries next cycle. No queue, no lock, no cond, no dedicated sender goroutine — two goroutines fewer per mover.

RESOLVED: the open question below this diagram used to be whether the SPSC outbox queue had a lock-free form. It no longer applies — there is no queue and no lock: each pair of adjacent movers talks over its own dedicated channel, and a full channel just means "try again next cycle," never a block.

1 · Trace.mu — the hot path is NOT locked

Every emitting goroutine sends on a channel, taking no lock. The mutex guards only the single drain goroutine's ledger. This is the most misread lock in the tree.

STAYING. An accumulator cannot be copied — copies give N partial streams and no whole picture, so the move that deleted RealClock.mu is unavailable here. But the framing above it is open: the merged log is already only arrival order at the drain — scheduler order, not causal order.

gone · LayoutHolder.mu — deleted by ownership, not by a smaller lock

A node never reaches into a neighbour's holder. It sends a message, and the neighbour's own goroutine does the write — so there was never anything for a lock to guard. Every production caller of the six accessors ran on the owning node's own goroutine already; the only concurrent second party the race detector ever found was a test polling live state from its own goroutine. The tests were fixed to wait on a happens-before edge production already emits (the "abc-drag" breadcrumb, or a tapped moveMsgKindNeighborSetC send); no synchronization was added to production. See docs/layout-holder-architecture.html and task/mutex-layout-holder.

2 · scene_persist — the second party is a timer goroutine

RESOLVED — task/mutex-debounced-persister deleted the debounce timer (and its lock) outright rather than keeping it: see docs/debounced-persister-architecture.html for the full analysis. Every domain persister below now writes synchronously on the calling goroutine; there is no timer goroutine left to race. This section is kept as the historical record of why the lock existed.

the standard library's timer-fire API used to run every debounced flush in a new goroutine. That was the easily-missed racer against a drag-time synchronous write on the same file.

gone · RealClock.mu — deleted by ownership, not by a smaller lock

Before, one object with a lock. After, a copy each and no lock. Nothing replaced it: the speed channels are built at load time and only the stdin reader's goroutine holds the send ends.
