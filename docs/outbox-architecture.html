outbox.mu — the last lock still called necessary

Three locks were described as needing to stay. geomMu, LayoutHolder.mu, and PacedWire.mu are all gone. This is the fourth, and the claim has been wrong every time so far.

goroutine locked / blocked queue free to proceed

1 · why it exists — the deadlock, drawn

Two mutually adjacent movers, both mid-handle, both with full inboxes. Each blocks writing into the other's inbox; neither returns to drain its own. Nothing breaks the cycle. This is the thing the outbox prevents.

2 · how it fixes it — the handler never blocks

The enqueue is an unbounded slice append. It cannot block, so handle always returns and always goes back to draining its own inbox. The blocking send is moved onto a goroutine whose blocking harms nobody.

3 · the shape — strictly one producer, one consumer

Per instance. Every mover has its own outbox, one enqueuer, one drainer. The third goroutine touches mu but never the queue — it only flips closed. SPSC is exactly the shape with well-known lock-free implementations.

4 · the obvious answer, and why it fails

"Just use a channel." A Go channel is bounded. Bounded means a full buffer blocks the sender — and a blocking sender in handle is exactly diagram 1. The unboundedness is not incidental; it is the whole mechanism.

5 · the track record on “this lock must stay”

Each of these was described in framings.md as needing its lock. Each was wrong. That is the prior this branch starts from.
