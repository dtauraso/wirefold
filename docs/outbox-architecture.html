outbox.mu — the record of a question that closed

geomMu was removed, LayoutHolder.mu was removed, and PacedWire.mu was removed — each was called necessary first. This page argued outbox.mu was the fourth and still standing. It is now gone too: task/mutex-outbox-restructure deleted the outbox struct entirely — no queue, no mu, no cond, no dedicated sender goroutine, no ctx-watcher goroutine. What's below is kept as the argument that led there, not as a description of the code that runs today (see diagram 6 for that).

goroutine locked / blocked queue free to proceed

1 · history — why the outbox was built in the first place

Two mutually adjacent movers, both mid-handle, both with full inboxes. Each blocks writing into the other's inbox; neither returns to drain its own. Nothing breaks the cycle. This is the deadlock the outbox was built to prevent — still true of the old synchronous send, still the reason ANY fix was needed.

2 · the shape that was replaced — outbox + dedicated sender

This was the first fix, and it shipped for a while. The enqueue was an unbounded slice append guarded by mu/cond; it could not block, so handle always returned. The blocking send was moved onto a SEPARATE goroutine per mover (plus a third ctx-watcher goroutine). This whole box — the queue, the mutex, the condvar, both extra goroutines — is what diagram 6 deletes.

3 · why the lock was even worth examining — it was SPSC

Per instance, every mover's outbox had exactly one enqueuer and one drainer; the third goroutine touched mu but never the queue — it only flipped closed. SPSC is the one shape a mutex can be replaced by ownership rather than by a cleverer lock. That is the property diagram 6's per-direction channels exploit directly: a Go channel IS an SPSC queue, so once the shape was SPSC the mutex was never load-bearing — it was guarding an invariant a channel already gives you for free.

4 · "just use a channel" — the argument that was wrong here

This diagram's ORIGINAL conclusion was: a Go channel is bounded, a full buffer blocks the sender, and a blocking sender in handle is exactly diagram 1 — so a channel alone can't replace the outbox. That conclusion was wrong, or at least incomplete: it only checked two shapes (block-on-full, drop-on-full) and missed the third, which is what actually shipped. Retain-and-retry — non-blocking send, and on failure the item stays with the SENDER to retry next cycle instead of blocking OR dropping — neither blocks handle nor loses a message. See diagram 6.

5 · the track record on “this lock must stay”

Each of these was described in framings.md as needing its lock. Each was wrong, including — as of this branch — the one this page itself argued for.

6 · what actually shipped — two directed channels, no lock

No shared inbox, no outbox, no mu, no cond, no dedicated sender goroutine, no ctx-watcher goroutine. Every pair of adjacent movers gets two plain Go channels, one per direction; each mover's own run loop (paced on its own Clock.Copy() + SleepCycle, same as every other pacing loop in the system) drains its inbound channels and retries its own outbound sends every cycle. Nothing is shared, so nothing needs a lock.
