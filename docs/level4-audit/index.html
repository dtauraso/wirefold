Level-4 Audit

wirefold · 2026-06-01

This audit examines the wirefold repo through a level-4 (Staff / model-correctness) lens. The optimization target is not lines of code or test coverage — it is AI re-derivation cost: every session where an AI agent must re-read stale docs, grep for constants, or discover silent failures from scratch burns tokens and produces drift.

Leverage Metric

Each finding is scored by:

leverage = (token cost per hit) × (sessions that hit it) × (drift likelihood)

Token cost per hit — how many tokens are wasted when an AI encounters this. Sessions that hit it — how often (stale CLAUDE.md loads every single session). Drift likelihood — probability the AI acts on bad info vs. recovers via grep.

Findings Ranked by Leverage

Finding 1 (stale docs) scores near-maximum because CLAUDE.md and MODEL.md load into context every single AI session, contain incorrect file paths, and actively instruct impossible actions.

Leverage Model

The level-4 lens asks: is the model correct? Not "does it compile?" (level 1) or "are modules well-bounded?" (level 3) — but "does the mental model encoded in the codebase match how the system actually works?"

For an AI-assisted project, a wrong model in context docs is worse than a wrong comment in code, because it is injected into every session's prompt window before any code is read.

Seniority Ladder of Refactoring

Why AI Re-Derivation Is the Cost Axis

Each session where an AI must re-derive a fact that was already known — because docs are wrong, constants aren't named, or invariants are runtime-only — costs real tokens and real money. Level-4 refactoring for AI-assisted projects means: encode the model correctly so the AI can read it once and stay correct, rather than re-grepping every session to recover ground truth.

Finding 1 — Stale Docs

HIGH

CLAUDE.md and MODEL.md describe the retired React Flow architecture as current, and both load into AI context every session, actively misdirecting all future work.

Evidence

(a) CLAUDE.md — Node primitive landing rule

Instructs AI to create:

tools/topology-vscode/src/webview/rf/nodes/<Kind>Node.tsx

and register in:

tools/topology-vscode/src/webview/schema/registry.ts

The webview/rf/ directory does not exist — React Flow was retired; the render path is now webview/three/. The schema dir is src/schema/, not webview/schema/.

(b) MODEL.md §slot-phase

Pins PUMP_SLOT_HANDLER, slots[port] writes, and GenericNode.tsx:142 reading slotEntry.phase. None of these exist. The pump.ts slot case is a no-op return.

Referenced line: pump.ts slot case — verified no-op in current code.

Proposed solution

  • Re-anchor CLAUDE.md and MODEL.md to the current three/ architecture in one pass:
    • In CLAUDE.md "Node primitive landing rule", replace the React-Flow path tools/topology-vscode/src/webview/rf/nodes/<Kind>Node.tsx and webview/schema/registry.ts with the real render path under tools/topology-vscode/src/webview/three/ and the real schema dir tools/topology-vscode/src/schema/.
    • In MODEL.md §slot-phase, delete the dead PUMP_SLOT_HANDLER, slots[port] writes, and GenericNode.tsx:142 description; replace with how pump.ts's slot case is a no-op return and where slot phase actually renders today.
  • Add a cheap guard so it can't silently rot again: a doc-lint step (greppable list of known-dead tokens like rf/nodes, GenericNode, PUMP_SLOT_HANDLER) wired into the Stop-hook / check scripts that fails when a dead token reappears in CLAUDE.md or MODEL.md.
  • Effort: small. Leverage: highest — these files load every session.

Failure Flow

Leverage: MAXIMUM. CLAUDE.md loads into every session before any code is read. The wrong paths mean the AI's first concrete action (file creation or grep) is incorrect. Recovery cost: ~2k–8k tokens of grep + backtrack per session. Drift likelihood: near 1.0 — no code guards against these wrong instructions.

Finding 2 — Silent-Failure Duplication

MED

Three verify-grep hotspots that fail silently when the codebase drifts — new node kinds produce wrong output, layout regressions are invisible, and trace exhaustiveness is unchecked.

Evidence

(a) node-override-text.ts — hand-repeated kind names

File hand-repeats node-kind strings ("Input", "ChainInhibitor", …) and field keys (raw["init"], state.held) in a switch statement. No codegen, not referenced in check-generated.sh. Adding or renaming a kind returns empty string with no error.

(b) Default node dims — duplicated across 7 files

The values 110 (width) and 60 (height) appear across at least 7 TypeScript files with no named constant. A layout change requires grep-and-replace with no compile-time guard.

(c) Trace-event kinds — re-typed as string literals

Trace kinds (recv, fire, send, slot, done) are generated into trace-kinds.ts (protected by check-generated.sh) but re-typed as raw string literals in pump.ts and messages.ts with no exhaustiveness check. A new kind added to the generator is silently missed.

Proposed solution

Three independent fixes:

  • node-override-text.ts: drive the kind names and field keys from the generated NODE_DEFS instead of a hand-written switch, OR add the file to check-generated.sh coverage with an exhaustiveness assertion keyed on NODE_DEFS, so a missing/renamed kind becomes a compile/check error instead of an empty string.
  • Default dims 110/60: extract a single exported NODE_DIM_FALLBACK = { width: 110, height: 60 } constant and import it in all 7 sites; delete the inline literals.
  • Trace kinds: add a TS exhaustiveness check in pump.ts — a switch over the generated TRACE_EVENT_KINDS union with a default: assertNever(kind) — so adding a Go trace kind that isn't handled fails tsc.
  • Effort: small–medium.

Leverage: MED. Hits when adding/renaming node kinds or changing layout. Each hit costs 1k–3k tokens of grep-to-diagnose. Drift likelihood is moderate — happens on feature work, not every session.

Finding 3 — SendRule Silent Degradation

MED

Invalid SendRule values in topology.json (e.g., a lowercase typo) are not rejected at parse time — they silently flip send semantics to consumeGated with zero signal.

Evidence

loader.go:288 — raw string cast into SendRule with no validation.

ports.go:171 — logic is Rule != RuleFireAndForget: anything unrecognized evaluates as consumeGated.

Example: "fireandforget" (missing camel case) in topology.json → silently treated as consumeGated. The node runs but blocks on slot fill instead of firing independently. No log, no error, no panic.

This is the only place in the Go model where a convention (the exact string "fireAndForget") is not enforced by structure.

Proposed solution

  • Add parse-time validation in validate.go: reject any node.data.sendRules value not in { "consumeGated", "fireAndForget" } with a clear error, instead of the silent cast at loader.go:288.
  • Make it structural: give SendRule a ParseSendRule(string) (SendRule, error) constructor that is the ONLY way to build one, so ports.go can never fall through to the consumeGated default on an unrecognized string.
  • Effort: small. This converts the one model-as-convention gap into model-as-structure.

Failure Path

Fix

In validate.go (or as a dedicated parse step), validate SendRule strings against the allowlist {consumeGated, fireAndForget} and return an error for unrecognized values. This moves a runtime convention to a parse-time invariant — the correct shape for the Go model.

Leverage: MED. Hits only when topology.json has SendRule values. But the failure is completely silent — a debugging session to diagnose wrong firing behavior can cost 5k–20k tokens. Drift likelihood: moderate (typos in JSON are common).
