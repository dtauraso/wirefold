Wirefold — Design Spec
Go Backend / TS Frontend
Spec is authoritative. Where the implementation and this spec conflict in interpretation, the spec wins — change the code to match the spec, not the spec to match the code. If the spec is genuinely wrong or ambiguous, stop and flag it for revision rather than silently diverging. Also flag any implementation code this spec does not mention — the spec's silence on a piece of code is itself something to surface for review, not a default-keep.

Go is the backend. TS is the frontend.

Go is the backend: holds all geometry, runs all animation calculations. TS is the frontend: renders what Go sends, owns the viewpoint (camera + scene navigation), and forwards geometry edits to Go.

Where each property lives
# ✓ Property Lives in Why
1 Concurrent Go (backend) Self-scheduling nodes, local rules over held state and wires. No global clock or round.
2 Geometric Go (backend) Holds all geometric data (nodes, ports, edges — endpoints and curves, wires, pulses); computes arc length, positions, durations.
3 Animating Go calculates, TS draws Go computes all animation values (progress, timing); TS only renders the result.
4 Clock (the system monotonic clock Go reads) Go (backend) One clock; wires scale its deltas by pulseSpeed to human speed; the gate stops the arithmetic, not the clock; no shared tick.
5 Viewpoint TS (frontend) How the diagram is viewed, not what it is — camera, orbit/pan/zoom, world→screen projection, raycast-picking, GPU render — in TS (three.js). Go owns the diagram; TS owns how it's viewed.

The contract in one line.

Go (backend) holds all geometry (nodes, ports, edges — endpoints and curves), shading, and animation data, runs the clock, and computes every bead position. TS (frontend) runs the GPU to display Go's data, owns the viewpoint (camera and scene navigation), and forwards editor geometry and animation updates to Go; Go updates its data accordingly. The bridge is: geometry + shading + position data out, editor geometry + animation updates in.
Human-viewing speed is Go's: Go runs the animation at a human pace; TS sends CRUD edits and the animation updates at that same pace. TS contributes data, never timing.
First concrete step

Go computes bead positions from clock + path geometry and emits them; TS is reduced to plotting the positions it receives and forwarding editor geometry and animation updates to Go, which updates its data accordingly.

What each goroutine does.

Concurrency is the nodes and wires: each is its own loop on its own goroutine, coordinating only through channels — no global clock, no central round. Active set: Input loop (in08), i0, i1 (ChainInhibitor), PacedWire (i0→i1).

The goroutines
# ✓ Goroutine Source file How many Its job
1 Input loop input/node.go one per data-source Input (e.g. in08)
  1. Read the signal on the wire i0->in08 (1 or 0).
  2. i = (i + signal) % len(Init) — 1 advances the read head (wrapping at the end), 0 holds it.
  3. Send Init[i] on the wire in08->i0.
  4. Sleep a pacing interval (the node runs slowly), then loop.
2 i0 loop chaininhibitor/node.go one (i0)
  1. Read a value v from in08 on the wire in08->i0, if a pulse arrived this round.
  2. If v differs from the held value: set held = v and send 1 on the wire i0->in08. Otherwise (v same as held, or no pulse this round): send 0 on i0->in08. (held inits to -1.)
  3. Send the held value on the wire i0->i1.
  4. Sleep a pacing interval (the node runs slowly), then loop.
3 i1 loop chaininhibitor/node.go one (i1)
  1. Receive i0's held value on the wire i0->i1.
  2. Its output wires are inactive here, so i1 is a sink — it runs the standard ChainInhibitor rule on i0's value and produces nothing downstream (ToNext unwired).
4 PacedWire loop Wiring/paced_wire.go goroutine — one per destination input port
  1. Add the beads it received this round to the set it already holds.
  2. Each round, for every bead it holds, compute the bead's next position from the Go clock and the human-speed slowdown (the pulseSpeed scaling).
  3. If that position is still on the wire, send the bead’s 3D data (its position) to TS.
  4. If that position is past the end of the wire, the bead is done — hand the value the bead is holding to the destination node and remove the bead from PacedWire (the off-the-end position is never sent to TS).
  5. Loop.
PacedWire is the only wire goroutine kind. It owns no clear/busy state and applies no send policy: a source places a bead whenever its own rule says to, and the wire may carry more than one bead at once.
In the target model each PacedWire wakes ~16 ms, advances its bead, emits the position, and self-delivers when elapsed reaches inFlightTime.
Geometry is Go's; TS captures edits and draws. Control points, arc length, world positions derived in Go at load and on each edit. TS sends edit messages; Go returns 3D data; TS plots only.

The goroutines, wired.

Every connection is a PacedWire goroutine: carries beads over a channel, holds no clear/busy state, may carry more than one bead at once. Nodes and wires are all goroutines — channels only.

Singletons, off to the side. main (spawns everything, then waits), the stdin reader (CRUD + pause/resume in), and the trace drain (orders and writes the trace) — process lifecycle and I/O, not part of the bead path.

What the TS frontend runs.

TS has no goroutines; its job is to render the stream Go emits and send CRUD + pause/resume back. Its execution contexts are below; the data work is Go's (see the TS → Go tab).

  1. Read each in-flight bead's current position from the stream Go emits (the pulse map, pulse-state.ts).
  2. Set the bead mesh position to that point.
  3. No curve rebuild, no progress t, no curve sampling — Go computes every position; no delivered post — Go times its own delivery.

  2. Debounce the camera matrix and fire a settle callback once motion stops (CameraSettleDetector).
  3. Track the nearest N nodes to the camera for occlusion/label decisions (NearestNTracker).
  4. Screen-space and GPU-tied — stays in TS.

  1. handleTraceEvent switches on the event kind (pump.ts).
  2. send → match edges by source+sourceHandle and setPulse on each.
  3. done → match edges by target+targetHandle and clearPulse on each.
  4. fire / recv → no-op.
4 Message receiver main.tsx event-driven (window "message")
  1. parseHostToWebview narrows the incoming message (main.tsx).
  2. trace-event → handleTraceEvent (the pump).
  3. load → store.load(text); run-status → set run state; flush → flush saves.
  4. On startup, post {type:"ready"} back to the host.
5 Outbound IPC handle-message.ts on user action
  1. vscode.postMessage from the webview; the extension's dispatch writes a JSON line to the Go process stdin (handle-message.ts).
  2. Out types: CRUD + control + raw input (deleteEdge, addEdge, fade, node-move, pause/resume), each carrying the picked Go id plus geometry/animation edits — never delivery, timing, or camera nav (orbit/pan/zoom stays in TS).
  3. Fire-and-forget — the writeStdin is not awaited; no request/response.
6 Input handlers interaction-controls.ts DOM pointer / wheel events
  1. pointer down: raycast-pick a port, node, or empty space; capture the drag/wiring origin (interaction-controls.ts).
  2. pointer move: drag a node (→ node-move IPC, rAF-throttled), preview a wire, or arcball-rotate the camera.
  3. pointer up: complete wiring (createEdge), persist a node move, or reset.
  4. wheel/pointer navigation: TS interprets it (ctrl = pinch-zoom to cursor, else pan on z=0) and commits the camera locally — TS owns the camera (three.js), nothing forwarded to Go.
7 React / Zustand store store.ts on state change
  1. Keeps a passive render copy of nodes/edges/selection (store.ts); Go owns the authoritative data.
  2. moveNode updates a node position locally and posts node-move; it derives no curve geometry and patches no pulse timing — Go does that.
  3. createEdge / deleteEdge / toggleFade apply optimistically and post addEdge / deleteEdge / fade to Go.

The TS frontend, wired.

Two pipes to Go — nothing else crosses. In: position stream → trace pump → render (draw beads). Out: CRUD, pause/resume, action input (Go id) via writeStdin. Camera stays in TS; TS computes no geometry.

Legend. Green = Go; blue = trace pump (Go→TS); purple = render path; amber = writeStdin (TS→Go). Two pipes; nothing else crosses. Dashed user arrow = the human sees the render, their actions go to Go.
TS only renders Go's stream, owns the camera + scene navigation, and forwards the user's actions. The authoritative data — positions, geometry, timing — lives in Go; the TS zustand store is a no-op holder Go writes.

Trace out, geometry in.

The whole boundary is two arrows. Nothing else crosses.

This matches uniform pulse speed — speed is a shared constant; the only per-wire variation is arc length, which is geometry.
TS sends editor geometry and animation updates to Go; Go updates its data accordingly. Go remains the single authoritative holder of all geometry and animation state.

The actual path. Conceptually two arrows; concretely the VS Code extension host relays them between Go's stdio and the webview.

Fire-and-forget both ways. Go→TS: position stream via stdout → postMessage. TS→Go: CRUD + pause/resume via postMessage → stdin. Extension host only relays; holds no state.

Wire format — JSONL. Both directions: one JSON object per line, split on \n — same framing as the probe logs. Pinned: the framing and the single id space, not field names.

Relay is a dumb pipe. Extension host forwards each line verbatim — no parse, no transform. stdout carries protocol only — logging goes to stderr or .probe/*.jsonl; a stray write corrupts the stream.

One id space — Go's

  • Every object Go streams to TS carries its Go id (node / edge / port).
  • A user interaction forms its outbound message directly from that id — e.g. {"t":"node-move","id":"i0","x":480,"y":63}. TS keeps no TS-specific id and does no lookup.
Illustrative shapes. Go → TS, a bead position on a wire: {"t":"pos","id":"i0ToI1","b":0,"x":1.2,"y":0.4,"z":0}. Go → TS, recomputed edge geometry after an edit: {"t":"edge","id":"i0ToI1","cp":[…],"arc":12.3}. TS → Go, a wiring edit: {"t":"add-edge","src":"in08","dst":"i0"}. Every id is Go's, both ways.
Escape hatch (not built). If postMessage can't handle the stream volume: localhost WebSocket from webview to a server in the Go process; extension only spawns Go and hands over the port. Unbuilt until profiling says otherwise.

Go calculates the animation; TS only draws it.

Go does all animation calculation — timing, progress, durations — and emits ready-to-render messages. TS draws them. The frontend never calculates animation; it renders frames Go describes. Today's inversion (TS pacing Go via NotifyDelivered) is what this design removes.

Before
After
Go computes and emits each position directly

Each wire's loop computes its bead's current position from the clock and path geometry (bezier math) and emits it; when elapsed reaches inFlightTime, the wire puts the bead on the destination channel.

  • Position computation is part of the wire's own loop — not a separate reader polling state at a fixed rate.
  • Pause freezes motion naturally: when the clock is paused, elapsed stops advancing, so the computed position is unchanged.
Sleep paces the loop

The ~16 ms sleep is the report rate (matching a 60Hz display), not bead speed — speed stays pulseSpeed, arrival stays at inFlightTime regardless of sleep duration.

  • 16 ms is the starting point — it matches a 60Hz display, which is the most a monitor can show. Emitting faster than the refresh is wasted.
  • The sleep sets report smoothness, not speed. The bead still arrives at exactly inFlightTime no matter the sleep — a longer sleep just shows fewer waypoints along the path.
  • Pause still freezes motion: while paused the clock stops advancing, so each wake computes the same position.
Human speed belongs to Go

pulseSpeed sets the human-watchable pace; Go emits positions at the ~16ms report rate; TS sends CRUD edits only and never drives timing.

Keeping them locked

Go is the single source of all animation values; TS renders exactly what Go sends, so there is nothing to keep in sync.

Play / Pause: the button is in TS, the clock is in Go.

Pause and resume act on Go's single clock. TS does not hold time — it holds a button. TS sends a pause or resume message to Go; Go's clock halts or runs. The frontend never owns any time.

Pause

  • TS sends "pause"
  • Go freezes its clock
  • In-flight beads stop progressing
  • No new firing (no wire completes)
  • Editor shows last known state

Resume

  • TS sends "resume"
  • Go's clock advances from where it left off
  • In-flight traversal continues
Freeze means stop accumulating

Pause means stop accumulating elapsed time — not diff wall-clock on resume. A bead 40% across stays 40% on resume. Implement as a monotonic clock with a paused accumulator (elapsed-while-paused is zero), behind an injectable interface — a real impl plus a fake the tests advance, so timing is deterministic.

Why the gate lives in Go

Play/pause halts Go's clock — not a TS render toggle. If TS paused its render loop while Go kept running, Go's clock would race ahead and be out of sync on resume. Button in TS; clock in Go.

How much of TS converts to Go?

The dividing line is data vs. the GPU.

  • Geometry, animation, and shading are data — Go owns and emits them.
  • Running the GPU and capturing input need the browser — these stay in TS.
  • Input that produces geometry (e.g. a node drag) is captured in TS and sent to Go as an edit, so Go stays authoritative.

Stays in TS (render + input + authoring)

  • GPU draw calls — WebGL/three.js render: sphere/torus/tube geometry, renderer.render(), the @react-three/fiber Canvas loop (scene-content.tsx: GraphNode, SingleEdgeTube, PulseBead).
  • GPU-resident resources — WebGL context, compiled shaders, baked textures/env maps; Go supplies the parameters, TS bakes + binds them (scene-content.tsx: ProceduralEnvProvider).
  • Human input — camera arcball/pan/pinch-dolly committed locally, never forwarded; raycast pick identifies node/port/edge; drag + wiring go to Go as geometry edits (useInteractionControls, RaycasterHelper).
  • DOM editor chrome — label overlays, occlusion badges, spec-override text, run button, camera-UI widgets; HTML stays in the browser (ThreeView.tsx).
  • Authoring lifecycle — editor state, JSON parse/serialize, VS Code save IPC: authoring, not Go-network compute (store.ts, save.ts, SaveLifecycle.tsx).
  • No authoritative store — TS holds no source-of-truth graph state; the zustand store is a no-op holder Go writes (store.ts).
  • Fire-and-forget sends — no await/Promise/reply; optimistic local apply for responsiveness, Go's echo reconciles.
Dividing line: data vs. GPU. Geometry, animation, shading are data — Go owns them. GPU + input capture require the browser — stay in TS. Geometry-producing input (node drag) is sent to Go as a geometry edit.
Moving geometry into Go doesn't mean serializing the derived geometry into topology.json — node x/y is the geometric input and still persists. File holds: network spec + view block (node x/y, camera, fade/label); port positions, curves, and arc lengths are recomputed at load. Node drag writes back one x/y.

New Go builds

Static geometry is already Go's — delete TS duplicates: BezierArcLength, PortCurveArcLength, nodeWorldPos, portWorldPos, CurveParamPulseSpeedWuPerMs. Go must newly build Chain 1 (clock + per-frame streaming; the bézier eval already exists).

Chain 1 — animation (first domino):
  • Clock + per-frame stepping — NEW. Monotonic clock; stdlib time.
  • Bead-position eval — EXPOSE, don't build. The 3D bézier eval and vec3 already exist inline in PortCurveArcLength; factor out and call per frame: bead t → 3D pos, streamed.
Diagram vs viewpoint.
  • Go: diagram (nodes/edges/beads, control points, arc length, world positions, animation).
  • TS: viewpoint (camera, orbit/pan/zoom, projection, picking, GPU). "TS computes none" = no diagram math.
  • CRUD surfaces send actions to Go with the Go id (one id space).
Fade reachability → Go. SetFaded + fade broadcast are already Go; the reachability that drives them is Go's too — diagram logic, not camera-dependent. Camera-dependent label-display (nearest-N-to-camera, occlusion) stays in TS.
Go libraries. stdlib time for the clock; the existing vec3 + bézier eval for beads — no 3D-math library. Camera, projection, and picking (Perspective, UnProject) stay in three.js.

Coding plan: clock first, then migrate data.

One model shift (Go owns the clock) plus staged data migration (positions, geometry, shading). Clock first — it unblocks the rest by letting Go run and deliver without TS. Each phase leaves the build green and the editor working.

The five phases at a glance

  • 1. Clock into Go — each wire goroutine elapses inFlightTime on Go's clock and delivers by putting the bead on the destination channel; wire loops sleep ~16ms. Deletes the NotifyDelivered-from-TS delivery trigger. (The unblock.)
  • 2. Position stream — Go computes each bead position and emits it; TS plots; delete TS position/timing math.
  • 3. Geometry data into Go — nodes, ports, edges (endpoints + curves) become Go-owned; TS sends geometry edits.
  • 4. Shading data into Go — materials, colors, glass and environment parameters become Go data; TS applies them on the GPU.
  • 5. Bridge → CRUD + docs — reshape messages into plain CRUD, delete notifyDelivered, finalize MODEL.md + CLAUDE.md.
Ordering principle — each phase ends green (go build/test, tsc, npm run build, all guard scripts). The clock phase is the keystone; the data phases are independent migrations after it.
The authoritative new-Go-builds list is in the TS → Go tab. Chain 1 (clock + per-frame bead eval) is the PacedWire rewrite — the first phase. Camera and picking are TS (three.js), not a Go phase.

The five phases in detail.

Phase 1 · Clock into Go (the unblock)

  • Scope: active net in08 / i0 / i1 and the i0→i1 edge; inactive nodes/edges stay in topology.inactive.json.
  • Goal: clock first — Go's clock drives delivery on i0→i1; no TS round-trip required.
  • Go: monotonic clock + paused accumulator (injectable: real + fake for tests); stamp start on Send; inFlightTime = arcLength / pulseSpeed. Wire sleeps ~16ms; deliver when elapsed ≥ inFlightTime — no TS message. Pause freezes clock; deleteEdge cancels via Reset() (paced_wire.go).
  • TS: none yet — keeps animating on its own (temporary cosmetic dual); delivery no longer depends on it.
  • Exit: headless go test runs the active net (in08 / i0 / i1) to completion with no editor attached (today it deadlocks after the first hop).
  • Docs: draft the MODEL.md Driver + Cross-boundary changes.

Phase 2 · Position stream

  • Go: each wake computes position = eval(controlPoints, elapsed / inFlightTime) and emits a position message (new trace kind); deliver at t ≥ 1.
  • TS: pump.ts sets the bead sprite to the received position. DELETE: PulseBead's per-frame getPointAt + t computation (scene-content.tsx), rfArcLength (geometry-helpers.ts), and moveNode's arc-length + simLatencyMs re-derive (store.ts). TS plots only.
  • TS: zustand store (store.ts) becomes a no-op holder; Go's position stream feeds the imperative path (pulse-state.ts, edge-curve map) directly.
  • Exit: beads animate from Go's stream; TS computes no positions; tsc + build green; golden position-sequence parity. Run trace-kind parity guard (TRACE_EVENT_KINDS ↔ pump switch).

Phase 3 · Geometry data into Go

  • Go: hold node positions, port positions, and edge data (endpoints + curve control points) as authoritative state; compute curves in Go.
  • TS: send geometry edits to Go (node-move, edge create/delete — extend existing moveNode/deleteEdge IPC to full geometry CRUD); render from Go's stream, computing none locally.
  • TS: zustand store is a no-op holder, Go sole writer; remove moveNode arc-length re-derive, curve management, and fade fixpoint from it, not relocate.
  • Exit: dragging a node updates Go's data, Go re-streams geometry + positions, TS renders without local geometry math.
  • Note: edge RECONNECT is a network redesign — scope separately; target-end = delete+create, source-end needs a new rewire IPC verb.
  • Delete edge with a pulse in flight: Go atomically cancels clock-delivery, drops the bead (Reset), emits pulse-cancelled so TS removes the bead. The existing deleteEdge handler (stdin_reader.go) must be rewritten — Reset alone is no longer sufficient.

Phase 4 · Shading data into Go

  • Go: hold material/color/fill/stroke (partly in NODE_DEFS already), glass/material parameters, and environment/lighting parameters as data.
  • TS: receive shading parameters, apply to GPU materials; bake environment map on the GPU from Go-supplied parameters.
  • Exit: appearance driven entirely by Go data; TS only executes GPU material setup.

Phase 5 · Bridge → CRUD + docs

  • Reshape bridge into plain CRUD: data stream out (geometry + shading + position); edits in (create/update/delete geometry + animation updates); pause/resume as a control signal.
  • Delete notifyDelivered and its Go stdin handler entirely.
  • Run message-kind parity (Go stdin_reader ↔ TS WEBVIEW_TO_HOST_TYPES), no-TS-timers, and slot-phase-boundary guards.
  • Finalize MODEL.md (Driver, Cross-boundary, Geometry/time, Editor-surface) and CLAUDE.md (pump description, bridge surface, drift rule) in this commit.
  • No await / no Promises on the TS→Go path — fire-and-forget; never block on Go; no request/response.
  • Fold deleteEdge into the generic CRUD edit path — one CRUD shape, not per-operation verbs; clock-cancel + pulse-cancelled echo from Phase 3 ride along.

Verification and risks.

How to test

Two surfaces: the live editor (every phase), and — new from Phase 1 — headless in-process go test driven by a fake clock (deterministic). Signal: the headless cascade completes (today deadlocks at hop 1); the live editor shows pulses from Go's stream.

Can it be tested in the editor?

  • Yes — at every phase. Load topology.json (click it → custom editor opens), Run, and watch pulses animate. The editor stays the integration test surface throughout.
  • From Phase 1: headless go test / go run . runs the cascade with no TS. Today it deadlocks after the first hop — "headless completes" is the proof Phase 1 worked.
  • Inspection tool both ways: the .probe breadcrumb logs (go.jsonl / ts.jsonl). Clear them, Run, read them to see Go deliveries firing on Go's clock and TS applying the stream.

  • Pause/resume — verify the paused accumulator preserves each bead's progress across resume (a bead 40% across stays 40%, no teleport).

Risks to watch

  • Bandwidth / busy stdout pipe: N beads × ~60/s over stdout — if the pipe fills, Go stalls and TS re-paces Go. Escape: emit-on-change deduplication (not a separate tick).
  • Two arc-length implementations must agree until Phase 2 deletes the TS copy — until then a drift risk.
  • Bead speed vs report rate: confirm the sleep never affects speed — speed stays pulseSpeed, arrival stays inFlightTime, the sleep only changes how many waypoints are shown.
  • Run an industry bug-class scan on the Phase 1–2 timing change before declaring it ready.
  • Zustand store (no-op holder): once store.ts is stripped of authority, grep its consumers — structural rendering must read from Go's data, not a local model.
  • Guard the TS→Go path against await/Promises: no await on a send-to-Go, no request/response. Consider extending the existing no-TS-timers guard to also reject awaiting the bridge. A blocking TS→Go call reintroduces event-loop coordination — the exact thing this avoids.
  • deleteEdge rewrite: confirm the new path cancels clock-delivery atomically and emits the pulse-cancelled echo; grep old deleteEdge call sites so nothing still relies on the Reset-only behavior. Removing the bespoke verb must not orphan any sender still using it.
This plan stays a proposal until Phase 1 lands; MODEL.md remains current for existing code until then.

This is a model amendment — update MODEL.md and CLAUDE.md when implementing.

This spec deletes NotifyDelivered as the delivery trigger, moves the clock and all position computation into Go, and makes each wire its own goroutine. Update MODEL.md and CLAUDE.md in the same commit as the code.

MODEL.md

  • Driver / Firing rule: delivery is no longer triggered by TS animation completion. Go's clock elapses inFlightTime and the PacedWire puts the bead on the destination channel.
  • Cross-boundary contract: NotifyDelivered is removed as the delivery trigger and is no longer "the only cross-boundary signal." The bridge becomes: position/geometry/shading data out; editor geometry + animation updates in (plus pause/resume).
  • Geometry and time: Go computes each bead position from its clock and path geometry; node and wire loops sleep (~16ms) to set the report rate; bead speed stays pulseSpeed (uniform), arrival stays at inFlightTime.
  • Editor surface realization: pump.ts no longer sends notifyDelivered; TS plots the position stream Go emits. The Go runtime no longer waits on a TS message to deliver.
