Go is the backend. TS is the frontend.

The system splits cleanly into a backend and a frontend. Go is the backend: it holds all geometric data and runs all the animation calculations — it owns what is happening and when. TS is the frontend: it renders the messages Go sends, and it sends geometric edits from the editor back to Go so the network updates.

Where each property lives
Property Lives in Why
Concurrent Go (backend) Self-scheduling nodes, local rules over slots and wires. No global clock or round.
Geometric Go (backend) Holds all geometric data; computes arc length, positions, durations.
Animating Go calculates, TS draws Go computes all animation values (progress, timing); TS only renders the result.

Backend and frontend.

Go — backend

  • Holds all geometry data — nodes, ports, edges (endpoints, curves), wires, pulses
  • Holds all shading data — materials, colors, glass and environment parameters
  • Computes all animation values — arc length, positions, traversal progress, durations
  • Computes every bead position and emits it to TS
  • Runs the substrate (nodes, slots, wires, send rules)
  • Sends messages describing what to render — geometry, shading, and position data

TS — frontend

  • Builds nothing on its own — takes geometry and shading data from Go and displays it
  • Runs the GPU: issues draw calls, manages the WebGL context, compiled shaders, and baked textures
  • Captures human input: camera/arcball/pan/pinch, pick/select, drag/wire
  • Sends editor geometry and animation updates to Go; Go updates its data accordingly — no geometry math, no animation calculation, no clock

What each goroutine does.

Go does not run one scheduler ticking everything in lockstep. The concurrency is the nodes and the wires: each node — and each wire — is its own small loop on its own goroutine, coordinating only through channels. There is no global clock goroutine and no central round. A handful of singleton goroutines handle I/O around the edges.

W (coincidence window): how long a windowed gate (ReadGate, InhibitRightGate) holds an early-arriving input while waiting for its partner. W = 1.5 × the gate's longest input-wire traversal time (traversal time = arcLength ÷ pulseSpeed, in milliseconds (ms)); recomputed from live wire geometry, elapsed on Go's sim-clock.
The goroutines
Goroutine How many Its job
main one
  1. Parse the CLI flags — topology path, trace-output path, run duration.
  2. Load topology.json via LoadTopology, which instantiates every node and builds the wires between them.
  3. Create the trace with NewWithSink — this also starts the trace-drain goroutine.
  4. Launch the stdin-reader goroutine; cancel the run context when stdin closes.
  5. Launch one goroutine per node (node.Update) under a WaitGroup, then block on Wait().
  6. After Wait() returns, tr.Close() closes the event channel so the trace-drain goroutine flushes its buffer and signals done; then, if -trace was set, tr.WriteJSONL writes those buffered events to the file as JSONL.
Input loop one per data-source Input (e.g. in08)
  1. No inputs — it is a source, driven by its init values.
  2. Take the next init value (wrapping if its repeat flag is set).
  3. Send it on the channel to the ToReadGate wire, but only once its ack edge shows that wire clear; never drop — if the prior bead isn't consumed yet, restart the loop and retry next round.
  4. Advance to the next init value; stop when init is exhausted unless repeat is set.
Bootstrap loop one (bootstrap_rg)
  1. A one-shot Input node (bootstrap_rg, init:[1], repeat:false) whose only job is to seed the ring.
  2. Send its single value 1 on the channel to the ToReadGate wire into readGate1.FromChainInhibitor — only once its ack edge shows that wire clear; never drop.
  3. That seed lets ReadGate fire its first round before the ChainInhibitor feedback edge has produced anything.
  4. repeat is false, so after the single send the loop exits and the node never fires again.
ReadGate loop one per ReadGate node
  1. Poll FromInput and FromChainInhibitor (non-blocking); on the first arrival, open a coincidence window W. On the first round FromChainInhibitor is supplied by the bootstrap Input node (bootstrap_rg, init:[1], repeat:false); on every round after, it comes from the ChainInhibitor feedback edge (i1ToReadGate1).
  2. Fire only when both arrive within the coincidence window W (measured on Go's sim-clock). If W elapses with only one input present, clear the partial arrival — empty its slot and ack its source over the ack edge, releasing a consumeGated sender — then restart. A timeout sends nothing downstream; the lone input decays.
  3. On fire, consume both inputs — clear their slots and ack each source over its ack edge so each source's send-check sees its wire clear.
  4. Send the value on the channel to the ToChainInhibitor wire, only once its ack edge shows that wire clear; never drop — if the prior bead isn't consumed yet, restart the loop and retry.
ChainInhibitor loop one per ChainInhibitor node
  1. Check every ToNext output wire; if any still carries a pulse, restart the loop and wait — backpressure means it does not consume its input yet.
  2. When all outputs are clear, take the pulse from FromPrevChainInhibitorNode and consume it — clear its slot and ack the source over its ack edge.
  3. Spawn one goroutine per ToNext port to fan out the held value concurrently; each sends the value on the channel to its wire, only once the ack edge shows that wire clear, never dropping.
  4. Store the value as Held and loop.
InhibitRightGate loop one per InhibitRightGate node
  1. Poll FromLeft and FromRight (non-blocking); on the first arrival, open a coincidence window W.
  2. Fire only when both arrive within W; if it times out, clear the partial arrival and restart.
  3. Compute result = 1 only if Left==1 and Right==0, otherwise 0.
  4. Consume both inputs — clear their slots and ack each source over its ack edge.
  5. Send result on the channel to the ToPassed wire, only once its ack edge shows that wire clear; never drop — otherwise restart the loop.
stdin reader one
  1. Read the next line TS wrote to the pipe and JSON-parse it.
  2. deleteEdge / addEdge → look up the wire and Delete() or Restore() it.
  3. fade → set the faded flag on the matching wires; node-move → apply the new x/y/z to the node.
  4. Loop, never stopping — intake stays live regardless of run state.
trace drain one
  1. Receive the next event from the shared channel that every node loop writes to.
  2. Assign it the next Step ordinal (the current event count) under a lock, giving all events a total order.
  3. Append it to the in-memory event buffer.
  4. If a sink is set, marshal the event to JSON and write it as one line.
  5. Loop until the channel closes at shutdown, then signal done so main can finish.
PacedWire goroutine — one per destination input port
  1. Poll its inbound channel for a bead a source node sent — every source converging on this port sends here, so fan-in is correct by construction.
  2. Mark the wire in-flight and elapse the traversal itself — inFlightTime = arcLength ÷ pulseSpeed on Go's sim-clock — advancing the bead and emitting its position each report (~16ms) for the renderer to draw. No TS NotifyDelivered: the wire times its own delivery.
  3. On traversal-complete, put the bead on the channel connecting it to the destination node.
ack edge goroutine — one per source node
  1. Poll its inbound channel for a consume-ack a destination node sent when it consumed its input (node-loop step 3).
  2. Mark the edge in-flight and elapse the traversal — inFlightTime = arcLength ÷ pulseSpeed on Go's sim-clock — advancing the bead backward (destination → source) and emitting its position each report (~16ms) for the renderer to draw.
  3. On traversal-complete, put the ack on the channel connecting it to the source node, so the source's send-check learns its outgoing wire is clear and it may send the next value.
Of the last two rows, PacedWire is now a goroutine — its own wire loop (see its row above). The ack edge remains data, not a goroutine: a backward-signalling wire carrying the consume-ack between decoupled nodes.
Where the clock change lands: in the target model each wire loop (PacedWire and the ack edge) wakes about every 16 ms to advance its bead, emit the bead's position, and self-deliver when elapsed reaches inFlightTime. Today that delivery instead waits on the stdin reader receiving a "delivered" message from TS — which is exactly the inversion Phase 1 removes.

What the TS frontend runs.

TS has no goroutines. Its work happens in a few execution contexts: the GPU render loop (driven by requestAnimationFrame inside the @react-three/fiber Canvas), an event-driven trace pump, the message receiver, the outbound IPC path, DOM input handlers, and the React/Zustand store. The render loop and input handlers must stay in TS — they need the WebGL context and the DOM event loop. Most of the data work migrates to Go; see the TS → Go tab.

  2. Rebuild the edge curve from current node positions (buildPortCurve) so a mid-drag move tracks.

  2. send → match edges by source+sourceHandle and setPulse on each.
  3. done → match edges by target+targetHandle and clearPulse on each.

  2. pointer move: drag a node (→ node-move IPC, rAF-throttled), preview a wire, or arcball-rotate the camera.
  3. pointer up: complete wiring (createEdge), persist a node move, or reset.
  4. wheel: ctrl = pinch-zoom toward cursor, else pan on the z=0 plane; commit camera.

  1. Holds the authoritative nodes/edges/selection today (store.ts).
  2. moveNode updates a node position and re-derives curve geometry + patches in-flight pulse timing.
  3. createEdge / deleteEdge / toggleFade update state and post addEdge / deleteEdge / fade to Go.

Stays in TS: the GPU render loop (draw calls, WebGL context), screen-space overlays, and DOM input capture — they need the browser. Moves to Go: bead positions, curve geometry, arc length, timing, and the authoritative store. The dividing line and the per-symbol migration list are in the TS → Go tab.

Trace out, geometry in.

The whole boundary is two arrows. Nothing else crosses.

This matches uniform pulse speed — speed is a shared constant; the only per-wire variation is arc length, which is geometry.
Go computes every bead's position from its clock and path geometry and emits it. TS sets each sprite to the position it receives — no interpolation, no math on the frontend.
TS sends editor geometry and animation updates to Go; Go updates its data accordingly. Go remains the single authoritative holder of all geometry and animation state.

Go calculates the animation; TS only draws it.

Go does all animation calculation — timing, progress, durations — and emits ready-to-render messages. TS draws them. The frontend never calculates animation; it renders frames Go describes. Today's inversion (TS pacing Go via NotifyDelivered) is what this design removes.

Before
After
Go computes and emits each position directly

Each node's loop computes the bead's current position from the clock and the bead's path geometry (plain bezier math, no library) and emits a position message for it. Computing the next position IS the loop's work — there is no separate tick reading state at a fixed rate. When the bead reaches the end of its path (elapsed reaches inFlightTime), Go delivers to the slot.

  • Position computation is part of the node's own loop — not a separate reader polling state at a fixed rate.
  • Pause freezes motion naturally: when the clock is paused, elapsed stops advancing, so the computed position is unchanged.
Sleep paces the loop

The node loop and the wire loop sleep between iterations. The sleep sets how often a position is emitted; the clock still sets where the bead is. Start with a 16 ms sleep (about 60 emits per second), matching a typical display refresh. Each wake reads the clock, computes position = eval(path, elapsed / inFlightTime), emits, then sleeps again.

  • 16 ms is the starting point — it matches a 60Hz display, which is the most a monitor can show. Emitting faster than the refresh is wasted.
  • The sleep sets report smoothness, not speed. The bead still arrives at exactly inFlightTime no matter the sleep — a longer sleep just shows fewer waypoints along the path.
  • Pause still freezes motion: while paused the clock stops advancing, so each wake computes the same position.
Human speed belongs to Go

Go runs the animation at a human-watchable pace — the bead moves at pulseSpeed and Go emits positions at the ~16ms report rate. TS sends CRUD edits only; the resulting update also plays at Go's human speed. TS never sets, drives, or accelerates timing — an edit's visible effect plays out on Go's clock, not instantly because TS sent it. Discrete cancellations (a delete) are immediate, but continuing motion stays human-paced: move a node mid-flight and the bead keeps traveling at human speed along the re-derived curve.

Keeping them locked

Go is the single source of all animation values. TS renders exactly what Go sends, so there is nothing to keep in sync — the frontend has no independent animation state.

Play / Pause: the button is in TS, the clock is in Go.

Pause and resume act on Go's single clock. TS does not hold time — it holds a button. TS sends a pause or resume message to Go; Go's clock halts or runs. The frontend never owns any time.

Pause

  • TS sends "pause"
  • Go freezes its clock
  • In-flight beads stop progressing
  • No new firing (no wire completes)
  • Editor shows last known state

Resume

  • TS sends "resume"
  • Go's clock advances from where it left off
  • In-flight traversal continues
Freeze means stop accumulating

"Freeze the clock" must mean stop accumulating elapsed time — not diff wall-clock on resume. Pause records how far each in-flight bead had elapsed; resume continues from there. A bead that was 40% across is still 40% across on resume. (If you just diffed time.Now() on resume, beads would jump forward by the pause duration and deliver instantly.) Implement as a monotonic clock with a paused accumulator so elapsed-while-paused is zero.

Why the gate lives in Go

Play/pause is substance, not a TS animation control — it halts the system's clock. If TS paused by stopping its own render loop while Go kept running, Go's clock would race ahead and the frontend would be out of sync on resume. So the button is in TS; the clock it controls is in Go. This fits the bridge: messages out (Go's timeline, including "paused"), control and geometry edits in.

The contract in one line.

Go (backend) holds all geometry (nodes, ports, edges — endpoints and curves), shading, and animation data, runs the clock, and computes every bead position. TS (frontend) runs the GPU to display Go's data and forwards editor geometry and animation updates to Go; Go updates its data accordingly. The bridge is: geometry + shading + position data out, editor geometry + animation updates in.
Human-viewing speed is Go's: Go runs the animation at a human pace; TS sends CRUD edits and the animation updates at that same pace. TS contributes data, never timing.
First concrete step

Go computes bead positions from clock + path geometry and emits them; TS is reduced to plotting the positions it receives and forwarding editor geometry and animation updates to Go, which updates its data accordingly.

How much of TS converts to Go?

The dividing line is now data vs. the GPU. All geometry, animation, and shading are data — Go owns and emits them. TS only runs the GPU (issues draw calls, manages GPU-resident resources) and captures human input. Everything TS currently computes that is data — bezier geometry, arc length, bead positions, traversal timing, material parameters, color, glass settings, environment parameters — moves to Go as data Go produces and streams out. What cannot leave TS is whatever physically requires the browser's WebGL context or the DOM event loop.

Geometry data → Go

  • Node sphere geometry — center position, radius, and the 3-D coordinates of every port position on the sphere surface; border ring geometry anchored to each node (geometry-helpers.ts: nodeWorldPos, portDir, portWorldPos)
  • Edge data — endpoints (source node/port → target node/port) and curve geometry (quadratic bezier control points: midpoint, perpendicular lift, bulge-factor offset); the full tube path TS uses to build SingleEdgeTube is held and computed by Go (geometry-helpers.ts: buildPortCurve, buildEdgeCurve)
  • Pulse / bead geometry — the 3-D path a bead rides, expressed as bezier control points Go already stores
  • Arc length — 50-segment numeric integration currently duplicated in TS; Go's own BezierArcLength is the single source (geometry-helpers.ts: rfArcLength)
  • Node positions — world coordinates authoritative in Go; TS forwards drags to Go as geometry edits and receives updated positions back (store.ts: moveNode)
  • Bounding box — min/max over all node positions and dimensions, used for camera fit; input positions are geometry Go holds (geometry-helpers.ts: boundingBox)

Animation data → Go

  • Bead position along the path — curve.getPointAt(t) currently evaluated every frame in TS; Go emits the 3-D position directly and TS plots it (scene-content.tsx: PulseBead useFrame)
  • Traversal progress fraction — t = (now − startTime) / simLatencyMs, clamped to [0, 1]; Go owns the clock and simLatencyMs so it computes and emits t (pulse-state.ts, scene-content.tsx)
  • Timing — simLatencyMs derived from arc length ÷ uniform pulse speed; delivery detection when elapsed reaches inFlightTime; bead progress preservation across geometry edits (store.ts: moveNode; pulse-state.ts: claimDelivered)
  • Pause-adjusted clock — getPauseAdjustedNow() is a TS re-implementation of a pauseable clock; Go's clock already embodies this (state/run-status.ts, pulse-state.ts)

Shading data → Go

  • Material / color / fill / stroke — node.data.fill and node.data.stroke keyed off NODE_DEFS; the definition of what color a node is is data Go supplies
  • Glass / material parameters — roughness, metalness, transmission, ior, thickness, clearcoat, reflectivity, and any per-node overrides; Go holds the parameter set, TS feeds it into MeshPhysicalMaterial
  • Environment / lighting parameters — sky color, sun direction, ambient intensity, and any other parameters that define the scene's lighting; Go supplies the values, TS bakes them into the PMREM environment map on the GPU
  • Wire / tube appearance — tube radius, opacity, color; Go provides these per-edge so TS can configure SingleEdgeTube without computing them
  • Pulse / bead appearance — bead size, color, emissive intensity; Go emits these alongside the bead position

Stays in TS (GPU + input only)

  • Issuing the GPU draw calls — the actual WebGL / Three.js render: constructing sphere, torus, and tube geometry objects; calling renderer.render(); managing the render loop via @react-three/fiber Canvas (scene-content.tsx: GraphNode, SingleEdgeTube, PulseBead)
  • GPU-resident resources that must live in the browser: the WebGL context itself; compiled shader programs; baked textures and environment maps. Go supplies the parameters (material values, environment parameters); TS bakes and binds them on the GPU (scene-content.tsx: ProceduralEnvProvider)
  • Human input capture: camera arcball / pan / pinch-dolly — DOM pointer and wheel events that must be consumed in TS (interaction-controls.ts: useInteractionControls); raycast pick — THREE.Raycaster hit-testing the live GPU scene to identify clicked node, port, or edge (scene-content.tsx: RaycasterHelper); drag / wire — node drag is captured in TS and sent to Go as a geometry edit so Go stays authoritative; port-to-port wiring gesture captured in TS and forwarded to Go
  • React / DOM editor chrome — label overlay divs, occlusion badges, in-node spec-override text, run button, camera-UI widgets; HTML rendering stays in the browser (ThreeView.tsx)
  • Zustand store, spec parse, save lifecycle — editor state management, JSON parse/serialize, VS Code IPC for save; these are editor authoring concerns, not substrate computation (store.ts, save.ts, SaveLifecycle.tsx)
  • No authoritative data store. TS holds no source-of-truth graph state — it applies Go's data to the scene. The per-frame loop reads imperative buffers fed by Go; the Zustand store is removed.
  • Sends to Go are fire-and-forget messages on the pipe — no await, no Promise, no waiting for a reply. Optimistic local apply covers responsiveness (e.g. a drag moves on screen immediately); Go's echo reconciles.
The dividing line is data vs. the GPU. Geometry (control points, arc length, wire path, node and port positions), animation (bead position along path, traversal progress, timing), and shading (materials, colors, glass and environment parameters) are all data — they belong in Go. Running the GPU and capturing human input stay in TS because they require the browser's WebGL context and DOM event loop. Input that produces geometry — a node drag — is captured in TS and sent to Go as a geometry edit, so Go remains the single source of authoritative data.
Moving geometry into Go does not mean serializing it into topology.json. The file holds only two things, the same as today: the substrate spec (nodes, ports, edges, send rules, slots) and the authored layout (the view block — node x/y positions, camera pose, fade/label UI state). Everything Go now owns is derived from those inputs — port world-positions from node position + side, curve control points from endpoints + route, arc length from the curve, bead positions streamed at runtime. Go recomputes derived geometry at load (it already computes arc length there); persisting it would create a second source of truth that drifts. The file does not grow — a node drag writes back one changed x/y.

This is a model amendment — update MODEL.md and CLAUDE.md when implementing.

This spec changes the substrate model. It deletes NotifyDelivered as the delivery trigger and moves the clock and all position computation into Go. The pinned docs must change in the same commit as the code, or the model and the implementation will disagree.

MODEL.md

  • Driver / Firing rule: delivery is no longer triggered by TS animation completion. Go's clock elapses inFlightTime and Go delivers to the slot itself.
  • Cross-boundary contract: NotifyDelivered is removed as the delivery trigger and is no longer "the only cross-boundary signal." The bridge becomes: position/geometry/shading data out; editor geometry + animation updates in (plus pause/resume).
  • Geometry and time: Go computes each bead position from its clock and path geometry; node and wire loops sleep (~16ms) to set the report rate; bead speed stays pulseSpeed (uniform), arrival stays at inFlightTime.
  • Editor surface realization: pump.ts no longer sends notifyDelivered; TS plots the position stream Go emits. The Go runtime no longer waits on a TS message to deliver.

CLAUDE.md

  • Substrate model pointer / pump description: pump.ts is now a position-stream plotter, not the driver of delivery timing.
  • Bridge surface note: the Go to TS channel carries a per-bead position stream; TS to Go carries geometry edits and pause/resume.
  • Drift rule check: timing/pacing logic stays in Go; TS must not compute positions or interpolate.
Until this lands, the spec and MODEL.md disagree on who triggers delivery. Treat MODEL.md as current for existing code; treat this spec as the target.

Coding plan: clock first, then migrate data.

The change is one model shift (Go owns the clock) plus a staged migration of data (positions, geometry, shading) from TS to Go. Each phase must leave the build green and the editor working. Order matters: do the clock first because it is the only step that unblocks the rest — it lets Go run and deliver without TS, and removes the timing dependency. Everything after it is incremental data migration that can land one piece at a time.

The five phases at a glance

  • 1. Clock into Go — Go elapses inFlightTime and self-delivers; loops sleep ~16ms. Deletes the NotifyDelivered-from-TS delivery trigger. (The unblock.)
  • 2. Position stream — Go computes each bead position and emits it; TS plots; delete TS position/timing math.
  • 3. Geometry data into Go — nodes, ports, edges (endpoints + curves) become Go-owned; TS sends geometry edits.
  • 4. Shading data into Go — materials, colors, glass and environment parameters become Go data; TS applies them on the GPU.
  • 5. Bridge → CRUD + docs — reshape messages into plain CRUD, delete notifyDelivered, finalize MODEL.md + CLAUDE.md.
Ordering principle — each phase ends green (go build/test, tsc, npm run build, all guard scripts). The clock phase is the keystone; the data phases are independent migrations after it.

The five phases in detail.

Phase 1 · Clock into Go (the unblock)

  • Scope (minimal slice):
    • Run only in08 (Input) → readGate1 (ReadGate) via edge in08ToReadGate1 (target port FromInput). All other nodes/edges in topology.json are parked, not deleted.
    • JSON has no comment syntax, so "commenting out" = move the other nodes and edges out of the live nodes/edges arrays into an ignored key (e.g. _parked: { nodes: [...], edges: [...] }). The loader only walks nodes/edges, so _parked is inert; restoring later is moving entries back. This preserves every other item exactly (no deletion) while leaving only the slice live.
    • Note: readGate1 has other input edges (i1ToReadGate1, bootstrapRgToReadGate1) that get parked too, so readGate1 may not FIRE in this slice. That is fine — Phase 1's goal is delivery INTO readGate1's FromInput slot from in08, not readGate firing. ReadGate firing returns once its other inputs are handled in a later step.
  • Go: add a monotonic clock with a paused-time accumulator; on Send, stamp start time and compute inFlightTime = arcLength / pulseSpeed (Go already has BezierArcLength). The node/wire loop sleeps ~16ms; each wake checks elapsed ≥ inFlightTime and triggers delivery itself (calls the existing NotifyDelivered slot-fill path internally instead of waiting on a TS message). (paced_wire.go, the source node Update loop.)
  • Go runs a continuous inbound reader on the pipe TS writes to (RunStdinReader on stdin). It is ALWAYS reading, independent of pause/play. Pause freezes the clock (bead motion + delivery); it does NOT freeze message intake — Go keeps receiving and acting on messages while paused.
  • Now that Go self-delivers, a deleteEdge races Go's own clock-driven delivery (not a TS message). The cancel must be atomic: delete and clock-delivery serialize on the wire's lock, and a reset/deleted wire must never deliver. Erase the in-flight pulse via Reset() before delivery can fire.
  • TS: none yet — TS keeps animating on its own this phase (a temporary cosmetic dual); delivery no longer depends on it.
  • Exit: headless go run . completes the cascade. Today it deadlocks after the first hop because delivery waits on the visual layer; after this phase it should run to completion with no editor. Add a Go test that runs a cascade with no TS attached.
  • Docs: draft the MODEL.md Driver + Cross-boundary changes.

Phase 2 · Position stream

  • Go: each loop wake computes position = eval(controlPoints, elapsed / inFlightTime) and emits a position message (a new trace kind); deliver at t ≥ 1.
  • TS: pump.ts handles the new position message and sets the bead sprite to the received position. DELETE the TS position/timing math: PulseBead's per-frame getPointAt + t computation (scene-content.tsx), the duplicated rfArcLength (geometry-helpers.ts), and store.ts moveNode's local re-derive of arc length + simLatencyMs. TS plots only.
  • TS: Remove the authoritative Zustand store (store.ts): Go owns the data, so TS keeps no authoritative graph state. The per-frame path already bypasses React/Zustand by reading imperative stores (pulse-state.ts, edge-curve map); Go's position stream feeds that imperative path directly.
  • Exit: beads move from Go's stream; TS computes no positions; tsc + build green; visual parity with before. Run the trace-kind parity guard (TRACE_EVENT_KINDS ↔ pump switch).

Phase 3 · Geometry data into Go

  • Go: hold node positions, port positions, and edge data (endpoints source→target, curve control points) as authoritative state; compute curves in Go.
  • TS: send editor geometry edits to Go (node move, edge create/delete — partly exists via moveNode/deleteEdge IPC; extend to full geometry CRUD); render from the geometry Go streams back, computing none of it locally.
  • TS: Any remaining structural state (which nodes/edges exist) is a passive render copy of what Go sent — not an authoritative store, holds no geometry logic. Zustand's data-owning/computing role (moveNode arc-length re-derive, curve management, fade fixpoint) is deleted, not relocated to TS.
  • Exit: dragging a node updates Go's data, Go re-streams geometry + positions, TS renders without local geometry math.
  • Note: per the handoff caveat, edge RECONNECT (moving a wire's endpoint) is a substrate redesign, not a small edit — scope it separately; target-end move is delete+create, source-end needs a new rewire IPC verb.
  • Delete edge with a pulse in flight:
    • TS can send deleteEdge while a pulse is still inside that edge, during pause or play. The message is fire-and-forget; TS does not wait for Go.
    • Go receives it and erases the edge AND drops the in-flight pulse (Reset) before the pulse reaches the destination slot — the pulse must never land on a deleted edge.
    • TS removes the bead from the scene on Go's echo (edge-gone / pulse-cancelled message), not on its own — Go is authoritative about whether the pulse delivered or was dropped.
    • The existing deleteEdge handler (stdin_reader.go → Reset only) must be REWRITTEN: it was built for TS-triggered delivery and only drops the bead + frees a parked sender. It now must atomically cancel Go's clock-driven self-delivery (deleted wire never delivers) AND emit a pulse-cancelled echo so TS removes the bead. Reset alone is no longer sufficient.

Phase 4 · Shading data into Go

  • Go: hold material/color/fill/stroke (already partly in NODE_DEFS data), glass/material parameters, and environment/lighting parameters as data.
  • TS: receive shading parameters and apply them to GPU materials; bake the environment map on the GPU from Go-supplied parameters (the bake itself is a GPU op and stays in TS).
  • Exit: appearance is driven entirely by Go data; TS only executes GPU material setup.

Phase 5 · Bridge → CRUD + docs

  • Reshape the bridge into plain CRUD: data stream out (geometry + shading + position); edits in (create/update/delete geometry + animation updates); pause/resume as a control signal.
  • Delete the notifyDelivered message and its Go stdin handler entirely.
  • Run message-kind parity (Go stdin_reader ↔ TS WEBVIEW_TO_HOST_TYPES), no-TS-timers, and slot-phase-boundary guards.
  • Finalize MODEL.md (Driver, Cross-boundary, Geometry/time, Editor-surface) and CLAUDE.md (pump description, bridge surface, drift rule) in this commit.
  • No await / no Promises on the TS→Go path. TS communication to Go is fire-and-forget: write a message to the pipe Go reads (stdin) and return immediately. TS never awaits a Go response and never blocks on Go. Inbound data from Go is handled as separate independent messages (write to buffer, return). No request/response, no promise chains gating other work.
  • Fold deleteEdge into the generic CRUD edit path — remove it as a bespoke verb. Edge deletion becomes a delete edit like any other geometry change; the clock-cancel + pulse-cancelled echo from Phase 3 ride along. One CRUD shape, not per-operation verbs.

Verification and risks.

How to test

There are two test surfaces — the live editor (works at every phase) and, new from Phase 1 on, a headless Go test that runs the network with no editor attached. The single clearest signal the model works: the cascade runs to completion headlessly (today it deadlocks after the first hop), and the live editor shows the same pulse animation now driven by Go's stream.

Can it be tested in the editor?

  • Yes — at every phase. Load topology.json (click it → custom editor opens), Run, and watch pulses animate. The editor stays the integration test surface throughout.
  • New capability from Phase 1: a headless Go test. Because Go owns the clock and self-delivers, the cascade can run inside go test / go run . with no TS. Today that deadlocks after the first hop because delivery waits on the visual layer — so "headless run completes" is itself the proof Phase 1 worked.
  • Inspection tool both ways: the .probe breadcrumb logs (go.jsonl / ts.jsonl). Clear them, Run, read them to see Go deliveries firing on Go's clock and TS applying the stream.

What shows it worked — per phase

  • Phase 1 (clock): go run . / a Go test completes the full cascade with no editor (previously impossible). go.jsonl shows deliveries firing on Go's clock timing (~inFlightTime apart), not on a TS message. Stronger check: freeze or detach the renderer — Go keeps delivering on schedule.
  • Phase 2 (position stream): beads animate from Go's emitted positions with visual parity to before. probe shows position messages arriving; TS computes no positions (the deleted PulseBead/rfArcLength/moveNode math is gone). A Go test can assert positions are emitted ~16ms apart with monotonically increasing t and arrival at t=1.
  • Phase 3 (geometry): drag a node in the editor — the wire re-curves and the in-flight bead re-derives remaining travel, all from Go's data. Proof TS no longer computes geometry: dragging stays correct after the local TS geometry math is removed. probe shows the geometry edit round-tripping through Go.
  • Phase 4 (shading): appearance is unchanged at parity; changing a shading parameter in Go's data is reflected on screen. Proof: materials are driven by Go-supplied parameters, not TS constants.
  • Phase 5 (bridge/CRUD): the notifyDelivered message is gone — grep confirms nothing sends it; all parity/guard scripts pass; save/load through VS Code still works end to end.

Reference / regression

  • Capture a visual reference run before Phase 2 (a short recording or screenshots of a cascade) and compare after each phase for parity — motion should look identical; only the source of the motion changed.
  • Add the headless cascade Go test as a permanent regression: it locks in that the substrate runs without the editor and never silently re-couples to it.
The litmus test for the whole effort: pull the editor away and Go still runs the network; reattach the editor and it draws exactly what Go is doing. If both hold, the clock and data truly moved to Go.

Risks to watch

  • Bandwidth/backpressure: the position stream is N beads × ~60/s over stdout; if the pipe fills, Go stalls on the write and TS re-paces Go. Watch for it; emit-on-change deduplication is the escape if needed (not a separate tick).
  • Two arc-length implementations must agree until Phase 2 deletes the TS copy — until then a drift risk.
  • Bead speed vs report rate: confirm the sleep never affects speed — speed stays pulseSpeed, arrival stays inFlightTime, the sleep only changes how many waypoints are shown.
  • Run an industry bug-class scan on the Phase 1–2 timing change before declaring it ready.
  • Zustand store removal: after removing store.ts as authoritative, confirm nothing else silently relied on it as truth — grep its consumers; structural rendering must read from Go's data, not a local model.
  • Guard the TS→Go path against await/Promises: no await on a send-to-Go, no request/response. Consider extending the existing no-TS-timers guard to also reject awaiting the bridge. A blocking TS→Go call reintroduces event-loop coordination — the exact thing this avoids.
  • deleteEdge rewrite: confirm the new path cancels clock-delivery atomically and emits the pulse-cancelled echo; grep old deleteEdge call sites so nothing still relies on the Reset-only behavior. Removing the bespoke verb must not orphan any sender still using it.
