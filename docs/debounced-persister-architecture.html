the two unsettled locks

Most locks in this repo have been deleted — sceneFileMu and entityFileMu are both gone — or settled with a red-proof test (Trace.mu). Two remain unsettled: debouncedPersister.mu (scene_persist.go:63, five instances) and LayoutHolder.mu (layout_holder.go:118, one instance per node). Both survived the file split because what they guard is in memory, not on disk — splitting files cannot reach either.

goroutine locked state timer safe / resolved

1 · what it guards — four fields, four methods

The whole lock is one small struct. Every method takes mu; none holds it across a write to disk.

2 · what debouncing is for — a drag is one write, not sixty

Every arm resets the window. A continuous stream of updates coalesces into one write after activity settles.

2b · is 60 writes/sec actually a problem? — nobody has shown it is

The same question that deflated RealClock.mu's performance premise. Asked here, the throughput case is weak, and the one argument that might hold is a different argument entirely — and unmeasured.

3 · the two parties — and the second one is spawned, not chosen

time.AfterFunc runs flush on a brand-new goroutine each time it fires. Nobody picks that goroutine; the runtime makes it. That is what makes this two-party rather than one-owner.

4 · why take() zeroes pending — the non-obvious reason the lock earns its keep

Two of the five persisters hold a map and mutate it in place. If take() handed the map out without clearing it, a schedule arriving mid-flush would write into the very map flush is iterating.

5 · five instances — independent, not one shared lock

Unlike sceneFileMu, this was never one lock over many writers. Each persister has its own, so they never contend with each other — only with their own timer.

6 · the single-owner candidate

Every lock removed so far fell the same way: give the state one owner. Here that means the timer stops spawning an anonymous goroutine, and the flush happens on a goroutine that already exists.

7 · measured: deleting mu is red, not clean

Unlike sceneFileMu, this one was actually tried. Removing the locking from arm/take/stop/recordWrite and re-running the package tests under -race produces a genuine data race — not a hypothetical one.

0 · what a LayoutHolder actually is

Before the lock: this is one node's record of where its neighbours are, in polar terms, quantized to integer steps. Real values below are node 9's, read from topology/nodes/9/local-polars.json.

1 · what the lock guards — two fields, six methods

One LayoutHolder is embedded in EVERY node kind's struct — one instance per node, not a package-wide singleton. Every accessor takes mu; none holds it across anything but a slice/field read or write.

2 · the issue — every production caller is the owning node's own goroutine

Grepped every non-test caller of all six accessors. Every one runs on the holder's own node's goroutine. A node never reaches into a neighbour's holder — it sends a message and that neighbour's goroutine writes its own. So in production this is single-owner state carrying a mutex.
