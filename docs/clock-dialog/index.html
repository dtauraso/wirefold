Go ↔ TS Pulse Dialog

How a bead travels a wire — the 4-message synchronization protocol

Key idea: there is no global clock. Go decides how long a hop takes (simLatencyMs, derived from wire geometry arcLength / pulseSpeed). TS decides when those milliseconds elapse (animates the bead, and can freeze time on pause). They synchronise via exactly 4 messages per pulse. Pausing TS halts Go because Go's Recv() blocks until TS says "delivered."

Why it works this way

Go owns duration, TS owns the timeline

Go says "this hop takes 340 ms" (simLatencyMs). TS decides when those ms elapse — and can freeze them during pause via getPauseAdjustedNow(). No shared clock is needed; the two sides only agree on a duration, not a wall-clock moment.

Animation completion IS the delivery signal

There is no countdown timer in Go. Recv() blocks on a channel that only unblocks when TS fires the "delivered" postMessage. That is why pausing TS halts Go execution — Go literally cannot proceed until the animation finishes.

Two exactly-once guards

claimDelivered() on the TS side ensures only one frame posts the message, even if useFrame fires multiple times at t ≥ 1. NotifyDelivered() on the Go side handles the deleted and hasSend edge cases so a stale or racing delivery never double-fills a slot.

What actually crosses the Go↔TS boundary is small. Most state is LOCAL to one side. This tab separates the wire format from each side's private state.
