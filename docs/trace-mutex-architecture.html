Trace.mu — the record of a question that closed

This page used to describe Trace.mu as present and its removal as open. It is now deleted — the seventh and last sync.Mutex/Cond in non-test Go. All six others were already deleted before it: geomMu deleted, RealClock.mu deleted, PacedWire.mu deleted, outbox.mu deleted, LayoutHolder.mu deleted, debouncedPersister.mu deleted. Its stated job was guarding the event accumulator, and that claim was true but vestigial — nothing in production ever raced it. What the lock was actually still doing — serializing Breadcrumb's direct sink writes and surviving a shutdown that didn't wait — is what got fixed. The diagrams below are kept as the historical record of the diagnosis; each is marked with what changed.

1 · the hot path never touched the lock — still true, now trivially

Every node and wire goroutine emits events. None of them ever took the mutex. They send on a buffered channel and return. The lock used to live entirely on the far side of that channel, on one goroutine — the diagram below is unchanged except that the mu.Lock() box on the right no longer exists; the drain goroutine writes events with no lock at all now.

2 · what it used to guard — and one field the comment claimed but did not

Historical: what mu covered before it was deleted. All four fields it protected are gone from the lock's job list now — the drain goroutine is the sole writer of events/closed (a sync.Once now), and sink/debugSink are only ever written by the drain too, since Breadcrumb was changed to send on t.ch instead of writing them directly.

3 · who actually took the lock — and which of them existed in production

Historical: this is where the stated justification came apart. Events() had zero non-test callers, and WriteJSONL ran only after the drain goroutine was already dead — so of the five callers below, only drain's own record() and Breadcrumb were ever concurrent in production, and both are now folded onto the drain goroutine with no lock.

4 · what the lock was really doing — shutdown, not accumulation (RESOLVED)

This diagram found the lock's real job: production shutdown used to wait at most 250 ms for node goroutines, then call Close() regardless, so Close could run while goroutines were still calling Breadcrumb. That was genuine production contention, on sink/debugSink, not on events. Fixed at the source, not narrowed: the 250ms grace is gone — MoveDispatch.Start now returns a WaitGroup covering every nodeMover.run/edgeMover.run, RunStdinReader got its own, and wg.Wait() now means nothing is running, so nothing can call into Trace during Close() — the race this diagram found is now unrepresentable, not just unlikely.

5 · unlike every other lock here, this one was held across I/O (now moot — the lock is gone)

Historical: every lock deleted before this one released before touching the outside world. This one didn't — which no longer matters, because there is no lock left to hold across anything. Breadcrumb and WriteJSONL still do the same I/O; they just do it from the drain goroutine with nothing else able to race them.

6 · the single-owner question — drain already was one (RESOLVED)

The move that removed the other six: give the state one owner. events already had one — the drain goroutine was its sole writer, so the accumulator was never the obstacle. What this diagram called "the whole difficulty" — Breadcrumb writing to sink/debugSink directly, on arbitrary node goroutines, not through the drain channel — is exactly what got changed: Breadcrumb now sends on t.ch too, so the drain goroutine is the sole writer of everything the lock used to guard.

7 · the framing above it — Step is arrival order, not causal order (STILL OPEN)

This question does not close with the lock. Trace.mu is gone, but the event channel and the drain's single-writer ordering are unchanged — Step is still assigned in scheduler-arrival order, not causal order. See task/framing-event-order and task/per-owner-buffer-rows.

8 · the red proof — settled by the deletion itself

This section used to say the RED claim was a comment's assertion, not a measurement made here. It is now settled the direct way: the lock was actually deleted and go test -race ./... stayed clean, which is a stronger result than deleting-and- reverting a `Lock()`/`Unlock()` pair to check a comment. The "no panic on a closed channel" claim below is unaffected by any of this — it was never about the mutex.
