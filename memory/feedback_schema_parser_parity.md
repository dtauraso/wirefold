**Superseded 2026-07-14.** This note described a small cluster of hand-authored + generated
TS schema-validation files under `tools/topology-vscode/src/schema/` (a spec parser, its
node/edge/meta/primitive helpers, and a generated node-data validator) — left over from
before Go owned all spec parsing. A code-smell audit (task/audit-2-fixes) confirmed by
import trace and bundle grep that this whole cluster had ZERO production importers — only
one now-deleted test kept it alive. It was deleted, and the Go generator step that emitted
the generated member of the cluster was removed so it does not get silently recreated.

There is no longer a TS-side schema parser to keep in parity with `types.ts` — Go
(`nodes/Wiring/loader.go`) is the only spec parser. If a future find reintroduces a TS-side
validator for `node.data` or graph structure, re-derive the parity rule fresh against
whatever that code actually does rather than trusting this note's original text below (kept
for history only):

Extending a TypeScript spec type (e.g., new enum value on `PortDef.side`, new optional field like `slot`) required updating the matching validator in the (now-deleted) TS spec-parser module in the **same commit**.

**Why:** The runtime validator was independent of TypeScript types — it used `oneOf(...)` against a hand-written allowed-values list. The type-checker wouldn't catch the mismatch. Once new code wrote a value the parser didn't accept, the next load threw and the React Flow canvas rendered blank with no nodes — symptom looked like "diagram disappeared" but the data was fine; only parsing was broken.

**How it was applied:** When adding/extending a field on `PortDef`, `NodeSpec`, `EdgeSpec`, etc., the hand-written validator had to be updated alongside the type change. Build + reload editor before committing so a stale validator couldn't ship. Concrete incident: commit `6c2a238` added top/bottom to `PortDef.side` in types but left `parsePort` allowing only `left|right`; first drag persisted `"bottom"` and the editor went blank on next load.

**Scope update (collapse-representations step 3):** `node.data` was later exempted from this rule — its validator was generated by `tools/gen-node-defs` from Go `wire:"data.*"` struct tags, auto-updating on next `npm run gen:node-defs`. The remaining parity risk was structural fields: `PortDef.side/slot/kind`, `EdgeSpec.kind/arrowStyle`, etc. — still hand-maintained. (All of this is moot now that the whole module is gone.)
