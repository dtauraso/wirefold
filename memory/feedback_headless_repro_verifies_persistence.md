Fixing overlay + node-drag persistence on the agnostic-content-buffer branch, three consecutive fixes shipped with GREEN unit tests but did NOT work live. The unit tests exercised the persister in isolation (a tmpdir in directory form) so they passed, while the real failure was upstream/at-the-seam: (1) `EnableEditPersist` set the tree `root=""` for a FILE-form `topologyPath`, so `posPersist`/`overlaysPersist` no-op'd — the test used the dir form and never hit it; (2) `main.tsx` still pushed overlays to Go on `load` (`encodeOverlaysSet` with empty→default state), clobbering Go's persisted state — invisible to any Go-only test; (3) `LoadOverlays` had an `if !found { return }` that SKIPPED the emit when scene.json had no overlay keys, so the buffer streamed all-off — the round-trip test only checked a non-default key so `found` was always true. Each was only pinned once an agent DROVE the built binary (`go build -o /tmp/wf-verify`, framed `[len:u32][record]` `play`+toggle+move on stdin, past the 250ms debounce) and asserted the actual `scene.json`/`meta.json` bytes before/after, plus a second fresh-binary run to prove the reload-restore. Compounded by [[feedback_two_process_editor_reload]] (a webview-only reload keeps the stale Go process, so "reload didn't help" was ambiguous) and [[feedback_cost_overruns]] (each blind fix was speculative tooling on an unverified diagnosis).

**Why:** A unit test can only prove the unit; persistence/IPC bugs live in path resolution, cross-process seams, and load-order — exactly what an isolated test stubs out. A passing test on a live-failing feature is worse than no test: it manufactures false confidence and invites the next blind patch.

**How to apply:** For persistence / disk / TS↔Go-bridge behavior, treat "green test" as necessary but NOT sufficient. VERIFY by driving the real binary end-to-end and inspecting the on-disk / on-the-wire bytes (headless: build the binary, send framed stdin records, read the files; two runs to prove write THEN reload-restore). Make the test use the SAME input form the runtime uses (file-vs-dir topologyPath, empty-vs-populated file, default-vs-non-default value) — the bug hid in the form the test didn't cover. When a fix "passes but the user still sees failure," stop patching and get runtime evidence (headless repro or a breadcrumb, see [[feedback_runtime_breadcrumbs_beat_static_analysis]]) before the next change.
