What is Navigation Tax?

Navigation tax is the number of distinct files and locations a developer must open to make one logical change to the system. A high tax means the cognitive cost, merge-conflict surface, and error probability all scale with the number of files touched — not with the complexity of the change itself.

Change Scenarios

Each scenario lists the files that must be edited (or generated) and the reason for each touch.

Scenario 1 — Add a pseudo projection 8 files  ▲ worst
  1. tools/pseudo/<kind>.go — Implements From, Parse, Render, and To for the new projection.
  2. tools/pseudo/<kind>_test.go — Unit tests for the pseudo projection logic.
  3. cmd/pseudo/main.go — Must register the new kind under both render and save subcommands.
  4. src/messages.ts — 2 new WebviewToHost variants + 3 new HostToWebview variants + 2 Set literals + new parser cases; message-type strings duplicated here.
    ✓ Resolved — message-type strings are now single-sourced (Phase 2, 34158a2).
  5. src/extension/handle-message.ts — 2 new dispatch cases + 2 new handlers + a findOutNeighbor-style topology helper per new message type.
  6. webview/rf/panels/PseudoPanel.tsx — New MSG lookup row + addition to the Kind union type.
  7. webview/rf/nodes/GenericNode.tsx — Hardcoded data.type === "..." check (~line 211) that gates PseudoPanel rendering.
    ✓ Eliminated by hasPseudo flag (Phase 1, task/haspseudo-flag, 4777b3b) — no longer a manual touch-point.
  8. nodes/<Kind>/node.go — Port names in Go structs must match the topology handle names used by handle-message.ts edge-finding.
  9. tools/pseudo/<kind>.go (port literal cross-check) — Port handle names are re-hardcoded here as string literals rather than imported from the Go struct definition.
    ✓ Resolved — port-handle names are now deduped (Phase 3, 8b1838c).
Scenario 2 — Add a node kind 7 files
  1. nodes/<Kind>/node.go — Go runtime implementation + Wiring.Register call that registers the kind with the substrate.
  2. nodes/<Kind>/SPEC.md — View metadata, port declarations, and default values consumed by codegen.
  3. node-defs.ts [GENERATED by npm run gen:node-defs] — Auto-derived from Go + SPEC.md; must be regenerated for the new kind to appear in the editor.
  4. node-data-types.ts [GENERATED] — TypeScript data-shape types for the node's props; codegen only.
  5. node-types.ts [auto-derived, no manual edit] — Union type over all node kinds; updated by codegen, not by hand.
  6. registry.ts [no manual edit for generic kinds] — RF_NODE_TYPES is derived from NODE_DEFS automatically via specKindToRfType; generic kinds register themselves.
  7. webview/rf/nodes/GenericNode.tsx — Requires a manual data.type check (~line 211) only if the kind has a pseudo panel; the hardcoded branch is the tax.
    ✓ Resolved by hasPseudo flag (Phase 1, task/haspseudo-flag, 4777b3b) — pseudo rendering is now data-driven; no manual edit needed for new pseudo-capable kinds.
Scenario 3 — Add a node-spec field 5 files
  1. nodes/<Kind>/node.go — Add the struct field with a wire:"data.X" tag.
  2. nodes/<Kind>/SPEC.md — Document the new field's default value and view metadata.
  3. node-data-types.ts [GENERATED] — Codegen emits the updated TypeScript type; no manual edit.
  4. nodes/Wiring/loader.go — If the field is a new top-level NodeData property, add it to the loader's NodeData struct.
  5. nodes/Wiring/builders.go — reflectBuild may need updating if required-field validation is needed; note that type mismatches are silently skipped.
Scenario 4 — Add a wire prop 3 files  ▼ best
  1. nodes/Wiring/loader.go — Add the specEdge field and the wire:"prop,..." struct tag.
  2. wire-defs.ts [GENERATED] — Codegen picks up the new tag and emits the TypeScript wire-shape type automatically.
  3. webview/rf/edges/SubstrateEdge.tsx — Destructure and render the new prop in the edge component.

Ranked Findings

Ordered by leverage — impact per unit effort, highest first. These are observations, not prescriptions for large refactors.

1
Effort: Low Impact: Med ✓ Shipped hasPseudo flag in NODE_DEFS — breadth: scenarios 1 & 2 (task/haspseudo-flag, 4777b3b)
The hardcoded data.type === "Input" | "ReadGate" check in GenericNode.tsx (~line 211) lives outside the registry, so every pseudo-capable kind requires a manual edit there. A hasPseudo: boolean flag in NODE_DEFS (set by codegen or SPEC.md) would let GenericNode use data-driven branching and remove the manual touch on every new kind. Cheapest change, recurring benefit on every new kind. Highest leverage.
2
Effort: Med-High Impact: High ✓ Shipped — 34158a2 Single-source the pseudo message-type strings
Every new projection requires the same string written in three separate files (messages.ts / handle-message.ts / PseudoPanel.tsx). A single authoritative source — either a shared constant object or a codegen step off cmd/pseudo/main.go — would cut scenario-2 tax by 2–3 files and eliminate the class of "forgot to add the Set literal" bugs. Biggest absolute cut but more work (codegen or shared const across all three files).
3
Effort: Med Impact: Med ✓ Shipped — 8b1838c Dedup port-handle names
The Go struct already owns the canonical handle name via its wire:"..." tag, but tools/pseudo/*.go and handle-message.ts re-hardcode the same string. A generated port-name constants file (from the same codegen that produces node-data-types.ts) would give pseudo logic and edge-finders a typed import instead of a free string. Go struct already owns the names; tools/pseudo/*.go and handle-message.ts re-hardcode them.
✓
Effort: n/a Impact: reference Wire props (scenario 3) — the low-tax model to replicate
The wire-prop path — Go struct tag → codegen → wire-defs.ts → SubstrateEdge.tsx — achieves a 3-file tax with zero manual string duplication. Scenario 3's Go-tag→codegen→component flow is the low-tax model the above items should converge toward. Informational, not a fix.

Long-Term Fix Workflow

Phased by leverage. Phases are independent and shippable on their own; order reflects payoff per unit effort. End state: every change-type converges toward the wire-prop codegen pattern (Go tag → generated TS → component). All three phases shipped and merged to main.

Phase 1 — Per-kind hasPseudo flag in NODE_DEFS Effort: Low Impact: Med ✓ Shipped — 4777b3b

Goal: Kill the hardcoded data.type === "Input" | "ReadGate" checks in GenericNode.tsx.

Files: node-defs generator (gen:node-defs script + SPEC.md source), node-defs.ts (generated), webview/rf/nodes/GenericNode.tsx.

Steps:

  1. Add a hasPseudo (or pseudoKind) field to NodeDef, sourced from SPEC.md or the Go package.
  2. Regenerate node-defs.ts.
  3. Replace the hardcoded type check in GenericNode.tsx (~line 211) with a NODE_DEFS lookup.
  4. PseudoPanel kind derives from the same field.

Payoff: Removes the manual GenericNode edit from both scenario 1 (add node kind) and scenario 2 (add pseudo projection).

Dependencies: None — ship first.

Phase 2 — Single-source the pseudo message-type strings Effort: Med-High Impact: High ✓ Shipped — 34158a2

Goal: Stop triplicating <kind>-render / <kind>-save / -result / -error literals across messages.ts, handle-message.ts, and PseudoPanel.tsx.

Files: src/messages.ts, src/extension/handle-message.ts, webview/rf/panels/PseudoPanel.tsx.

Steps:

  1. Derive the message-type strings from the kind name (${kind}-render, etc.) in one place.
  2. Generate the union types + the WEBVIEW_TO_HOST / HOST_TO_WEBVIEW Set literals + parser cases from that registry.
  3. Build the PseudoPanel MSG table from the same source.
  4. Dispatch in handle-message.ts keys off the derived names.

Payoff: Collapses the largest chunk of the 9-file scenario-2 tax; new pseudo kinds stop editing messages.ts by hand.

Dependencies: Independent of Phase 1, but lands cleaner after it (PseudoPanel already kind-driven).

Phase 3 — Dedup port-handle names Effort: Med Impact: Med ✓ Shipped — 8b1838c

Goal: The Go struct already owns handle names (FromInput, ToChainInhibitor, …); stop re-hardcoding them in tools/pseudo/*.go and handle-message.ts edge-finding.

Files: tools/pseudo/*.go, src/extension/handle-message.ts.

Steps:

  1. Expose handle names from one source (Go struct field names, surfaced via the loader/codegen or a small exported list).
  2. Have the pseudo render/save logic read handles from it.
  3. Replace the hardcoded sourceHandle === "ToChainInhibitor" matches in handle-message.ts with the shared source.

Payoff: Renaming a port stops being a 4-file hunt.

Dependencies: Independent.

Definition of done: all four change-types (node kind, pseudo projection, node-spec field, wire prop) touch only their owning layer + auto-generated artifacts — no manual cross-layer string edits.
