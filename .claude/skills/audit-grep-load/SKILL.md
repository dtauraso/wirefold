You are running a code-self-defends audit. Goal: identify places where an AI doing a small task would need to grep to verify that the code matches a documented claim. Each finding is a candidate for tightening — either move the claim into the type system, generate the duplicated string, or move a runtime check into a parser.

# Procedure

1. Spawn an Explore subagent (model: haiku) with the prompt template below. It surveys all four categories in one pass.

2. When it returns, present findings to the user as a ranked menu:
   - Findings sorted by leverage (how many drift hazards each one eliminates).
   - Each finding: 2-3 lines max — what duplicates, what could drift, estimated LOC to fix.
   - DO NOT auto-fix. The user reviews and picks.

3. If the user picks one or more, spawn a sonnet general-purpose subagent to do the fix(es), one commit per finding.

# Subagent prompt template (haiku Explore)

Use this prompt verbatim, substituting nothing:

---
Repo: <current repo>. Identify verifying-grep hotspots. Four categories:

**A. String/key duplication across boundaries.** Same string in N≥2 files that must agree for the system to work, with no static check. Examples to look for: identifier names that span Go ↔ JSON ↔ TS, struct-tag literals that mirror JSON keys, port/handle name strings, kind name strings, file-path conventions encoded in code.

**B. Documentation claims about code that could drift.** Prose in `*.md` files (especially MODEL.md, CLAUDE.md, SPEC.md, handoff.md, memory/*.md) that makes a factual claim about code (e.g. "X is called before Y", "field Z is required", "this is generated"). For each, ask: would this survive a refactor without anyone updating the doc?

**C. Runtime-only validation that could move to a parser.** `return ... fmt.Errorf(...)` or `panic(...)` or `throw` calls that catch a shape problem at runtime, where a parser/loader/schema validator could have caught it earlier.

**D. Files claiming to be generated that are hand-edited.** Files with "DO NOT EDIT" or "GENERATED BY" headers — verify they're actually regenerated by a script. Any hand-maintained file with a generated-file header is drift waiting to happen.

Output: max 3-5 highest-leverage findings per category. For each finding: file:line, one-line description, one-line fix idea. Total under 500 words. Be concrete (file paths, line numbers, exact quotes).
---

# After findings

Sort findings by leverage. Heuristics for "high leverage":
- Eliminates duplicated strings that ANY code change might need to update → high.
- Catches an entire class of topology/spec mistakes that today are silent → high.
- Fixes a single instance with no broader pattern → low (still worth fixing, just last).

Present as numbered list. Wait for user pick. If user picks several, do them as separate commits.
