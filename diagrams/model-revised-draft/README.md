# Model Revised Draft — Diagrams

Illustrates the substrate model described in `MODEL-revised-draft.md`. All files are additive; `MODEL.md` and existing diagrams are unchanged.

---

**01-old-vs-new-ownership.svg** — Side-by-side comparison of the two models using the same source→destination topology. The left half shows the current model: the wire owns the parked `loaded(v)` state after pulse arrival, represented as a filled circle on the wire near the destination end. The right half shows the revised model: the wire is transient (empty after arrival), and the slot — represented as a small filled square on the destination node's input port — is owned by the destination node. This directly corresponds to the opening paragraph of `MODEL-revised-draft.md` ("The pivot").

**02-wire-and-slot-phases.svg** — State machine diagrams for all phase machines. The left panel shows the old four-state wire machine (`empty → in-flight(v) → loaded(v) → taken(v) → empty`) with `loaded(v)` and `taken(v)` marked as banned vocabulary under the revised model. The right panel shows the two revised machines: the new two-state transient wire (`empty → in-flight(v) → empty`) and the three-state ordinal slot (`empty → filled(v) → consumed → empty`). Each transition is labelled with the responsible actor (source, wire-arrival, firing-rule). Corresponds to the "What things are" and "Who does what" sections of `MODEL-revised-draft.md`.

**03-fill-consume-sequence.svg** — Animated sequence (27 s SMIL cycle) showing one full fill→consume round: source calls `dest.fill(slotId,v)`, the pulse travels in-flight along the wire, arrival writes the destination slot, the firing rule fires and transitions the slot through `filled(v) → consumed → empty`, and finally the source observes `dest.slotPhase(slotId) = empty` and is free to reload. The observation arc uses the open-arrowhead convention (read, not consume). Corresponds to "Who does what" and the "React surface realization" section; also illustrates how backpressure now lives in the source observing slot phase directly (replacing the old `wire.ack` / `take` signals described in "What this retires").

**04-readgate-multi-input-firing.svg** — Static diagram showing ReadGate with three input wires from three source nodes. All three input slots are shown in the `filled(v)` state (green squares on ReadGate's input ports). A callout explains the firing condition: all slots filled → firing rule runs locally → slots transition to `consumed` → ReadGate calls `fill` on its output wire. Corresponds to "Destination node consumes" in the "Who does what" section and illustrates why the firing rule is node-local under the revised model.

**05-q3-slot-visual-depiction.svg** — Resolution diagram for Q3. Three phase snapshots (empty / in-flight(v) / filled(v)) of a single source→destination wire under the new convention: a small square on the dst's input port renders the slot state (empty when blank, filled(v) when carrying a value), and the wire shows a mid-wire pulse glyph only when in-flight. After arrival the wire is empty and the parked value sits on the node, matching the ownership split in the revised model. Resolution panel and legend describe the new visual rules. Directly corresponds to `MODEL-revised-draft.md § Open questions → Q3`.

**07-q2-firing-rule-and-slot-ownership.svg** — Resolution diagram for Q2. Four destination nodes (ReadGate with 3 slots, AndGate, StreakDetector, EdgeNode/XOR — each with 2 slots) shown with their incoming wires arriving at the node boundary, each wire labeled `→s_k` to indicate its construction-time-bound slot. Slots are drawn as passive cells inside the node, and each node's header carries its rule (e.g. "fire when s0 ∧ s1 ∧ s2"). A resolution panel notes that the original "how does the firing rule subscribe?" framing dissolves under the corrected model: slots don't notify anyone; the node receives the wire's arrival, sees the bound slot id, writes that slot, then re-evaluates its rule. Replaces and merges the previous 07 (A vs B subscription comparison) and a transient 15 (multi-node example). Directly corresponds to `MODEL-revised-draft.md § Open questions → Q2`.
